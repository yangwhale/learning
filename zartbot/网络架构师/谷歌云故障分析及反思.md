Google一直是技术扶贫我们的好老师，没想到也有需要技术扶贫的时候。前几日正好在<包处理的艺术(2)>中阐述了超大规模云网络控制平面设计的问题，谷歌云就给了一个真实的案例来检验。**他们的实现证明了原则1已经被用了很多年，而他们的故障证明了没有实现原则2.**

原则1：控制平面采用CP

正如前面所说，在SDWAN等复杂场景中，控制平面可达性本身就无法保障，经常会存在Headless的情况，那么Availability自然就可以舍弃了，找个CP实现的系统作为控制平面则是非常自然的取舍，一致性的约束范围也有取舍，不保证链路状态一致性，而仅保证资源归属一致性。后面路径计算小节会详细叙述原因。 

原则2：数据平面采用AP->BASE

数据平面才是真正承载用户业务的地方，选择AP的原因是为了实现BASE. 如前文所述，使用SR是必然选择，但是在这一点上不要想太多做太复杂，使用SR只是利用Segment的思想完成LFA就好

我们首先来分析故障原因：

Google’s underlying networking control plane consists of multiple distributed components that make up the Software Defined Networking (SDN) stack. These components run on multiple machines so that failure of a machine or even multiple machines does not impact network capacity. To achieve this, the control plane elects a leader from a pool of machines to provide configuration to the various infrastructure components.
这段话是Google描述自己的分布式控制系统需要分布式控制平面，而分布式控制平面因为需要一致性，需要选择leader，那么也就是说证明了我前文所述的关键点，在CAP理论**中，控制平面的选择为CP，而Leader选举过程中的可用性无法保障，这就是这个事情的大概原因。

**The leader election process depends on a local instance of Google’s internal lock service to read various configurations and files for determining the leader. **The control plane is responsible for Border Gateway Protocol (BGP) peering sessions between physical routers connecting a cloud zone to the Google backbone.
但是我们需要注意到一个很有趣的实现细节，Leader的选举取决于一个内部的分布式锁服务来读取不同的配置和文件，那么这里其实就是说的Chubby service了，出现故障的原因就甩锅给了一个ACL**

Google’s internal lock service provides Access Control List (ACLs) mechanisms to control reading and writing of various files stored in the service. A change to the ACLs used by the network control plane caused the tasks responsible for leader election to no longer have access to the files required for the process. 
最根本的问题在哪？Google控制面也实现了CP，所以基于ETCD等CP构建的路由协议控制面借助这次故障又一次泄露出来了。问题的关键是什么？Google的数据面没有满足原则2，数据面需要采用BASE的设计方式，Segment Routing是必须的，因此在控制面可用性不满足时，数据面跟着也挂了。

所以思科在设计SDWAN时就考虑到了Headerless的问题，在控制器出现故障后，转发平面还有一定时间（通常12~24小时）的缓存时间来维持路径和转发策略。而我设计Ruta的时候，ETCD的lease time也定义了两组timer，一个短timer用于节点和linkstate的通告，另一个长的timer用于EVPN的Type2、Type5以及一些ACL策略通告。这就是我在前文讲的：
**一致性的约束范围也有取舍，不保证链路状态一致性，而仅保证资源归属一致性。**
因此当控制面故障时，资源是保持的，转发面还可以进一步的尝试。控制器不可达时，通常我们也有两种做法，一种使用ETCD代理的方式，这种方式也有一定的考虑就是降低ETCD节点的规模，采用层次化的部署模式隔离故障域。

另一方面则是我们最近研究的一个项目，即对已经处理过的SID标记链路状态（SR+INT)，通过这样的方式完成自组织的带内链路状态通告和快速分发。

这种方案整合了SR和INT，事实上使得路径控制(Path Enforcement) 和路径遥测(Path Telemetry)可以完全的随路同步，因此各个TOR交换机就可以用非常小的代价来测量全网的性能并生成链路状态拓扑。这也是SRoU随路测量和链路状态随路通告功能的一部分，因为在传递的过程中，中间路由器或者交换机也能很好的感知链路上游的拥塞状态。我们会在下一版的RFC draft中添加上。
