## 1. 当前现状及挑战

### 1.1 计算芯片微架构

从计算的维度来看, 现有的处理器微架构已经遇到很多问题了, 以典型的数据中心内三颗Socket为例

CPU基本上已经卷到了极致, 最近十年微架构的变化是相对缓慢的, 当然伴随着Agentic Infra的大量商用也有很多新的机会.

GPGPU虽然过去快20年的发展, 基于SIMT的抽象解决了很多问题, 伴随着TensorCore的引入确实在很多推理workload上逐渐的遇到瓶颈.

DPU的微架构还存在很多问题, 定义不清, 用途不明. 无法在体系结构上说明和CPU以及GPGPU的区别, 以至于BlueField-4这种缝合怪的出现, 上面Grace+DSA+PSA复杂的三套处理器逻辑, 无论是从功能上还是从PPA上都无法说明其体系结构上的取舍.

### 1.2 互连架构

除了微架构以外, 互联架构和协议上也存在大量的争议, NVLink / UALink / SUE / ESUN, 再加上国内也存在 OISA / UB / ETH-X / ETH+ / ETH-Link / ICN 等多种协议. 其中还有很多连内存模型都搞不明白单纯的把ScaleUP仅作为传输协议考虑的人.

### 1.3 存储架构

伴随着HBM的产能约束, 同时DDR价格飞涨, 以及Agentic执行对CPU的大量需求进一步加剧了DDR供应紧张的局面. 当然一些新的介质也在出现, 例如HBF / 3D-DRAM / 3D-SRAM等. 这些也带来了新的挑战和新的机会.

另一方面是这些介质以哪种互连的方式接入到整个系统, 也是值得研究的.

## 2. 从推理Infra看微架构

针对数据中心需要的三颗Socket, CPU / GPU / DPU的区别是什么? 其实这个问题一直都没很好的被定义, 特别是在Agentic Infra的场景下. 三者该如何协同? 这些问题定义清楚了才能展开进一步谈论微架构及其互连架构,  当然有些话题有很多详细的分析只能打哑迷了.

### 2.1 CPU微架构

伴随着Agent今年开始的大规模部署, 对于CPU有更高密度的需求, 例如Intel Clearwater Forest(CFR)会携带288个核, AMD Venice也有256个核, 或许到2028年单芯片的核数还会进一步翻倍以应对更高密度的Agent沙箱执行环境的需求.

高密部署实际的挑战也就来自于内存子系统和NOC, 还有针对高密部署下中断等很细节的处理上. 具体的细节就不公开多说了, 私下会和CPU厂商进行一些交流. 能公开讲的或许是平均每核的内存带宽伴随着内存墙在快速降低, 这种情况下是否会逐渐出现类似于CXL的总线扩展? 或者是统一到一个ScaleUP协议下进行内存池化扩展?

这些是针对纯通用计算的视角, 另一个视角是针对GPU互连视角. NVLink C2C是否真的是一个好的解决方案, 还是更应该统一的把CPU直接接入到ScaleUP总线上? 这涉及到一些内存模型的问题就不多展开了. 内存如何共享如何池化的很多技术细节只能打哑迷了.

### 2.2 GPU微架构

这方面的分析其实在几个月前的一篇文章[《Inside Nvidia GPU: 谈谈Blackwell的不足并预测一下Rubin的微架构》](https://mp.weixin.qq.com/s?__biz=MzUxNzQ5MTExNw==&mid=2247496740&idx=1&sn=c9403138fa59d126fe6cfda19d9b2f76&scene=21#wechat_redirect)已经分析的相对清楚了, 老黄也在内部转发过. 实质的问题就是当TensorCore和TMA都可以单线程issue后, 用SIMT warp specialization来做一些处理Mbarrier的代价和延迟都是非常高的. 代码也相对丑陋, SM引入一个超标量的核来辅助做一些warp调度和这些DSA的调用是一个更好的trade-off

我们也可以看到最近对NV GPU在推理方面的一些挑战, 例如收购Groq等. 实际上后续有一些对Groq的微架构分析[《谈谈那个被NV看上值20B的Groq》](https://mp.weixin.qq.com/s?__biz=MzUxNzQ5MTExNw==&mid=2247497068&idx=1&sn=833aaeb0dd37e9badf3115edac7666b4&scene=21#wechat_redirect)以及[《Nvidia如何整合吸收Groq的技术》](https://mp.weixin.qq.com/s?__biz=MzUxNzQ5MTExNw==&mid=2247497157&idx=1&sn=d968b1f17c477ba08adeb92add92fcc7&scene=21#wechat_redirect).

其实在3D-DRAM和HBF等路径上, 像前面这篇文章分析的, 3D-SRAM 并携带一些PNM的能力或许是一个更好的选择

![图片](assets/062e06460e5c.png)

当然对于3D-SRAM这个路径还有更多的分析需要去做, 例如Global MEM在基于3D-SRAM的架构下有更大容量去承载BDP(带宽 x 延迟)时, 延迟的约束放松一点对于GPU解决内存墙的一些问题, 甚至是拉远池化的处理都是有显而易见的好处的.

### 2.3 DPU微架构

实质的问题是, 我们需要回答什么是DPU?在什么场景需要DPU? 很多人只看到了部分的Offload然后沉迷于算力Offload. 当然在这里还有很多哑迷要继续打.

引用一篇夏Core蛮久以前的文章《要做一个政治正确的DPU哦》[1] 引用一段话:

换位思考一下，你如果是cloud的运营者，扪心自问，面对稳定大于一切的KPI，你想不想要中国政府对街道办级的把控力？ 某些US的XPU经常性升级一些patch，你真的不想稍微防备一下吗？ 如果某天触发它疯狂对外DDOS怎么拷住它？

一个合格的cloud运营者，一定要通过能把控的DPU获得对整个cloud深入到毛细血管的把控力。

题外话：如果购买的第三方DPU出了问题，cloud的运营者大概率得背锅吧。但如果是自家的DPU出问题，还可以选择要不要干掉不听话的DPU芯片负责人，换个自己人……

实质上**DPU是云的特殊需求下为了解决弹性和多租户的问题而存在, 进一步由于多租户引入了安全隔离相关的问题**, 而偏偏在商业DPU上又捅了一些篓子[《世界终究是属于草台班子的-2》](https://mp.weixin.qq.com/s?__biz=MzUxNzQ5MTExNw==&mid=2247497392&idx=1&sn=33956b13ed5fbac6a0d9f1cbc83cc590&scene=21#wechat_redirect).

然后这些草台班子没做过云, 甚至他们口中的Neocloud都是IDC卖卡卖盒子的思路.

当然除了安全的因素外, 还有一些如何适配GPU的微架构进行的一些I/O处理的优化, 提供更好的更适应的GPU Initial的接口. 显然像BF4这样的Grace+DSA+PSA的架构是错误的, 微架构上还需要更多的改变. 当然这部分内容无法对外阐述了.

## 3. 从推理Infra看系统架构

从体系结构的视角上来看, 将GPU作为数据中心的一等公民, 以GPU为中心重构服务器, 在ScaleUP上扩展内存、存储和IO将成为未来几年的体系结构演变的最关键的一点.

![图片](assets/b6daa97227b1.png)

背后的原因及推理Infra的一些分析在[《谈谈基于ScaleUP的存储扩展和三网融合》](https://mp.weixin.qq.com/s?__biz=MzUxNzQ5MTExNw==&mid=2247497421&idx=1&sn=e921514eb03119288952963e82b47850&scene=21#wechat_redirect)中有详细的阐述. 另一方面在[《大语言模型推理硬件的挑战与研究方向》](https://mp.weixin.qq.com/s?__biz=MzUxNzQ5MTExNw==&mid=2247497260&idx=1&sn=eb8c75809a0664e64b4f948ae0032089&scene=21#wechat_redirect)中也有很多详细的分析.

但是三网融合的目的是什么? **为什么要把存储拉入ScaleUP或者存储接入ScaleOut这件事情上没有阐述清楚. 下面将详细的展开.**

对于存储的需求,包含KVCache / RAG Cache / Agent Memory等多个场景, 我们以KVCache进行分析.

### 3.1 从KVCache谈起

实质的原因就是在Agentic workload或者一些Coding任务下, Context会越来越长. 基于HBM的扩展无疑在容量上受到了很大的限制. 即便是到Rubin每个GPU具有288GB HBM4. 但是针对很多任务接近百万的Context长度整个GPU的并发处理能力还是受到很大的约束.

而现在主流模型通常单个Token需要40KB~70KB的KVCache, 当然算法上Sparse Attention / Linear Attention在做一些优化.

**Linear Attention**: 假设一个相对较小的KV用量规模, 例如按照3:1的Linear:Full 的Hybrid attention, 假设Full的有15层, head_dim 128, num_attn_head 64. 按照GQA=8, FP4量化存储, 单个Token需要15KB. 这已经是一个相对较小的估计了, 另一方面针对Linear层, 通常采用存储每500~1000个token的state snapshot缓存. 针对200K context这部分的存储远小于Full层的容量需求.

**Sparse Attention**: 以DeepSeek DSA为例, indexer需要存储 seq_len x (1 + 1//block_size) * indexer_head_dim.假设seq_len =200K, block_size = 128, indexer_head_dim = 128, 则indexer的KVCache开销为25,800KB. 对于MLA部分, 虽然Sparse Attention基于indexer_top_k=2048仅会选择2048个K参与运算. 但是对于一个200K context, 每个token都会选择其中2048个key,对于单次选择token不被选中的概率为由于选择需要执行seq_len次, 那么一个token的key从未被选中的概率为　 由此可知单个token未被选中的概率几乎为零. 因此KVCache的用量为 layers x seq_len x (kv_lora_rank + qk_rope_head_dim) = 61 * 200K * 576= 7027,200KB. 即单个Token需要35KB.如果能够采用NVFP4量化, 则需要17.5KB. **对于Sparse Attention来看, 在Decode阶段的访存容量会小很多, 仅有indexer kcache需要全量访问, 而计算出topk后仅需要访问topk个MLA的 Key.**
容量分析
如果按照单卡32并发, coding/agent这类应用平均200K的上下文长度, 以单个Token 15KB计算需要93GB. 如果按照现在主流模型则需要256GB~448GB的容量存储KVCache, 显然GPU的HBM容量是不够的. 如果未来Context长度进一步增加到400K~1M则需要单卡超过1TB的容量.

对于容量约束, 通常需要HiCache或者Mooncake这类的方案充分利用Host CPU Memory.

![图片](assets/863caf4e3677.png)

另一方面,针对超长的context, 其实也有一些chunk prefill的处理降低显存OOM的影响.

按照当前的硬件架构, 通常分为如下两种形态(NVL72的超节点和普通8卡服务器).

![图片](assets/d8b525536d04.png)

**超节点**: 对于Blackwell架构 Grace CPU的配置的DRAM容量为480GB, 在Rubin的架构配置的Vera CPU DRAM容量为1.2TB. 如果仅针对推理的那些batch所用的KVCache, 容量上基本上是足够的.

**8卡服务器**: 通常这类服务器Host Memory的容量, 按照Venice这样的CPU 16个Channel, 单条DDR内存128GB的容量, 双socket的平台可以支持4TB内存, 如果仅针对推理的那些batch所用的KVCache, 理论上每个GPU卡的内存也够使用.

但是考虑到推理过程中, 系统能够Cache更多的用户请求, 以及未来DeepSeek这类模型使用的Engram对Host Memory的占用, 容量还是不够的. 最近也有一些基于CXL进一步扩展Host memory内存容量的方案, 例如阿里云的Beluga

![图片](assets/8fcc9999d143.png)

具体可以参考《Beluga: A CXL-Based Memory Architecture for Scalable and Efficient LLM KVCache Management》[2]

这样就构成了一个多层次化的存储系统

![图片](assets/7c81fc4ef13c.png)

对于容量的需求有一个大致的结论:
对于Active Request由于显存不够, 很大程度上需要快速的和CPU DRAM进行置换, 然后需要一些外置的内存池和外部的SSD存储池来保持历史会话. 当然对于历史会话的保存时间长度还需要根据业务不同有所考虑. 例如简单QA场景 / 多轮对话场景和Agent/Coding场景通常有一些不同.

特别的来说, 针对Coding场景, 通常我们处理一些coding任务可能需要一周到数周的时间, 特别是针对一些大型的开源项目, Context Prefix Match的概率还是很高的, 可能更好的让这些内容拥有更长的生命周期.

而对于Agent执行来看, 几乎很难有其它用户的执行序列能够有相同的prefix匹配. 或许这部分的内容只需要考虑执行过程中的一些active session的KVCache存储, 生命周期大概只需要数分钟到数十分钟即可. 当然针对某些Agent执行的整个过程,例如企业内的一些research项目, 可能会存在在这个执行trace的基础上, 团队内其它人接着做一些其它分支的任务处理的需求, 针对这部分可能需要以租户粒度去维持更长的生命周期.

带宽分析
对于带宽需求,主要为两个方面:

Prefill阶段, 由于超长的Context容易导致GPU显存OOM, 因此需要一些chunk prefill的处理,或者CP并行, 这些对于KVCache命中部分的加载和未来写出都有相应的带宽需求.

PD分离场景, Prefill节点的KVCache向Decode节点传输的带宽约束. 当然从Prefill到Decode节点的传输很大程度上可以通过一些Layerwise的传输overlap.

具体的带宽因为需要和计算Overlap, 因此需要和算力进行匹配的设计, 并且和模型架构也有很大的关系. 假设按照前一节的计算, 200K tokens 每个Token 15KB左右的存储空间, 如果按照BatchSize 32, Layerwise传输计算大概单层需要6.4GB. 如果使用800Gbps FrontEnd传输大概需要60ms. 使用6.4Tbps ScaleOut传输仅需要7.5ms.

从物理服务器上来看, 带宽的约束分为如下两种情况:

**超节点**: 连接GPU的NVLink C2C带宽是足够的,如果使用Host自带的一些NVMe盘带宽也是足够的, 如果使用其他机器甚至外部的存储集群, 潜在的瓶颈在FrontEnd的DPU上.

**8卡服务器**: 通常这类服务器Host Memory自身的内存子系统带宽约束以及CPU到GPU之间的PCIe带宽约束等.

对于从外部存储获取KVCache的带宽需求需要和实际的请求workload / Cache命中率 / 负载均衡调度的方式有关. 当然还有Cache在一个大集群内一些命中偏斜的问题, 是否KVCache本身也需要一定的副本来降低单点的热点? 或者单点有更大的带宽需求来避免workload偏斜导致的长尾延迟?

在这里就有一个讨论, 是否能够通过ScaleOut网络接入存储? 优点是数据不需要通过FrontEnd经过CPU节点中转, 但是也有一些难题, 特别是对于一些8卡服务器而言需要很高的拥塞控制技巧, 并且避免对EP流量的影响.

### 3.2 从服务调度的视角

从调度的视角来看, 虽然很多Active Request需要Offload存放到Host CPU内存中. 但是对于Historical Session存放到本地NVMe盘还是有一些问题的. 也就是说NV定义的G3存储的处理

![图片](assets/fd9ad6f4e8fa.png)

对于一个大规模多模型推理的MaaS平台而言, 使用Local SSD当存在调度的处理, 例如峰谷的资源弹性伸缩场景下, 例如夜间需要将实例释放并售卖给其它租户进行一些其它工作时, 通常需要将Local SSD中的内容备份到G4共享存储, 并且擦除Locall SSD的内容再售卖. 或者高峰时期资源需要能够做到分钟级别的扩容时, 例如将推理A模型的实例快速倒换到推理B模型, 此时A模型的Local SSD 中的KV Cache需要快速的备份到G4共享存储. 针对这一类场景, NV其实也在构建Rack/Pod Level的存储, 但是当前技术的限制导致他们只能做到Rack Level. 真正能够做到数百个PB容量,并支持数百个TB/s带宽的存储系统还需要很多云服务提供商级别的系统能力.

从调度的视角来看, 实际上的问题是针对warm KV reuse的场景, G3这样微秒级别的延迟需求能否通过集中池化的AZ-Level的共享存储提供?

### 3.3 Warm KV存储选择: 集中式? 分布式?

关于G3和G4存储构建的选择, 其实突然想到有点类似于IBM DB2中的PureScale和Oracle RAC中CacheFusion区别.
PureScale
在IBM大型机上有一个独立的Coupling Facility(CF). 任何成员需要一个数据块时都要向这个集中式的CF发送请求, CF内部有全局锁管理器以及Group Buffer Pool.

![图片](assets/4b5a19449daf.jpg)
CacheFusion
对于Oracle CacheFusion这种方案, 它采用一个分布式锁和分布式全局资源目录(GRD)机制.例如节点B需要请求一个Block

![图片](assets/526af9bc3eb3.png)

节点B以数据块Hash找到主节点D，并发起读取数据块请求

主节点D检查资源锁授权队列，发现C持有该块，并且资源锁属性为SL0和B的读取请求兼容，直接授予B锁，并通知节点C发送数据到B

C发送数据给B

B收到数据后，异步通知主节点D更新资源锁信息

具体的分布式集群管理和锁机制可以参考前一篇《大模型推理系统(1)--先从分布式系统DEC VAXCluster谈起》

两种方案对比如下:

特性

Db2 pureScale

Oracle RAC (with Cache Fusion)

核心架构

中央集权式协调

分布式对等协调

缓存一致性机制

中央锁管理器 (in CF) + 组缓冲池 (GBP)

Cache Fusion 技术 + 全局资源目录 (GRD)

关键组件

成员 (Members) + 集群缓存设施 (CF)

数据库实例 (Nodes) + 私有高速互联网络

数据块传输路径

成员 ↔ CF ↔ 成员 (通常不直接点对点传数据)

成员 ↔ 成员 (通过高速互联网络直接传输)

锁管理

集中式: 所有锁信息都由 CF 统一管理.

分布式: 锁信息 (资源元数据) 分布在所有节点上, 每个节点"主宰"一部分资源.

对于基于pureScale的方案有如下优点:

**极低的协调开销和出色的扩展性**: 由于锁管理是集中式的, 成员之间无需"交谈", 只需与 CF 通信. CF 被设计为超高效的锁和数据仲裁者. 这使得 pureScale 在扩展到非常多的节点 (如 32, 64甚至更多) 时, 依然能保持非常低的协调开销和近线性的性能增长.

**应用透明性**: 对应用程序来说, pureScale 集群就像一个超大的、永不停机的单机 Db2 数据库, 应用无需做任何修改即可享受其带来的好处.

**快速的故障恢复**: 当一个成员宕机时, CF 中保留了其所有 "脏" 数据和锁信息, 其他成员可以非常迅速地接管并完成恢复, 实现了秒级的故障切换.

当然pureScale相对于CacheFusion也有一些缺点:

中央组件的瓶颈和单点风险:

CF 本身可能成为整个集群的性能瓶颈. 如果 CF 的处理能力达到上限, 整个集群的性能将无法再提升.

CF 也是一个潜在的单点故障. 为了解决这个问题, 生产环境的 pureScale 必须部署两个 CF (一主一备), 形成双活模式, 这增加了架构的复杂性和成本.

数据传输路径可能更长: 在某些场景下 (如 Member-to-Member 的数据传递), 数据需要经过 CF 中转, 路径 Member A -> CF -> Member B 比 RAC 的直接路径 Member A -> Member B 要长. 但 pureScale 认为其超高效的中央协调带来的总体收益远大于这点路径开销.

### 3.4 推理存储Infra架构

接着前一节的内容, 实际上Warm KV构造分为两种方案:

第一种方案类似于CacheFusion, 复用GPU实例的Host内存和本地SSD资源构建分布式KV Cache层,由此构建NV定义的G3层存储.

由于推理业务有极致的弹性需求, 从云服务提供商的角度来看, 个人可能更倾向于采用类似于PureScale的方案:

对于GPU实例做更好的存算分离部署, 即尽量不使用GPU实例上的Local SSD. 这样GPU实例可以很容易的开给不同租户不同模型使用.

类似于PureScale的方案, 对于集群的弹性缩扩容会更加方便一些.

即便是这种场景, 本地的Host memory也可以作为KVCache的第一级缓存, 但是需要Write Back到全局的KV Cache memory pool中.

KVCache管理层负责管理全局的Radix Tree和更新这个Tree的一些锁资源以及如何Evict到存储池的策略等.

当然具体的实现和架构就不展开多说了. 只是从体系结构的视角来看, ScaleUP或者ScaleOut接入存储的需求大概率会发生.
