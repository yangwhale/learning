### 测量网络性能的难题

测量本身很简单，就不发包回包算延迟和丢包嘛，然后Traceroute**多跳，反查DNS-PTR和Geo IP地址归属，产生报告。

![图片](assets/294c92708368.jpg)

但是从系统的角度，还需要处理几个难题。

准确性
通常一个网络的质量采用延迟/抖动/丢包率来衡量, 测量方式也有主动或者被动测量, 为了避免测量误差,通常都需要多测测量求平均的方式,并且抖动通常采用标准差的方式来计算. 而丢包率则是一个更头疼的问题 ，它的结果是一个{0，1}的值，所以导致测量精度=1/测量次数， 例如测量10次，那么丢包率精度为10%(1/10)，测量100次为1%.
实时性
另一个问题是多次测量才能估计出准确值，而一个节点同时可能要探测其它几千个节点的可达性和网络性能，如果单次测量为了获得秒级实时性需要发100个包，再乘以几千个节点并行发包， 光测量的带宽就5000Kpps(假设要探测5000个节点)。显然这样的做法会带来大量的带宽消耗。通常我们需要保证测量的背景流量不影响到业务。通常一个探测会话一秒钟只能让它发一个包，这样累计5Kpps就不会影响到整个系统。
两难问题
所以再测量过程中面临了带宽使用和用户对准确性和实时性的需求就产生了这样一个两难问题，传统的测量都是以Tumbling Window，也就是滚动翻窗的方式，例如每10分钟一个窗口，窗口内探测600次，最终统计每十分钟上报一次。然后下一次上报要等10分钟以后了。这样的颗粒度对于控制器来分析数据和运维可能够了，但是对于一些实时性和容错性要求较高的也就这种滚动窗口可能就有问题了，而且我们也接到过很多客户需要近实时的延迟丢包检测需求。

### 滑动窗口近实时测量

所以常见的做法就是采用滑动窗口，计算滑动平均的方式，滑动的平均值计算很容易:

但是有一个jitter需要求方差，似乎滑动计算不好做？很多工程师的算法基本上都是直接硬算：

```
func std_stats(num []float64, N int) (float64, float64) { var sum, mean, sd float64 for i := 0; i < N; i++ {  sum += num[i] } mean = sum / float64(N) for j := 0; j < N; j++ {  sd += math.Pow(num[j]-mean, 2) } sd = math.Sqrt(sd / float64(N)) return mean, sd}
```

这样对于嵌入式平台和需要计算大量的session，每秒计算一次似乎太蠢了吧， 我们来看看方差能不能增量计算呢？ 为最近一次收到的延迟值， 为需要移除滑动窗口的延迟值

所以说最终的延迟和抖动滑窗的计算就变得很容易了， 构造一个环形队列，然后按照如上一次把平均值和方差都算了:

```
type RollingStatus struct { windowSize int Mean       float64 VarSum     float64 data       []float64 index      int loss       *LossBitMap  }func (r *RollingStatus) UpdateLatency(xN float64) { idxN := (r.index + 1) % r.windowSize x0 := r.data[idxN] meanN := r.Mean + (xN-x0)/float64(r.windowSize) r.VarSum = r.VarSum + (xN-x0)*(xN-meanN+x0-r.Mean) r.data[idxN] = xN r.index = idxN r.Mean = meanN}
```

然后您注意到了结构体里面有一个lossbitmap,正因为Loss本身就是每次探测成功了就0， 丢包了就置1的取值， 所以这里采用了Bitmap来处理,只是我们针对Loss做了一些加权处理，因为Internet上延迟和抖动的变化基本上是不明显的，而丢包非常敏感，所以对于最近8次探测采用了一个uint8的值来记录。其余的值则是放到若干个uint32的数组中处理，使用uint32是为了bit计数方便。

```
type LossBitMap struct { index        uint64 data         []uint32 size         uint64 weightedLoss uint8}func (b *LossBitMap) Set(pos uint64) bool { if pos >= b.size {  return false } b.data[pos>>5] |= 1 << (pos & 0x1F) return true}func (b *LossBitMap) Unset(pos uint64) bool { if pos >= b.size {  return false } b.data[pos>>5] &= ^(1 << (pos & 0x1F)) return true}func (b *LossBitMap) UpdateWeightedLoss() { b.Set(b.index) b.index = (b.index + 1) % b.size b.weightedLoss = b.weightedLoss<<1 + 1}func (b *LossBitMap) UpdateWeightedPass() { b.Unset(b.index) b.index = (b.index + 1) % b.size b.weightedLoss = b.weightedLoss << 1}
```

统计有多少个1bit除以windowSize就可以知道丢包率了， 此处针对uint32采用了swar算法，而uint8的近8次加权丢包率值直接采用了查值的处理方式


最终延迟、抖动、丢包的值计算如下：

```
func (r *RollingStatus) GetWeighted(N uint64) (float64, float64, float64) { sd := math.Sqrt(r.VarSum / float64(r.windowSize)) lossCnt := r.loss.BitCount() +  N*uint64(bitsInByte[r.loss.weightedLoss]) return r.Mean, sd, float64(lossCnt) / float64(r.loss.size+N<<3)}
```
