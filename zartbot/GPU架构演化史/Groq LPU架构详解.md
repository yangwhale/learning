## 1. Background

首先有几个背景需要阐述一下

### 1.1 训推一体的故事变了么?

这是整个叙事逻辑的初始点, 以往的故事是, 用最新一代的GPU做训练, 然后前一代的做推理. 那么暗含着一个假设是训推比例大致在1:1左右. 而其实在Blackwell发布初期很多指标都对应着推理的性能提升, 特别是GB200这些超节点配合大EP并行的情况下, 已经有一些用户开箱即拿来做推理了.

另一方面我们注意到Nvidia已经有一些Rubin CPX的方案, 以及前一个月的AWS Re:invent给出来的数据:

![图片](assets/69c5f9c0980e.png)

那么, 我们需要回答的问题是, NV对Groq的收购是否意味着技术路线上面临着训推的分池? 例如训练继续维持GPGPU的架构, 而推理平台出现异构? 当然Rubin CPX算是其中一步, Groq能否算第二步?

### 1.2 从业务模式上分析

其实从AI还能Scaling的角度来看, 前段时间有一个分析《谈谈AWS Trainium3/4 及未来的AI Infra演进》, 未来Scaling的方向来看, RL后训练/Reasoning模型/Agent和具身其实都会显著的拉高推理平台占比.

![图片](assets/2da4a9f9de7a.png)

如果训推比到1:10以上, 似乎独立部署新卡的推理集群这件事情就成立了, 那么某种程度上也有了Groq的机会.  另一方面是来自竞争对手的压力. 例如Google TPU 据说普遍的ROI比Blackwell好30%, 然后AWS Trn3估计也有类似的收益, 并且AWS还要把编译器开源?

关于推理成本的竞争就非常剧烈了, 好在NV还可以拿到充足的HBM供应, 而Google这些即便TPU再厉害, 产能还是受限, 这不前段时间有个消息:**“Google reportedly fires procurement execs amid HBM supply crunch"**

### 1.3 从技术路线来看

其实前一篇[《谈谈AWS Trainium3/4 及未来的AI Infra演进》](https://mp.weixin.qq.com/s?__biz=MzUxNzQ5MTExNw==&mid=2247496916&idx=1&sn=38a19ad579eff61c1da44d57fd2cf1df&scene=21#wechat_redirect)已经有些阐述了.

首先, Nvidia自身的技术路线其实也遇到了一些挑战和瓶颈, 具体可以参考[《Inside Nvidia GPU: 谈谈Blackwell的不足并预测一下Rubin的微架构》](https://mp.weixin.qq.com/s?__biz=MzUxNzQ5MTExNw==&mid=2247496740&idx=1&sn=c9403138fa59d126fe6cfda19d9b2f76&scene=21#wechat_redirect)

然后对于DSA or GPGPU有一个分析:

从体系结构的视角来看, 通常的一些解决办法分为4个方向:

提高并行性

降低数值精度

更好的DataLocality

DSL

对于提高并行性而言, 无论是DSA还是GPGPU都在演进到Tensor Core + Vector Core + Scalar Core的架构. 对于数值精度而言, MXFP8/MXFP4/NVFP4等基于block scaling的低精度压缩也都在支持.

其实本质的区别是在最后两点, DataLocality的处理上, DSA需要更底层的去管理内存和排布流水, 对编译器的要求会比GPGPU更高. 而DSL则是在易用性上解决一些Datalocality复杂的问题, 最近几个月都在卷的一个方向, cutlass-dsl/cuda-tile/tilelang等..

关于DataLocality的对比, DSA一类的通常就是一块很大的SRAM, 而GPGPU通常有更深的内存层次化结构.

![图片](assets/e7789fd6f415.png)

对于DSA和GPGPU, 实际来看追求极致性能的Infra工程师可能对DSL**的依赖并不大, 有些时候还不如自己内联一些指令方便. 其实在Tile-Based DSL架构下, 受众是配合算法开发的一些做训练的Infra同学, 他们有部分的内存管理提高DataLocality的需求, 但又不极致追求性能. 通常一个实验需要跑半个月, 因此算法研究员根本不会给配合他们的Infra同学长达一个月的算子开发时间. 那么给开发的时间周期可能少于几天, DSL便是一个很好的选择, 即便是这些临时的DSL的代码无法打满, 大概获得峰值性能的70%左右也是很有价值的.

那么唯一剩下的区别就是在内存子系统设计了, 实际上针对CUDA编程而言, 伴随着TensorCore的引入, 对于SMEM/TMEM的内存分配管理访问, 即便在Hopper/Blackwell上有更好的异步MBarrier的处理能力, 但整体编程的复杂度和用户心智的影响还是很大的. 特别来说在Blackwell上还存在跨物理Die引入的接近400Cycle的延迟差异, 对于一些较小的Kernel运算也有显著的影响.

**DSA架构和GPGPU架构在Tile-Based DSL下, 对于内存管理的复杂度实际上是有显著降低的, 那么事实上DSA和GPGPU的区别在逐渐模糊化, 唯一的区别可能就在一些算力调度上, GPGPU的指令集发射/Warp调度等通常在加速器内部实现.** 但也逐渐面临一些问题, 例如TMA/TensorCore的异步指令发射仅需要一个线程, 而通常又不得不调度到CUDA Core上一个Warp执行. 而Warp执行的时候为了保证吞吐, 很多指令发射又带有Stall Control-bits, 具体可以参考[《现代NVidia GPU架构》](https://mp.weixin.qq.com/s?__biz=MzUxNzQ5MTExNw==&mid=2247494136&idx=1&sn=958bc734efd43a59b03c04a2e65ec408&scene=21#wechat_redirect), 因此对于低延迟的业务而言存在显著的缺陷.

那么大概的背景就叙述完了, 下面我们详细来分析一下Groq的架构. 最后再来分析一下NV买Groq的一些意图.

## 2. Groq Arch Overview

在2022年的ISCA上有一篇论文《A Software-defined Tensor Streaming Multiprocessor for Large-scale Machine Learning》[1]比较详细的阐述了Groq的架构设计, 在HotChips 34上的一个Session 《HC34-S4: Machine Learning》[2]也有一个非常详细的介绍, 对应的PPT连接《The Groq Software-defined Scale-out Tensor Streaming Multiprocessor》[3].

### 2.1 什么是Software Defined?

在Groq的描述中, 一直在强调“Software Defined Tensor Streaming Multipleprocessor”. 软件定义这词描述不是那么的直接.

在论文的第一章介绍了**问题的背景和动机**: 开篇就点明了大规模计算系统设计的经典矛盾——能力(capability) vs. 容量(capacity), 这分别对应着strong scaling和weak scaling.

Strong Scaling关注延迟, 目标是"更快地解决一个问题".

Weak Scaling关注吞吐量, 目标是"同时解决更多问题".

作者指出, 现代机器学习工作负载(如训练和推理)同时对这两种模式有需求. 例如, 模型并行是一种强扩展, 对处理器间的通信延迟极为敏感. 而数据并行 是一种弱扩展, 需要高聚合带宽来同步梯度. 因此需要构建一个能够同时高效支持这两种模式的系统, 必须解决低延迟和高带宽的挑战. 论文以GPT-3为例解释大模型的计算困境, 指出大模型不仅需要海量的内存容量来存储参数, 还需要强大的计算能力并能将计算任务负载均衡到大量处理器上. 这进一步强化了对一个高效并行系统的需求.

然后作者阐述了**现有方案的缺陷**: 传统的HPC系统和多核心CPU/GPU架构在内存和网络资源上存在动态共享和竞争, 导致**不确定性**(non-determinism), 如下图所示, 这也是我在1.3这一节阐述的, GPGPU的内存层次结构看上去太厚了, 虽然某种层度上降低了编程的一些复杂度, 但是很多时候还是面临一些不可控的因素.

![图片](assets/0757665618f7.png)

不确定性的来源主要包含几个方面:

动态指令调度 (Dynamic instruction scheduling): 乱序执行核心会改变指令的完成顺序.

缓存层次 (Cache hierarchy): 缓存命中/缺失导致内存访问延迟不确定.

动态网络路由 (Dynamic network routing): 网络拥塞导致数据包延迟变化和乱序到达.

共享内存仲裁 (Shared memory arbitration): 多个核心争用内存控制器.

这对于一个超大规模的机器学习系统来说构成了一个巨大的挑战. 这种不确定性使得精确的并行协调变得非常困难. 在需要紧密耦合的并行任务中, 处理器A不知道处理器B的数据何时能到, 只能通过昂贵的同步机制(如屏障, 锁)来等待, 造成大量时间浪费. Groq认为, 与其在不确定的系统中通过复杂的软件来"管理"不确定性, 不如从硬件层面就构建一个确定性的系统.

这也是我一直在强调整个系统建模的时候, Little‘s Law和Kingman公式的重要性. Kingman’s公式是排队论中对于一个通用的单服务器队列模型 G/G/1, 其平均队列延迟为:

$$
\mathbb E(W_q)=(\frac{\rho}{1-\rho})(\frac{c_a^2 +c_s^2}{2})\frac{1}{\mu}
$$

其中 $\lambda$ 为到达速率, $\mu$ 为服务速率,即平均服务时间的倒数, $\rho = \lambda / \mu$ 为系统的使用率. $c_a$ 为到达时间的变异系数,$c_s$ 为服务时间的变异系数. 变异系数CV为标准差除以均值. 然后我们可以根据 $\rho / (1-\rho)$ 然后取不同的变异系数做出一个图来.

![图片](assets/2804d57aff68.png)

我们注意到为了使整个系统的平均服务等待时间变短, 那么就需要进一步去降低网络和计算的变异系数($c_\alpha$, $c_s$).

Groq的**解决方案**: 其核心是将单个TSP芯片内部的确定性(determinism), 通过一套精心设计的软硬件机制, 扩展到整个由数千个TSP组成的系统. 实现确定性的两大支柱:

软件调度的Dragonfly网络: 网络不再是"尽力而为"的动态系统. 相反, 编译器在编译时就规划好了所有通信路径和时序, 从源头上消除了运行时的网络争用和拥塞.

运行时去偏斜(runtime deskew)指令: 硬件提供了特殊的ISA指令, 允许软件在运行时周期性地校正由各个芯片独立时钟频率漂移带来的微小时间偏差, 从而维持整个系统在宏观上的"lock-step"同步.

另一方面, 作者指出了一个工程现实: 通信成本与物理距离(封装层次)强相关. 节点内通信远比跨机柜通信便宜且快速. 因此, 并行策略必须"感知"到这种物理层次, 这是一个复杂的优化问题.下的带宽曲线非常关键. 它不仅展示了系统的性能边界, 也暗示了编译器面临的挑战和机遇. 编译器在做任务划分和通信调度时, 必须把这个非线性的带宽特性作为核心约束条件之一, 尽量将高通信量的任务簇限制在封装层次的内部(如节点内).

![图片](assets/39a4b1d6adec.png)

Groq的TSP(Tensor Streaming Processing)的设计初衷便源于此, 通过静态调度和软件控制来从根本上消除这些不确定性来源. 基于这样的思路, Groq重新定义软件和硬件的角色, 来实现其确定性计算的目标.

在Hotchip34上, 作者开篇点明, "软硬件协同设计(Hardware-software co-design)并非新概念", 但 Groq 的做法是 **"重新审视软硬件之间的接口"**.

![图片](assets/b2a7d619e55d.png)

这不是简单的合作, 而是一场根本性的责任划分变革. Groq 主要关注两种接口:

静态-动态接口 (Static-dynamic Interface):

**定义:** 这个接口划分了哪些工作在"编译时"(statically)完成, 哪些工作在"运行时"(execution time)完成. 这个接口由运行时系统(Runtime System)管理.

**Groq的变革:** 传统处理器(CPU/GPU)将大量决策放在运行时动态处理, 例如指令调度, 缓存管理, 数据路由等. Groq 的理念是将几乎所有这些决策都前移到编译时, 由编译器进行静态分析和规划. 运行时, 硬件就变成一个高度服从的执行者, 严格按照编译器制定的蓝图(一个精确到Cycle级的执行计划)来工作.

软硬件接口 (Hardware-software Interface):

某个张量(tensor)存储在片上SRAM的哪个物理地址.

数据在功能单元(Functional Unit, FU)之间流动的确切路径和时间.

每个FU的流水线延迟.

**定义:** 这个接口决定了软件(特指编译器)能够"看到"和"控制"多少硬件的底层架构状态.

**Groq的变革:** 传统架构倾向于向软件"隐藏"硬件细节, 提供一个抽象层(例如, 程序员不关心数据在L1还是L2缓存). Groq 反其道而行之, 它向编译器"暴露" 了大量的架构可见状态(architecturally-visible state). 这意味着编译器能精确地知道:

这样做的目的是让编译器能够对程序的正确性和性能进行精确的"推理"(reason about), 从而实现对指令和数据的完美协同, 确保"在正确的时间, 正确的指令能遇到正确的数据".

注: 其实在[《现代NVidia GPU架构》](https://mp.weixin.qq.com/s?__biz=MzUxNzQ5MTExNw==&mid=2247494136&idx=1&sn=958bc734efd43a59b03c04a2e65ec408&scene=21#wechat_redirect)中介绍过, 部分的指令在指令发射和Warp调度时也有专用的Stall计数器和依赖掩码等类似的设计.

而Groq对于Software defined的意义是: 将硬件的复杂性与不确定性, 通过一个透明的底层接口, 转移给一个能力极强的编译器, 以换取在特定领域(AI推理)的极致确定性性能.

在整个Groq架构中, 和传统CPU/GPU的区别就在此. 整个系统通过大量的TSP互连而成, 将TSP的生产者-消费者流式编程模型扩展到包含全局内存(global memory). 该全局内存被实现为逻辑上共享(logically shared), 但物理上分布(physically distributed)在各个芯片上的SRAM. 每个TSP贡献220 MiB的全局内存容量, 总容量仅受网络规模(即系统中的最大端点数)的限制.

TSP既充当处理元件(端点), 也充当在通信链路上移动张量的网络交换机. 通过软件控制避免了因动态争用(dynamic contention)网络链路而引入的延迟变化(latency variation). 整个fabric最多可支持10,440个TSP, 提供超过2TB的全局内存, 端到端系统延迟低于3微秒.

### 2.2 确定性设计

Groq有一篇Whitepaper《Determinism and the Tensor Streaming Processor》[4]详细阐述了确定性设计.

Groq 将 "确定性" 定义为一种架构特性, 它能够提供完全可预测和可重复的性能. 这意味着一个工作负载的总执行时间在编译时就已经精确可知, 包括其运行时所需的时钟周期数和操作的精确顺序, 不存在任何性能波动.

Groq TSP 的确定性根植于其独特的 "软件定义硬件" (software-defined hardware) 的设计方法. 其核心思想是将所有控制流的决策从硬件层面移至软件层面 (编译器). 与传统的 CPU, GPU 或 FPGA 不同, GroqChip 在硬件上移除了所有反应式组件 (reactive components), 例如:硬件级的控制流逻辑 (Control flows),硬件互锁 (Hardware interlocks),仲裁器 (Arbiters),重放机制 (Replay mechanisms)..

这些组件在传统架构中用于动态地处理资源争用和数据依赖, 但也因此引入了不确定性和性能波动. Groq 的编译器在编译阶段就预先规划好所有操作. 当一个张量从内存中读出并送往一个FU时, 编译器精确地知道这个过程需要多少时间.

![图片](assets/67580fd37ded.png)

这种方法带来的直接好处是:

简化硬件设计: 移除复杂的反应式组件, 释放了宝贵的芯片面积, 这些面积可以用来集成更多的计算单元, 从而提升峰值性能

消除长尾延迟: 由于没有动态事件和资源争用, GroqChip 从根本上消除了长尾延迟.

精确的功耗预测: 因为每个时钟周期芯片上发生的事情都是预先确定的, 所以可以精确地预测功耗. 这也使得针对特定的功耗阈值优化工作负载成为可能, 从而提升总体拥有成本 (TCO).

和传统架构对比如下:

| 特性 (Feature) | Groq TSP (张量流处理器) | 传统架构 (CPU/GPU) |
| --- | --- | --- |
| 调度方式 | 静态调度: 所有操作由编译器在编译时预先规划好. | 动态调度: 由硬件调度器在运行时动态决定指令和数据流. |
| 核心组件 | 仅包含执行单元和内存, 无动态控制逻辑. 硬件执行一个预先确定的"脚本". | 包含复杂的反应式组件, 如缓存, 乱序执行引擎, 硬件调度器, 仲裁器等. |
| 性能可变性 | 零方差: 每次运行的性能完全相同. | 高方差: 性能受缓存命中率, 分支预测, "邻居"干扰等因素影响, 存在抖动. |
| 长尾延迟 | 不存在: 架构设计从根本上消除了计算核心的长尾延迟. | 普遍存在: 动态调度和资源争用是长尾延迟的主要来源. |
| 编译器角色 | 核心决策者: 负责全部的执行规划, 资源分配和时间编排. | 辅助角色: 主要负责将高级语言翻译成机器指令, 大量优化依赖于硬件在运行时动态完成. |
| 调试与分析 | 简单: 静态性能报告在编译时即可生成, 无需在硬件上运行. 行为可重复, 易于调试. | 复杂: 需要动态分析 (profiling) 工具来理解性能瓶颈, 结果可能因运行环境而异. |

作者总结了确定性带来的四大优势:

1. Ease of Design and Debuggability:

工作负载行为完全可重复, 一次调试, 永久有效.

开发者无需担心"吵闹的邻居" (noisy neighbors) 或进行复杂的动态性能分析.

Groq 的静态分析工具在编译时就能提供详尽的性能报告和芯片资源使用情况的可视化, 极大地提升了开发速度.

2. Predictability:

控制权移至软件栈, 硬件行为高度一致和可预测.

启动一个程序时, 你能 100% 确定它何时完成.

3. QoS - Quality of Service:

确定性与低延迟的结合, 提供了可预测性, 保证了吞吐量, 延迟和数值结果的一致性.

在调度模型集成 (ensemble models) 或多输入系统时, 这种零方差的 QoS 尤为宝贵.

4. TCO - Total Cost of Ownership:

可精确控制功耗, 允许在追求能效时主动降低程序运行速度.

无需为应对 GPU 巨大的峰值功耗而过度配置电源, 从而节约成本.

单核确定性架构消除了开发过程中的猜测工作, 加速了产品上市时间.

渣注
Groq 的方法可以看作是VLIW架构思想在现代 AI 加速器上的极致应用. VLIW 依赖编译器在编译时解决指令间的依赖关系和资源调度, 从而简化硬件. Groq 将这一思想从指令级并行扩展到了整个computation graph和多芯片系统, 将调度单元从"指令"提升到了"张量操作".

但是, 编译器必须扮演一个 "全知全能的上帝" 角色, 完美地解析数据流图, 解决所有数据依赖, 规划数百万个时钟周期内的所有硬件单元操作, 并将它们无冲突地映射到硬件上. 这是一个极其复杂的静态调度问题, 在理论上是 NP-hard 的. 另一方面对于一些具有动态特性的模型, 例如可变长度序列(variable-length sequences)或者MoE这类具有动态分支的模型. 传统架构通过动态调度可以灵活应对, 而静态调度器要么需要将动态问题转化为静态问题可能会带来一些问题.

另一方面, 编译器需要进行全局优化, 模拟和调度, 这可能需要数分钟甚至数小时. 对于需要快速迭代模型和实验的研发场景, 这种编译延迟可能会成为一个瓶颈, 降低开发敏捷性.

同时, TSP是一颗DSA, 它为具有静态数据流图的深度学习推理任务进行了极致优化. 对于通用计算任务, 甚至是一些探索性或结构复杂的 AI 研究模型, 其性能可能远不如通用性更强的 GPU. 开发者被锁定在一个由编译器严格定义的框架内.

在HC34上也有一页进行了阐述:

![图片](assets/ad29a54773c5.png)

"将硬件构建成一个高效的编译器目标.", 这是本页的中心思想. 传统设计中, 编译器是为硬件服务的, 努力去适应和优化已有的复杂硬件. Groq反其道而行之, 硬件的设计目标就是为了让编译器能够轻松、高效地控制它. 硬件不再是主角, 而是编译器的"傀儡". "设计过程中的所有选择都需要服从'为确定性而设计'的设计哲学." 它强调了"确定性"是最高优先级的设计约束. 当面临设计抉择时, 比如"要不要加一个小型缓存来提升某种情况下的性能?", 如果这个缓存会引入不确定性, 那么根据Groq的哲学, 答案就是"不要". 所有的决策都要在这个原则下进行权衡. 因此硬件必须使编译器和运行时接口能够对程序执行进行推理,为了达到这个目标, 硬件必须:

清晰的内存一致性模型 (Well understood memory consistency model): 传统CPU为了性能, 经常会对内存读写操作进行乱序执行(reordering). 比如, 一个写操作可能会被延迟, 而后续的读操作可能会被提前. 这使得程序员或编译器很难确定在某个时间点内存中的确切状态. Groq的TSP硬件则保证了内存操作的严格顺序, 编译器发出指令的顺序就是硬件执行的顺序. 这虽然可能牺牲了某些情况下的指令级并行性, 但换来了完全的可预测性.

移除所有"反应式组件" (No "reactive components").

暴露架构状态给软件: 软件必须能够访问架构可见的机器状态, 以便能够让即将执行的指令与其数据(操作数)精准地'拦截'.

然后阐述了编译器视角下的数据流编排: 编译器'知道'芯片上每个张量的确切位置.这是上述原则带来的直接结果. 在任何一个时钟周期, 编译器都能回答"张量T现在在哪里?"这个问题. 通过这种方式, 编译器正在编排(orchestrating)操作数的到达以及使用这些操作数的指令.并且基于生产者-消费者流式编程模型允许一组'流式寄存器文件'来追踪流经芯片的每个张量的状态.

#### 2.2.1 芯片层面的确定性

在确定性的设计原则下, 从在芯片层面来看, 增加指令并行而出现的乱序执行等技术会引起长尾, 另一方面在数据层面上cache的层次化结构也会带来不确定性, 虽然DRAM->L3->L2->L1->GPRs之间的数据自动流动可以隐藏访问内存延迟并简化开发者的负担, 但是从芯片面积上的开销来看效率并不高.

![图片](assets/4da31f0bb574.png)

TSP的解决方法是仅构建一个巨大的单层的ScratchPad SRAM, 它具有固定的确定性的延迟. 然后在空间和时间上显式地分配张量, 从而释放了巨大的内存并发性, 以及沿着多个维度的计算灵活性, 即将一个大张量拆分到不同的内存切片或bank, 以实现并行读写, 从而获得巨大的内存并发性. 将数据放置在离其计算单元最近的半球(hemisphere), 以减少数据传输延迟.

#### 2.2.2 系统层面的确定性

对于整个系统, 在传统的WSC和超算集群中, 系统节点数通常超过2万个, 并且架构逐渐的异构化(CPU/GPU/FPGA/智能网卡), 延迟的可变性限制了应用的扩展规模, 并且与网络直径有关. 另一方面全局自适应路由是复杂的, 虽然可以显著降低延迟的分布,但是由乱序消息, 故障, 热点, 路由/负载不均衡, 拥塞等大量问题引入了大量的复杂性.
![图片](assets/d9f711b699e4.png)

在Groq所讲述的SDN(Software-Defined Networking)将网络路由的智能从硬件(路由器)转移到了软件(编译器).编译器控制Groq C2C(Chip-to-Chip)网络中的流量模式, 并在芯片上实现自适应路由, 以减少延迟的可变性并降低缓冲区占用.

在网络拓扑上, 主要是选择更高Radix的交换机以及降低网络的直径. 最后Groq的选择是采用了Dragonfly的拓扑.

渣注:
其实这部分的分析是蛮有价值的, 在NV的GPGPU架构中, 确实大量的L2 Cache占用了很多芯片面积, 但是似乎对它又很难去做一些控制, 缓存层次结构比较厚. 另一方面对于系统层面, 举一个比较常见的例子, 例如一个在ScaleOut网络上的EP并行流量, 同时混入了KVCache的流量将会带来很多干扰和长尾的影响.

某种意义上来看, 系统需要一些确定性, 但是Groq这种绝对的确定性是否走向另一个极端了? 这是值得带着怀疑去深入研究的.

## 3. Groq TSP架构

### 3.1 TSP微架构

在传统的MultiCore架构中, 每个PE都是一个完整的多级流水线架构配置了独立的指令译码/派发器件和独立的Cache层次化结构:

![图片](assets/2c6ae9098687.png)

在TSP微架构上, 对传统的多核处理器微架构进行了重组(Re-organizing). 经典的处理器五级流水线被拆散在了整个芯片内:

![图片](assets/834f913c17f6.png)

相对于传统多核处理器, Groq整个芯片构建了独立的ICU(Instruction Control Units)用于取指和译码,  整个芯片被水平地划分为多个功能切片(Functional Slice), 包括 FP/INT(浮点/整数计算), NET/SXM(网络/数据交换), MEM等. 在每个时钟周期, 这些ICU会向下方对应的功能切片垂直地, 同步地广播指令.这种执行方式类似于 SIMD, 所有位于同一垂直列上的FU在同一时刻执行由顶部ICU发出的指令.

数据从一个功能切片(例如MEM)产生, 变成一个"流", 然后水平地流向下游的功能切片(例如FP/INT)进行计算. 计算结果再继续以流的形式向下游传递, 或者被写回MEM单元.

我们注意到右下角流水线的区别, 增加了一级OD(Operand Dispatch), 然后包含两级执行或者内存访问最后写回. 然后整个芯片根据执行功能不同被分为不同的单元:

![图片](assets/b4713fc76912.png)

整个芯片架构如下所示:

![图片](assets/83b97d8be165.jpg)

SRAM分为两块, 每块有44个独立的bank, 可以提供80TB/s的带宽, 互连网络支持480GB/s的带宽并支持多种拓扑的组网结构. 然后有矩阵计算的MXM单元/向量计算的VXM单元, 以及用于进行shift/transpose/reshape/permute的SXM单元. 以及前文描述过的指令控制单元(ICU).

整个芯片如下图所示, 它由两个对称的“半球(hemispheres)”组成:

![图片](assets/46586839b357.png)

芯片上有144个独立的指令控制器(ICU), 每个都控制着一个垂直的功能"列", 它们同步地向下分发指令, 实现了大规模的SIMD式控制.

然后在TSP上的一个SIMD基本单位是一个向量320个elements. 如下所示:

![图片](assets/11300defb1bf.png)

因此矩阵运算单元MXM为320x320. 向量单元(VXM)对齐了MXM的320lanes, SXM和MEM都会对齐320 lanes.

![图片](assets/fc018bb9b784.png)

VXM每个lane有16个处理单元(PE), 总计5,120个PE, 每个PE每个周期能够进行一次32位计算, 或四次INT8计算.

由于每个Lane有16个PE, 编译器还可以构建一些复杂的VLIW指令进行一些element-wise计算或者shuffle reduction的操作. 后面一节会详细展开.

最后在TSP中强调了一下它的架构可以支持每平方毫米大于1TOPS的算力, 相对于NV GPU的层次化内存架构(RF/SMEM/TMEM/L1/L2)和复杂的NOC设计, TSP这种极简的设计会获得更高的算力密度.

### 3.2 软件抽象

Groq对于其软件抽象解释为: SIMD + Spatial Arch + Streaming. SIMD使得一条指令就可以同时作用于大量的数据元素, 实现大规模的并行计算. 然后传统处理器大量的指令在少数的通用ALU**上计算, 而在TSP中, 计算任务被"展开"到芯片的物理空间上. 不同的计算操作被映射到物理上分离的, 专用的FU(MXM, VXM, SXM). 数据则在这些固定的FU之间沿着物理路径流动. 数据按空间路径传递.
SIMD
由于所有指令都对齐了320lanes, 因此Groq的运算采用lockstep执行SIMD指令, 芯片上所有处于活动状态的FU在每个时钟周期的驱动下, 统一步调地执行各自的指令.

![图片](assets/2596ab3d010f.png)

从另一个维度来看, TSP宏观上可以被看作一个144路宽的VLIW处理器. 在每个时钟周期, 编译器需要准备一个包含144个子指令的"指令包", 每个子指令对应一条分发路径, 用于控制该路径下的FU(MXM, VXM, MEM, SXM)在该周期的行为, 指令在多个FU上同步分发, 这144条路径共同构成了TSP大规模SIMD并行性的基础.
Streaming
然后Streaming就是连接SIMD和Spatial Arch的关键, 数据(张量)不被视为静态存储在内存中的块, 而是被看作在FU之间连续流动的"流"(Streams). 一个FU(生产者)完成计算后, 将结果作为一个流输出; 下一个FU(消费者)直接从这个流中获取输入. 在TSP的语境下, 流式计算是其编程模型和数据传输机制的核心. 它将SIMD的计算能力和空间架构的物理布局有机地结合起来, 形成了一个高效的, 生产者-消费者的计算流水线.

![图片](assets/547e2b62317a.png)

然后整个ISA的设计是为了赋能软件, 通过架构暴露的低级抽象, 实现了软件控制的内存.

![图片](assets/f53fe0535fef.png)

首先它非常激进的取消了所有的动态硬件缓存机制来获得确定性.  然后构建了一个扁平的内存结构(去掉了像NV-GPU那样RF/SMEM/TMEM/L1/L2/GMEM的复杂层次结构). 内存作为一组可被直接寻址的物理存储bank暴露给软件, 即每块芯片有220MB都可以构成一个有80TB/s带宽的Scratchpad Memory. 然后整个系统按照如下方式进行统一编址:

![图片](assets/80677b9a71c6.png)

编译器被赋能以Cycle级精度进行指令调度, 所有的FU以lockstep执行, TSP有144条指令分发路径, 每条路径每个周期都能发射一条指令(可以是有效指令, 也可以是NOP). 这使得指令吞吐量变得完全确定, 因此整个芯片可以被看作是完全流水化的, 144路宽的VLIW指令.

![图片](assets/7f373cb51b77.png)
由于所有复杂的调度工作都由编译器完成, 硬件的控制逻辑被极度简化. 它不需要检查数据依赖(由编译器保证), 也不需要处理资源冲突(由编译器避免), 因此指令分发逻辑的面积开销小于3%. 意味着超过97%的芯片面积都可以用于真正的计算单元(MXM, VXM)和存储单元(MEM), 从而实现了极高的算力密度和能效. 这正是Groq用编译器复杂性换取硬件效率的核心体现.

下图解释了TSP内部数据是如何在不同FU之间流动的, 并强调了这种机制如何让编译器能够轻松地推理数据移动.

![图片](assets/2a6b104c6dde.png)

数据只在水平方向上(东向或西向)流动. 没有复杂的二维或三维路由选择. 路径是预先固定的. 东向和西向的路径由'流寄存器'阵列组成.数据从一个FU移动到相邻的下一个FU, 其延迟是固定的, 就是一个时钟周期. 没有仲裁器/队列 = 软件可以轻松地推理精确的数据移动, 而无需仿真.这样编译器推理耗时也仅需要简单的一个减法计算即可.

### 3.3 ISA

TSP提供的主要的一些指令如下图所示, 整体设计思路还是对硬件进行显式的, 低级的控制, 并通过320B位宽进行向量/矩阵操作.

![图片](assets/6a5428a66181.png)

这个指令集兼具RISC和CISC**的特点:

RISC-like: 大部分指令功能单一, 格式规整, 易于流水线化(如VXM的基本运算).

CISC-like: 存在Repeat, Permute, 以及MEM的硬件循环指令. 一条指令可以完成传统RISC需要多条指令甚至一个软件循环才能完成的工作. 这种设计的目标是提升指令密度和执行效率, 将常见的复杂模式硬件化.

另一个显著的特点是, 表格中没有列出传统的条件分支指令, 这是因为TSP是为静态计算图设计的. 指令本身也存储在片上SRAM中, 每个FU"列"(Superlane)顶部的ICU负责从SRAM中获取它需要执行的指令序列.

![图片](assets/37fbe58c92b0.png)

每个FU都有其自己的指令子集, 大部分指令是专用的(如矩阵乘法指令只在MXM上执行), 但有一些"控制流"指令是所有FU都必须理解和支持的, 以确保整个芯片的协同工作.

IFETCH : 用于从SRAM中取出指令到ICU并开始执行

NOP: 当某个FU在某个周期没有任务时, 编译器必须用NOP指令填充该时隙, 以保持与其它FU的同步.

SYNC: 这是实现Barrier的核心指令. 多个FU可以通过执行SYNC指令在同一个逻辑点上等待.

NOTIFY: 这是与SYNC配对的"释放"信号. 当所有需要同步的FU都到达屏障(都执行了SYNC)后, 由一个(或多个)FU发出NOTIFY信号, 所有等待的FU便同时恢复执行. SYNC/NOTIFY机制是编译器用来精确控制跨FU依赖关系和执行顺序的工具.

渣注
这些和NV GPU在Hopper开始引入的MBarrier是类似的处理, 让TC/TMA/CUDA Core有更快速的协作. 不过TSP更加简单一些通过显式的SYNC/NOTIFY进行控制, 直接反馈到指令控制流中.

然后另一个特征是, 每个功能单元只执行与其操作相关的指令 (例如, 向量处理器只做逐元素的操作, 不做加载/存储).不同功能单元的职责是相互独立的, 没有重叠. VXM负责计算, MEM负责访存, SXM负责数据重排. 这里好像有点NV WarpSpecialization的味道.

### 3.4 MXM

整个芯片由4个MXM构成, 东西方向各放置了2个. 它们位于VXM和SXM之间. 这个布局使得数据可以方便地从内存(MEM)或交换单元(SXM)流入MXM进行计算, 计算结果也可以方便地流向向量单元(VXM)进行后续处理(如激活函数).

![图片](assets/8c6deff083ee.png)

然后4个MXM可以独立使用, 也可以合并为一个整体协同完成一个更大规模矩阵的运算. 每个MXM支持320B x 320B的点积. MXM指令集包含

LW(Load Weights): 将权重从主SRAM流式加载到MXM单元内部的专用缓冲区.

IW(Install Weights): 将缓冲区中的权重"部署"到实际的计算阵列上. `LW`和`IW`的分离允许编译器进行流水线操作, 在当前计算进行的同时预加载下一批权重.

ABC(Activation Buffer Control): 激活值缓冲区控制, 用于发起和协调到达的激活值

ACC(Accumulate): 控制矩阵乘法的结果如何被累加

MXM的指令集清晰地反映了矩阵乘法的标准流程: 加载权重 -> 加载激活值 -> 计算与累加.

### 3.5 VXM

VXM为计算提供了一个灵活且可编程的结构, 在VXM中单个Lane有16个PE, 因此可以执行一些复杂的功能:

![图片](assets/7b6fa570c924.png)

图示了数据流依次经过MatMul -> Dist -> Accum -> Add -> ReLu -> Cast这一系列操作, VXM支持多种Element-wise的指令, 例如常见的加减乘除/逻辑运算, 当然还包括一些Exp/Log/RSqrt超越函数计算, 同时也支持类型转换(cast)和一些基于mask的选择指令.

### 3.6 SXM

SXM(Switch Execution Module)专门用于在TSP架构中处理数据移动.

![图片](assets/aac5f1993c98.png)

SXM单元位于芯片的外围区域, 靠近IO和MXM单元. 从外部进入芯片的数据或者准备从芯片发出的数据, 往往需要进行格式转换或重新布局, SXM可以首先对它们进行处理. 而将SXM放在MXM旁边, 可以高效地为MXM准备数据或处理其输出.

用于数据操控和向量内字节操作的瑞士军刀, 能够解决各种数据布局和重排问题. 它能够通过指令执行Shift, Rotate, Distribute, Permute, Transpose, Transport等操作. SXM内部由几个子单元构成, 以实现其多种功能.

Distributor : 每个半球有4个, 在一个16字节的superlane内, 执行输入+掩码到输出流的映射.

Transposer: 每个半球有2个, 在20个superlane上, 对16个向量执行superlane内转置.

Permuter/Shifter: 输入+掩码的任意映射, 可以在320字节的向量元素之间进行Shuffle处理, 任意映射意味着可以将一个320元素的输入向量的任意一个元素, 移动到输出向量的任意一个位置.

### 3.7 MEM

在TSP的MEM上, 由东西两个半球两块构成, 每块有44个独立的MEM Bank/Slices, 然后每4个一组, 共计11组.

![图片](assets/afefad74a45f.png)

Read是流的生产者(Producer): 单次读取会从一个MEM Slice中创建一个320B的向量流, 然后这个向量被放置到一个"流寄存器"通道上, 开始了它在芯片上的旅程. Write是流的消费者(Consumer): 当一个流中的数据需要被保存时, 它会在MEM单元被"捕获"并写入到SRAM中. 这次写入标志着这个数据流生命周期的结束.

一个逻辑上的大张量, 在物理上会被编译器拆分并分布到多个独立的内存切片中.一个[M, N, K]的三维张量, 编译器可能会将它的不同部分存储在不同的内存切片里, 以便后续可以并行地读取张量的不同部分, 形成多个并发的数据流, 从而最大化内存带宽.

为了保证高性能的内存操作, MEM还支持了`Gather s, map` / `Scatter s, map`处理非连续内存访问, 并且配合(`Countdown`, `Step`, `Iterations`) 共同构成了一个强大的硬件地址生成器. 编译器只需配置一次, 硬件就能自动生成一系列复杂的地址序列(如跨步访问), 极大地减轻了循环中地址计算的负担.

### 4.3 芯片间的链路和流控

在互连时主要分为三种介质:

节点内 (Intra-Node): 最短距离 (<0.75m), 使用最便宜的AWG-34细铜缆. "low-profile" 的设计考虑到了机箱内紧张的空间和风道.

机柜内 (Intra-Rack): 中等距离 (<2m), 使用标准的QSFP DAC电缆. 这是一种在数据中心广泛使用的高性价比方案.

机柜间 (Inter-Rack): 长距离, 此时铜缆损耗过大, 必须使用AOC.

作为确定性网络设计的结果, 硬件被禁止施加会破坏网络确定性操作的反压(back pressure). 相反, 编译器显式地调度系统中每个物理链路上的vectors传输, 并同时考虑到每个通道的带宽和延迟, 以确保链路不会过载. 如下图所示,传统的网络中, 当出现反压信号后, 发送端会选择其它路径重路由.

![图片](assets/c5097098c3a2.png)

反压的不确定性原因
在传统网络中, 如果路由器A要向路由器B发送数据, 但B的输入缓冲区快满了, B会向A发送一个"暂停"信号(即反压). A收到信号后会停止发送, 直到B的缓冲区有空闲空间并撤销"暂停"信号. 这是一种基于信用的流控机制, 旨在防止数据包丢失.它引入了不确定性. 数据流的暂停和恢复时间是动态的, 取决于网络当时的拥塞状况. 这与Groq追求的**确定性(determinism)** 目标背道而驰.

论文提到"使用每个TSP上的本地SRAM存储来为张量的各个向量提供中间缓冲". 这不是传统意义上的动态排队缓冲区. 这是一个由编译器静态管理的"暂存区". 编译器知道一个向量会在何时到达, 并在SRAM中为其预留好空间, 之后在另一个预定的时间将其取出并转发或计算.

在Groq的设计中, 由编译器编排好了整个通信路径避免出现拥塞反压这样的不确定行为. 具体来说, 当一个张量(tensor)逐跳流经网络时, 使用每个TSP上的本地SRAM存储来为张量的各个向量提供中间缓冲, 由于网络路径是完全确定性的, 一个由一个或多个向量flit组成的张量可以使用virtual cut through进行调度, 因为接收TSP可以立即开始将向量发送到张量路径的下一跳.

Virtual Cut Through
在传统的交换技术中, 通常采用Store-forward的机制, 需要接收完整个数据包才能转发. 而Cut-Through在收到包头解析完目的地后就立刻转发.  VirtCT是两者的折中, 它至少要接收完一个完整的FLIT才能转发. 例如上图右侧, TSP-C会收到来自TSP-A的FLIT, 编译器保证了当FLIT到达TSP-C时, C到D的路径在下一个调度好的时刻是可用的. 因此, TSP-C的硬件控制逻辑可以被设计为: 一旦接收完一个完整的flit, 就可以立即根据指令流将其送往下一跳的发送队列. 这将每跳的延迟从"整个数据包的传输时间"降低到"一个flit的传输时间", 极大地减少了多跳通信的端到端延迟.

具体的实现我们将在后续的章节里详细展开.

## 5. 在分布式系统中维持确定性

在多TSP环境中, 需要解决一个难题: 如何在没有全局同步时钟的情况下, 维持多个独立芯片之间的确定性执行.

![图片](assets/8b76709586ea.png)

不确定性的来源有如下几点:

链路延迟变化 -> 需要精确估计链路延迟

所有TSP之间没有一个全局同步的时钟 -> 需要一个'逻辑上的'全局时钟. 这是因为在大规模系统中, 很难(甚至不可能)将一个完全同相的时钟信号分发到所有芯片. 每个TSP都有自己的晶振, 它们产生的时钟频率会有微小的差异, 并且相位也不同.

时钟漂移 -> 需要补偿时钟漂移. 即使两个时钟在某个时刻被对齐, 随着时间的推移, 由于频率的微小差异, 它们的相位会逐渐"漂移"开. 1GHz时钟(1ns周期)下, 百万分之一(ppm)的频率差异, 在1毫秒(1,000,000个周期)后就会累积出1个周期的偏差.

因此, 必须通过硬件和软件的协同支持, 才能确保跨多个TSP的同步通信.

### 5.1 Overview

每个TSP上的FU集合作为一个单一的逻辑核心运行, 这个核心是静态调度(在编译时)的. Groq将单芯片TSP的确定性扩展到一个多芯片的分布式系统, 以便能够高效地共享全局SRAM, 而无需使用互斥锁(mutex)来保证对全局内存的原子访问.

取而代之的是, Groq在特定的时间通过显式的发送或接收指令进行通信, 这样我们就可以从这种total ordering中推断出程序的正确性. 为了实现这种可预测的行为, TSP的硬件-软件接口暴露了所有架构可见的状态(所有的SRAM和流寄存器), 以便静态计算图可以表示为一系列施加了时间期限(temporal deadlines)的依赖关系, 这些期限针对的是正在通信的张量的操作数到达时间. 将这些依赖关系表示为一个DAG, 以便显式地调度通信流量. 下总结了为实现跨多个节点的确定性而提供的ISA支持.

![图片](assets/1ea9902a67df.png)

一个多TSP系统依赖三种机制来建立和维持同步性:

每个TSP内的一组硬件对齐的计数器(hardware-aligned counters,HAC), 它们会持续地(每256个Cycle)交换信息以维持一个全局共识时间;

一个初始程序对齐(initial program alignment)的过程, 利用链路来确保每个TSP同时开始执行其指令;

一个运行时重同步(runtime resynchronization)过程, 用于解决长时间运行计算中单个TSP的时钟漂移问题.

### 5.2 HAC

每个TSP维护一个自由运行的内部**硬件对齐计数器(HAC)**, 其溢出周期很短(< 256周期). 当两个TSP通过点对点C2C链路连接时, 每个TSP都会将其内部的HAC值传输给其对等方. 这提供了一种表征链路延迟的机制: 一个TSP传输其当前的HAC值, 然后其对等方将该值反射回来. 当反射的HAC值返回到发起TSP时, 将其与内部自由运行的HAC进行比较, 其差值即为链路延迟(模HAC周期的倍数).

![图片](assets/2e0c98c897f6.png)

此过程会重复进行, 直到对系统容差范围内的平均延迟和方差的估计有足够高的置信度. 下表总结了在一个8-TSP节点内执行此操作10万次的结果, 并显示了7条节点内C2C链路的延迟:

![图片](assets/26581af5e2c6.png)

在链路延迟被表征之后, 两个对等TSP $T_0$ 和 $T_1$ (其观测到的延迟均值为 $L$), 被配置为父/子关系, 使得 $T_0$ 内维护的HAC(即HAC0)作为参考, 并周期性地传输给 $T_1$. 当 $T_1$ 接收到HAC0的瞬时值时, 会将其值加上延迟 $L$ 与HAC1进行比较. 这个差值代表了由持续时钟漂移引起的初始未对准. HAC1的值会被调整以减小这个差值(最大调整速率是可配置的). 经过足够多次迭代(大约受HAC计数器周期的限制)后, 两个计数器HAC0和HAC1将收敛到一个由链路延迟抖动决定的邻域内. 因此, 该协议为两个TSP确定了一个共同的周期性参考.

为了将此协议扩展到TSP的多跳网络, Groq建立了一个父/子HAC关系的**生成树**, 以在整个网络中分发和维护一个共同的HAC参考时间.

HAC对齐过程是同步分布式系统编程的基础. 在这个基础上, 通过软件控制的ISA支持来构建一个共同的软件时钟参考.

### 5.3 初始程序对齐

回顾一下在单个TSP上确定性调度多个功能单元的基础: 依赖于一个`SYNC`指令来"暂停"各个独立指令流的指令发射, 随后由单个功能单元发出一个`NOTIFY`指令. 这个`NOTIFY`指令会产生一个全局控制信号(具有已知的延迟), 传递给被暂停的功能单元, 使它们在同一个时钟周期重新开始执行. 因此, 这个`NOTIFY`指令充当了所有其他静态调度所依赖的软件控制的时间参考.

`SYNC`和`NOTIFY`指令提供了一种芯片范围的同步机制, 它依赖于一个共享时钟和完全确定的控制传播路径, 这在多TSP系统中是不存在的, 因为每个TSP都有一个独立的时钟源. 然而, 共同的(周期性)HAC参考可以被结合起来, 以提供整个系统共享时钟的假象, 方式如下. 首先, 引入一个`DESKEW`指令, 它允许我们将程序执行与(本地的)HAC对齐. 当一个功能单元执行`DESKEW`时, 它将暂停在该功能单元上发射后续指令, 直到下一次HAC溢出或到达epoch边界. 这使我们能够确保分布式计算可以相对于一个在网络中所有TSP间共享的参考时间`HAC的epoch边界`开始.

下图展示了在调用一个多TSP程序前, 使用`DESKEW`指令进行基于HAC的同步过程.

![图片](assets/08280cb7305d.png)

在时间 $t_1$, 子设备在本地被置于一个轮询同步循环中. 在这个轮询循环中, 在每个epoch边界(图中显示为HAC=0), 子设备将等待从父设备接收一个向量. 如果该向量尚未被传输, 循环将继续. 在时间 $t_2$, 一个父程序被调用, 它执行一个`DESKEW`后跟一个`TRANSMIT`. 尽管事件 $t_1$ 和 $t_2$ 是非同步的, 但`DESKEW`的使用强制将后续指令与共享的HAC值对齐. 父节点的`TRANSMIT`发生在一个epoch边界, 向量在某个时间 $t_3$ 到达子节点. 这个向量将被`RECV`指令消耗, 该指令将在 $t_3$ 之后的下一个epoch边界后发射. 这将导致子设备在父节点向子设备传输后的 $\lfloor L / period \rfloor + 1$ 个时钟epoch之后(注: $L$ 是父到子链路的延迟, $period$ 是epoch长度或252个时钟周期. HAC是一个8位计数器, 但有4个值被保留用于特殊控制码.), 于时间 $t_4$ 退出同步循环. 最后, 在时间 $t_4$, 父设备和子设备都将发出一个(芯片本地的)`NOTIFY`指令, 开始同步计算.

为了支持扩展到多跳网络, 系统沿着HAC生成树的每一跳重复地进行基于`DESKEW`的同步过程, 总的同步开销为 $(\lfloor L / period \rfloor + 1) * h$ 个总周期, 其中 $L$ 是最大单链路延迟, $h$ 是生成树的高度. 注意, 这个开销只发生在分布式推理的开始阶段. 在程序执行期间, 使用下一节描述的一种更轻量级的(大约1个epoch)运行时重同步过程来调整时钟歪斜.

### 5.4 运行时去歪斜以实现重同步

上一节描述的基于HAC的方案为多TSP系统建立了一个共同的起始时间参考, 但由于多个TSP上独立时钟的频率不确定性而引起的TSP间漂移也必须被容忍. TSP提供了一个额外的**软件对齐计数器(SAC)**, 以允许TSP在计算过程中重新同步, 将累积的漂移保持在允许的公差范围内. SAC是一个与HAC周期相同的自由运行计数器. 然而, 与HAC不同, 它不会被更新以与上游HAC对应, 而是继续自由地计算本地时钟周期. 因此, 一个TSP的SAC和HAC之间的差值, 代表了自上次同步以来, "本地"时间视图(即SAC)相对于"全局"时间视图(即HAC)的累积漂移.

为了协调本地与全局时间, ISA中提供了一个`RUNTIME_DESKEW`指令. 该指令接受一个参数: 目标暂停的时钟周期数. 当执行时, TSP将延迟目标周期数**加上或减去** $\delta_t$, 即HAC和SAC的有符号差值. 如果 $\delta_t$ 为正, 意味着"本地"时间比HAC代表的"全局"时间快, TSP将暂停目标周期数**加上** $\delta_t$ 的时间; 如果差值为负, 则反之. 通过这种方式, "本地"时间被重新对齐到"全局"时间, 多TSP系统被重新同步, 累积的全局误差被减小到链路抖动的水平. 这个`RUNTIME_DESKEW`指令被调度为在网络中的每个TSP上同时执行. 在它执行期间, 每个TSP上的其他功能单元通过执行一个`SYNC`指令来使其ICU静默, 直到随后通过`NOTIFY`指令被唤醒.

## 6. 软件调度的网络(SSN)

TSP架构的一个基础特性是其**确定性的数据路径**. 所有指令的执行延迟在编译时是静态已知的, 因此通过ISA(指令集架构)暴露给编译器. 如前一章所述, 硬件将确定性执行的保证从单个TSP扩展到了一个多TSP系统. 在整个网络中实现这种确定性, 使得编译器不仅能对单个TSP内的所有数据移动有周期级的精确了解, 还能对连接TSP网络的链路上的数据移动有同样的了解.

在源TSP注入一个向量的精确时间(以周期为单位), 以及数据将到达目的TSP的精确周期, 都可以在编译时解决. 作者将这种网络范式称为软件调度网络(SSN), 因为它用在编译时调度张量取代了在网络中动态路由数据包的概念. 通过SSN方法, 所有对功能单元和网络链路的争用都在编译时解决. 在这一章将讨论使能软件调度网络的TSP多处理器系统互连网络的设计, 包括:

来自静态图分析的确定性流量模式

调度通信, 而非路由

确定性路由, 包括非最小化路由

无硬件仲裁和显式(软件)流控制

前向纠错(FEC)以避免响应式的链路层重传

### 6.1 流量模式先验已知

要启用SSN, 通信或网络流量模式需要是已知的. 对于多TSP所针对的大多数工作负载, 其通信模式, 无论在空间上还是时间上, 在通信发生之前的编译时都是先验已知的. 因此, 最佳的路由或"调度"决策可以基于该流量模式做出. 特别是, SSN利用静态计算图和对流量模式的先验知识, 来实现一种替代方案, 用于将张量作为一系列背靠背的向量(即网络中的流控单元, flits)进行路由. 这通过将单芯片的确定性扩展到整个TSP网络, 提供了TSP之间细粒度的(320字节)通信, 并为在网络链路上以时钟周期级精度调度张量奠定了基础.

流量模式是底层ML模型及其如何在处理元件间划分的涌现属性, 这种划分允许模型并行(即, 在TSP间分布不同的层)和数据并行(利用集群间的小批量并行). 模型的分解由编译器自动完成, 以将工作负载自动扩展到所需数量的TSP元件上. 编译器将工作负载划分为更小的子任务, 并将它们映射到负责执行的单个TSP上. 这个映射过程就产生了并行工作负载的流量模式.

对于流水线模型并行, 计算每个流水线阶段子任务的精确执行时间, 并在每个阶段的层之间交换激活值. 这个过程与传统方法有根本不同, 因为TSP知道每个阶段的精确执行时间(到时钟周期), 因此不需要动态分析(dynamic profiling)来提取子任务的运行时特性.

这使得并行分解步骤变得精确, 并完全在编译器的控制之下. 这种灵活性反过来又允许编译器的调度算法明确地进行计算和通信的权衡, 以改变计算与通信时间的比率, 并控制并行工作负载的表面积与体积比(surface-to-volume characteristics). 编译器将尽可能多地重叠计算和通信, 以有效地隐藏C2C链路延迟.

### 6.2 调度, 而非路由

路由算法决定了消息在网络中采取的路径, 通常通过查找表在硬件中实现, 通过检查数据包的目的节点为每个入包提供一个简单的出端口映射. 相比之下, 多TSP系统通过在源节点和目的节点上分别编排一系列的发送和接收指令, 显式地控制和调度消息逐跳通过网络的路径. 鉴于所有的数据移动都可以被静态推断, 编译器基于跨越时间和空间的全局信息来编排数据移动, 以消除对共享输出端口的冲突.

为了突显SSN的优势, 下展示了一个有四个TSP的例子:

![图片](assets/09e223c2e785.png)

通信的调度使得与传统网络相比, 一种替代的通信模型成为可能. 传统网络为系统中一对处理器A和B之间的通信提供了一个简单的抽象(例如, InfiniBand的QP抽象). 作为一个网络事务性本质的例子, 考虑一个简单的远程事务: 处理器A发送一条消息, 如"从处理器B读取地址X的值并回复给处理器A". 处理器B收到后, 发出一个DRAM**读操作, 并将回复发送回处理器A, 这产生了一次往返网络延迟. 如下图(a)

### 6.3 确定性负载均衡

Dragonfly拓扑的优势在于它提供了极高的路径多样性(path diversity), 特别是通过非最短路径(non-minimal routes). 要想充分利用这种路径多样性来避免网络热点和拥塞, 就必须依赖一个聪明的全局自适应路由算法.

![图片](assets/d3617b0673aa.png)

如果没有自适应路由, 流量可能会简单地沿着最短路径走, 一旦最短路径出现拥塞, 其他可用的非最短路径就被浪费了. 但是一个糟糕的路由决策(例如, 为了避开一个小的拥塞点而选择了一条非常长的路径)可能会比不进行自适应路由还要糟糕. 更严重的是, 多个独立的路由决策之间可能相互影响, 引发更复杂的网络问题. 作者引用了2篇论文, Indirect Adaptive Routing [ISCA’09]: 指的是一种经典的自适应路由算法, 它允许数据包在遇到拥塞时, 先被路由到一个中间节点, 再从中间节点路由到最终目的地. 而Phantom congestion [HPCA’15]: 指的是在自适应路由网络中, 路由器根据本地的队列占用情况来判断拥塞. 但有时, 队列的短暂排队并不意味着下游链路真的拥塞了, 路由器基于这种"幻象"做出绕路决策, 反而可能将流量导向真正拥塞的路径, 导致性能恶化.**引用这些学术论文是为了说明, 全局自适应路由是一个非常活跃但充满挑战的研究领域, 现有的硬件解决方案远非完美, 并且本身就带来了新的复杂性和不确定性.**

多TSP系统通过确定性地(deterministically)将负载流量(offered traffic)分散到可用的链路上, 来利用Dragonfly拓扑的路径多样性. 就像更传统的网络将一个消息内的数据包分散到胖树网络的可用上行链路一样, Groq是将一个大张量的320字节向量分散到源和目的TSP之间的C2C链路上. Dragonfly层次化拓扑丰富的路径多样性通过使用"非最小化路由(non-minimal routing)"来解锁, 以便将负载流量分散到每个TSP的多个注入链路上.

![图片](assets/c96650a5d441.png)

这基于负载量(tensor size)和对单个向量(flits)的精确调度, 对全局网络链路进行"负载均衡", 从而实现确定性的负载均衡. 在提供确定性负载均衡的同时, 这也避免了任何由力求达到相同目标的基于硬件的自适应路由所带来的端到端重排序问题.

与基于硬件的拓扑相比, 该网络拓扑的一个关键区别在于, "路由器"实际上是注入流量的"端点". 这是一个至关重要的区别, 因为注入的带宽量不再受限于注入带宽, 而是受限于交换路由器的带宽. 这也同样至关重要, 因为它要求非最小化路由是感知带宽的非最小化路由(bandwidth-aware non-minimal routing). 在一个硬件定义的Dragonfly网络中, 如果最小化路径上没有拥塞(或争用), 所有的包都会被最小化路由. 当感知到网络拥塞时, 路由可能会避开最小化路径, 转而走非最小化路径, 以使用其他未被充分利用的路径, 目标是减少延迟和提高整体吞吐量. 因此, 在硬件定义的Dragonfly网络中, 如果只有一个源和一个目的在通信, 非最小化路由是不必要的, 因为一个注入通道无法使单个网络通道过载. 然而, 在SSN中, 非最小化路由可以被用来为源和目的TSP之间提供更高的带宽量.

软件调度的路由仍然包含一个"决策"过程, 以确定数据包应该被最小化路由还是非最小化路由. 在先前利用硬件路由的工作中, 这些决策是基于路由器中可用的局部拥塞信息(如FIFO深度或传输信用)动态地对每个数据包做出的. 像TSP上的所有其他功能单元一样, 将路由决策在编译时做出, 使得编译器的调度算法能够最优地调度网络链路. Groq使用张量的物理数据量(即张量维度H×W×C的乘积)作为正在通信的数据量, 并根据张量的大小, 并选择用于分散流量的链路数量. 下图提供了随着消息大小变化和TSP机箱内非最小化路径数量变化时, 非最小化路由的性能增益分析.

![图片](assets/1e27fbe8d254.png)

该分析假设一个8节点全连接的TSP, 因此每个源-目的对有一条最小化路径和七条非最小化路径. 该分析提供了一种最小化和非最小化路由消息的最优分布, 使得总通信延迟最小化. 分析显示, 对于小于8kB的消息, 非最小化路由没有好处. 然而, 对于更大的消息, 非最小化路由的好处逐渐增加, 并且更多带宽(或更多非最小化路径)为更大的消息尺寸提供了更高的收益. 其中非最小化路由带来好处的实际交叉点是消息大小, 非最小化路径数量和每跳延迟的函数.

渣注
我们注意到上图中的结果:

小消息 (< 8kB): 传输时间本身很短. 此时, 走非最小化路径增加的跳数延迟(成本)超过了并行传输节省的时间(收益), 所以得不偿失. 编译器会选择只走最小化路径.

大消息 (> 8kB): 传输时间占主导. 此时, 将数据分发到多条路径上并行传输带来的时间节省(收益)远远大于增加几跳的延迟(成本). 消息越大, 使用的并行路径越多, 收益越明显.

"交叉点是...的函数". 这句话表明编译器的决策是一个复杂的优化问题, 需要综合考虑消息大小, 可用的非最小化路径数量, 以及每条路径的总延迟. 这就构成了一个编译器优化问题.

### 6.4 Flow Control

了启用SSN, 互连网络不允许动态仲裁(dynamic arbitration), 因为这会导致不确定性, 并使编译器无法显式地调度和步调(pace)每个链路.

除了在C2C逻辑和TSP核心时钟边界之间的接口处有非常浅的缓冲区外, 也没有传统数据中心网络中常见的"队列"或缓冲区.

此外, 虚拟通道(VCs)通常用于避免在使用非最小化全局自适应路由时产生的路由死锁, 并保证循环依赖不会发生. 使用软件调度路由, 数据包之间的循环依赖仍然可能发生; 然而, 路由死锁的根本原因在于数据包在请求另一个资源(如下游缓冲区)时还占有着一个资源(如缓冲区或VC). 使用软件调度网络, 数据包或消息是提前调度的; 因此, 数据包在请求另一个资源时不会占有当前资源, 路由死锁不会发生, 因此也就不需要VC.

在Groq的论文中Local Group有两种拓扑, 除了前述的Fullmesh(下图中的a)以外, 作者还认为在推理模式中可以使用Torus拓扑:

![图片](assets/c631320d132a.png)

考虑Torus networks中的环路死锁(toroidal deadlock)场景是由于Torus Link周围overlapping VC依赖关系引起的. Local Group Radix-8环形拓扑使得在使用流水线模型并行进行推理时, 能够与相邻TSP进行高效的最近邻通信. 对于Local Group Radix-8拓扑, 在Torus内三条连接物理链路以增加最近邻吞吐量. 软件调度方法避免了为避免死锁而增加额外的虚拟(或物理)通道所带来的复杂性和成本.

调度和缺少硬件流控制的一个好处是, 数据不再需要被封装成数据包和消息来穿越网络; 相反, 所有关于数据移动的信息都已编码在编译器产生的指令流中, 整个Packet的格式如下, 可以看到在header/tail flit中包含了指令,每个向量只有2.5%的编码开销.

## 7. RAS

Groq认为: 确定性硬件使得一种软件定义的方法来实现RAS成为可能.在一个确定性的系统中, 任何一个时间点, 硬件应该处于什么状态, 是完全由编译器预先知道的.因此, 任何偏离这个预定状态的硬件行为, 都可以被立即识别为"异常"或"故障".

在芯片级RAS上, 使用了ECC和内部的冗余superlane:
![图片](assets/0732b804c836.jpg)

另一方面在系统级设计了每个节点的冗余电源, 并且每个系统也有冗余的节点. 并且互连也有冗余的链路. 为了使用N+1冗余, Dragonfly拓扑的边和节点对称性(edge and node symmetric)是关键, 它使得网络在(替换节点后)仍然保持全连接(即系统中每个源-目的对之间都存在路径), 这使得这种N+1弹性方案变得切实可行. 该策略允许运行时监控系统健康状况, 并在运行时层为调用并行程序执行而调度资源时, 用备用节点替换任何不可用的节点. 通过为每个系统配置一个冗余节点, 可以降低这种开销, 例如, 一个有四个机柜的33节点系统, 可以将33个节点中的1个作为备用(将开销从11%降低到3%), 留下32个节点(256个TSP)用于执行并行程序. 通过配置热备份节点并提供软件重放机制, 我们可以通过让运行时重放推理来从严重故障中优雅地恢复(gracefully recover).

### 7.1 FEC和软件重放机制

像PCIe这样的点对点网络通常采用一种链路层重传(link-layer retry)机制在链路层重放数据包, 以便网络层或应用层不会观察到错误. 不幸的是, 这种链路层重传机制也引入了不确定性行为(non-deterministic behavior), 因为它改变了重传数据包的预期到达时间, 从而干扰了系统的全局同步.

相反, 为了在面对传输错误时保持确定性, Groq在每条链路上使用FEC来纠正简单的传输错误, 并检测不可纠正的突发错误. 这使得系统中任意源-目的TSP对之间的点对点物理链路集合保持确定性. 并且在网络中逐跳路由数据包, 原位(in situ)纠正任何传输错误, 并标记任何需要运行时系统"重放"推理的严重错误(即软件重放, a software replay), 以确定该故障是瞬时性的(在重放推理后消失), 还是在重试后仍然存在并需要物理干预(例如, 更换有问题的线缆, 电源单元或TSP卡)来修复故障.

![图片](assets/98996fa11ede.png)

对于异常处理, 在运算过程中出现的NaN 和 Inf 可能会由于SRAM或关键硬件结构(如矩阵单元)中的SDC(静默数据损坏)或SEU(单位事件翻转)而出现. 通过允许编译器在编译时决定算术错误的语义, 来避免算术异常.

![图片](assets/2ad942141296.png)

发生错误时的一次性重放允许运行时系统检查故障是瞬时的(可解决)还是持久的, (如果是持久的)则必须在另一组硬件资源上重放该推理任务.

### 7.2 ECC

并行计算机的规模**系统中处理元件的最大数量**受限于系统的可靠性, 因此在整个芯片中使用了ECC

![图片](assets/30f7774c7a7b.png)

TSP处理元件使用确定性的数据路径, 对于指令缓存队列和superlane上数据流的所有SRF使用了SECDED(single-error-correction and double-error-detection), 即对所有单位元错误(SBEs)进行纠正, 这些错误由TSP硬件原位纠正; 并且可以检测所有多位元错误(MBEs), 以便运行时软件可以在识别出严重错误时, 在一组已知的良好硬件上重放推理.

另外在SXM上仅使用了奇偶校验, 这可能是一个设计上的权衡. SXM主要负责数据的移动和重排, 数据在其中停留的时间相对较短, 发生软错误的概率较低. 奇偶校验的硬件实现比ECC更简单, 面积和功耗开销更小. 它虽然不能纠错, 但能可靠地检测出错误. 一旦检测到错误, 就可以触发异常, 同样能保证系统的最终正确性.

## 10. NV如何吸收Groq的技术

### 10.1 NV GPGPU的问题

其实老黄自己都在内部转过的[《Inside Nvidia GPU: 谈谈Blackwell的不足并预测一下Rubin的微架构》](https://mp.weixin.qq.com/s?__biz=MzUxNzQ5MTExNw==&mid=2247496740&idx=1&sn=c9403138fa59d126fe6cfda19d9b2f76&scene=21#wechat_redirect)这篇文章已经阐述的比较清楚了.

大致的思路是当GPU的Tensor Core和CUDA Core之间以及TMA都采用异步通信的方式, 利用MBarrier同步时, 如果采用WarpSpecialization在CUDA Core上运行的效率和代价都是非常高的.

那么实际上就是需要一个标量核来处理Warp调度和指令派发, 实质上和Groq的指令控制单元(ICU)的架构是不谋而合的.

### 10.2 是否真的需要确定性?

NV GPGPU实质性的问题还是通信延迟较高带来的影响, 正如前文所述, DSA一类的通常就是一块很大的SRAM, 而GPGPU通常有更深的内存层次化结构.

![图片](assets/d9716f04ede4.png)

整个计算在CUDA Core和TensorCore上有多次内存的搬运, 每一块专用的内存(RF/SMEM/TMEM)都有各种各样的限制, 相对于Groq的Streaming RF和确定性的编译处理, 需要更大的buffer来打满吞吐.

另外一个问题是, 通常在使用跨CGA/跨NVLINK进行一些集合通信的时候, 多种通信流量干扰也带来了一些不确定性和长尾. 事实上DSA架构相对独立的一个SRAM是有很大优势的.  我们来看看AWS Trn3, 实际上它没有那么极致的追求确定性, 但允许SRAM 到另一块卡SRAM的直接拷贝, 以及在ScaleUP Link上执行流量整形的处理实际上也是在进一步降低延迟和长尾.

对于NV的架构, 毕竟CUDA生态已经快20年了, 完全推倒重来, 或者在推理上另外再开一套技术栈也是不现实的.

其实我一直期望的架构变化大概有两点:

L2Cache中可以动态分配一部分作为用户控制的SRAM, 然后构成一个大规模的Device Level的ScratchPad.

对于很多通信的完成事件, 需要一个较好的机制从NVLink能够通知到SM内的SMEM上并修改MBarrier.

个人对于完全的确定性处理还是持有怀疑态度的, 是否真的需要这样的大规模数千颗芯片的确定性. 同步的代价, 时钟树的Skew到了一定规模下也都会产生问题. 可能在不破坏NV现有架构的基础上, 对于通信这些抖动最大的地方进行消除才是更有必要的. 这也是我在设计eRDMA的多路径拥塞控制算法时一直强调的一点. 而对于ScaleUP网络, 通过在I/O Die上添加一小块内存, 也可以提升很多确定性, 然后将指令和通信自包含在一起更快的触发计算, 这些做法和Groq也是基本上一致的.

只是NV的网党(Mellanox)是否对计党微架构有这方面的认知就很难评估了...

### 10.3 CUDA Core的演进

现在CUDA Core的Warp调度, 自身的RF, 在SM内部的TMEM/SMEM/L1-Cache都带来了很多数据搬运的复杂度. Groq按照320B的长度做向量化的处理, 将MEM/VXM/MXM/SXM等各种FU通过Streaming的RF连接在一起. 整个NOC上按照320B位宽对齐的, 效率上应该也会高于GPGPU中SM到NOC的位宽, 然后320B的传输对于FEC

然后传输延迟上因为支持SRAM到SRAM的快速拷贝以及在NOC上实现了Virtual Cut-Through的转发方式, 整体的传输Overhead不大并且延迟也低了很多, 相对于很多传统的Store-forward的交换机, 也可以进一步降低延迟. 同时整个完成通知机制可以通过SYNC/NOTIFY指令实现, 比起传统的Polling MBarrier也有一定的优势.

对于NV而言, 单颗芯片面积受限的情况下, 做大了TensorCore的算力引入了TMEM, 使得单芯片的SM数量反而减少了很多, 因此如何进一步优化CUDA Core的PPA, Groq是可以有一些参考的.

### 10.4 编译器视角

从编译器的视角来看, SM内部RF的处理还是有很大的复杂度的, 另一方面我们注意到cuda tile, cute-dsl, tilelang这些新的Tile-based DSL出现. 另一方面从以往的Distributed-Triton到现在的Tile-Scale以及NV的一些Distribute-GEMM相关的工作, 我们看到在多芯片构成的ScaleUP机柜内, 编译器有了更好的发挥空间. DSL本身也是非常重要的一环, 因为大量复杂的内存异步访问/Tensor Layout都是非常适合DSL去隐藏复杂性的.

另一方面是关于静态图编译, 其实最先想到的是Alpa的工作, 其实我一直很看好这块工作的, 可惜后面没有继续做下去. 而很高兴的是看到最近几天有《nncase: An End-to-End Compiler for Efficient LLM Deployment on Heterogeneous Storage Architectures》[5], 挺值得去仔细分析一下的. 另一方面是薛老师的TileRT相关的工作, 也挺有意思的.

当然还是有一些怀疑, 例如潜在的应用/算法变化, 不知道是否能够适应Groq这样的确定性的静态图编译?

### 10.5 3D架构

Nvidia以前也针对3D封装有些资料:

![图片](assets/6d4f16b85b6f.png)

我们注意到Groq在后续的Roadmap上也在讨论一些3D Stacking的技术, 特别是Logic-on-Logic时, 如何通过编译器以及确定性执行来控制功耗和散热, 这也是一个非常值得探索的路径.

另一方面即便是没有Logic-on-Logic, 使用一些3D Stacked SRAM配合周边的一些chiplet的计算Die, 类似于AMD的X3D, 似乎也是一条路径.

从NV的角度, Groq的MEM FU的一些技术还可以用在HBM的Base Die上, 应该也可以做出一些有趣的东西出来. 例如将L2Cache中置换为部分的Scratch Pad SRAM. 这样对于SM而言, 可以把HBM Base Die当成另一个TMA引擎使用, 去隐藏HBM的延迟. 从Tile-Based的编译器角度这条路应该也可以尝试一下, 甚至在Base-Die上还可以进一步在外部挂接一些低速的DDR来扩展容量.

### 10.6 关于通信

Groq的MEM FU的抽象是做的挺好, 另一方面整体的编码的效率也很高并且支持Cut-Through来低延迟转发. 然后也有直接的指令封装在FLIT触发计算, 降低通知交互延迟.

从本质来看, 它实际上也是直接在网络上Attach了一块Memory, 在几年前我也做过NetDAM, 同样的处理方式, 并且也能获得确定性的延迟, 这个测试是在两块FPGA并且中间连接了一个以太网交换机后得到的, 可以看到其延迟仅427ns, 

![图片](assets/71df4c57452c.png)

如果中间交换机换成Cut-through的, 应该可以把延迟进一步压低到300ns左右, 或者去掉交换机,  应该就和Groq 220cycle(按照900MHz频率算, 延迟为253ns)差不多了, Groq的延迟结果:

![图片](assets/e9fe806c4a88.png)

同样的也是在Packet中封装相应的Instruction:

![图片](assets/28947c3ddd6a.png)

例如Ring-Allreduce这样的操作可以一条自包含的指令沿着多颗芯片Streaming的完成, 整体小包延迟远好于现在NV的实现.

当然NetDAM这么多年过去了一直没有打算商用落地的原因是, CXL的进展太拉跨了, 然后过去几年又在ScaleOut上处理RDMA相关的事情, 而ScaleUP的标准也一直处于大量争议之中. 或许26年对于ScaleUP标准有了一个结论后应该可以做一些相关的工作了.

### 10.7 从稳定性的角度

其实还有一个视角, 例如当系统出现故障, 我们需要热迁移时, NV的GPGPU架构中有多块内存,大量的RF/SMEM/TMEM需要迁移到另一块卡时, 处理非常复杂并且非常缓慢. 而基于DSA这样的架构, 只需要对一整块SRAM进行拷贝, 然后对于流水线中的Streaming RF在软件上Replay即可. 同时对称的拓扑结构使得Groq构建N+1冗余变得更加容易, 这些因素是值得NV去考虑的..
