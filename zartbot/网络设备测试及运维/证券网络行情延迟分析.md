### 传输延迟 

光在光纤中的传输速度基本上为真空中光速的2/3,即200,000km/s，因此我们根据实际的物理链路来看，光纤传输数据延迟如下：

![图片](assets/22dccc68e5d2.png)

正也是这个原因，高频交易商要建设微波通信的原因，但是微波通信会有季节性的影响因素导致一些季节丢包比较厉害。而传输延迟上由于线路速率不匹配或者编码方式不同可能导致Store-foward的处理，同时一些高速链路上由于Re-timing或者RS-FEC也会增加数百个ns的延迟，

### MicroBurst 

其实很多交易所都会产生MicroBurst，这也是证券行情网络中最难处理的一环，而国内这种行情快照的发送方式更加容易带来影响：

![图片](assets/931087ffec41.png)

很多时候看到秒级或者分钟级统计时，行情网络带宽并不高，但是调到毫秒级的刻度，基本发现链路全被占满，大量的网络设备Buffer被占用甚至溢出丢包。通常是由于行情服务器设计不当产生的，它会按照接口线速发送大量行情数据。

这些影响在数据中心侧由于组播复制并不会在同一条链路上出现两份，因此影响微乎其微，但是在交易所连接客户的广域网路由器上就成了一个非常难解决的难题，通常交易所采用几条MSTP线路通过子接口连接大量的会员单位，因此大量的组播行情需要在这个接口上传递。

![图片](assets/c8960125f856.png)

Colo机房可以使用交换机或者分布式路由器进行大量的包复制，例如思科的1层交换机了解一下，但是本文重点谈的广域网

![图片](assets/77a26b0a1bc7.png)

但是广域网这个场景，这些设备的复制能力就完全无效了，因为都是子接口需要顺序发...

![图片](assets/231c23edbbbd.png)

而通常这样的转发每个包都还有不同的VLAN头，通常只能使用多核心的网络处理器来做，因此整个复制过程受到如下的一些硬件架构的影响

### 交易所视角 

对于交易所而言，最关键的一点是：保序和公平，延迟都是其次的，因此我通常建议交易所网络架构设计时将行情、交易、清算三个网络独立开来，通过独立的消息总线进行处理，降低干扰。

![图片](assets/abf9d9eb9c20.png)

主要分离设计原因:

报盘网需要有更多的安全和合规监控策略

行情网要求完全公平并降低延迟

行情报盘混跑无法精确控制延迟和降低抖动

另一方面，很多分布式设备并不适合交易所网络，因为公平性有影响：

![图片](assets/39b03a107c6b.png)

### 组播延迟测量 

通常我们的测试拓扑如下，需要按照如下方式构建网络，检测设备跨板和同板不同端口的延迟差距：

![图片](assets/0bfd784fd92b.png)

当然专业的设备如IXIA也有可以用来测试，更高精度的测试可以使用思科的Exablaze的75ps精度的测试方案，但很多券商和交易所自己并没有这样的设备，我们可以通过一个支持ERSPAN Type-3提供100ps精度的Nexus 93180FX交换机来实现，然后通过DPDK写一个接收带有ERSPAN时间戳和VLAN-ID的报文处理软件来计算延迟并评估乱序。

组播源也是我为交易所定制的一段程序,每个报文的前64bits定义了一个时间戳，可以定义每个Tick的间隔时间，每个行情报文间的gap，每次行情快照的包数量、以及每次行情的包尺寸大小，同时也可以定义一个tick需要分成多少个Bacth来发送，当然这个程序只提供给交易所使用。

```
kevin@latency3:~$ ./mcast_sender -hUsage of ./mcast_sender:  -batchsize int        Number of batch size  (default 1)  -dst string        Destination IP:port (default "239.1.1.1:8000")  -len int        Packet Length (default 256)  -num uint        Number of packets per tick (default 500)  -pktgap int        Interval between  multicast packet(nanosecond) (default 1000)  -src string        Source IP:port (default "1.1.1.4:8000")  -tick int        Tick interval(msec) (default 500)
```
