#### 协同(Coordination**)

如何使用一个有数千台计算机组成**的超大规模系统顺畅的工作,需要在协议上下功夫，而这些协议又刚好穿插在计算、存储和网络之间，另一方面就是从计算的模式上来看，由传统的单机计算演进成了Cloud SaaS和终端小程序的模式，体系架构上需要一个非常微妙的平衡，所以在设计Ruta的时候则是从整体效率上考虑，如何高效的利用资源,从某种角度就是降低数据本身的`物流`成本

![图片](assets/6b1610f8d3e0.png)

#### 评估(Evaluation)

DPU存在的价值是需要好好进行评估的，因为最近和很多人都聊到这个问题，加一块DPU对于主机成本上升了不少，但是是否能从CPU中压榨出来DPU的价值，或者纯软件的方案浪费一个CPU Core来处理DPU的业务是否可行? 这些就是本文下一个章节需要重点讨论的问题.

#### 设计(Design)

这部分就是一些算法和架构的问题了，解耦还是耦合，松耦合还是紧耦合?都是艺术...

### DPU的刚需：从WSC测量看起

Google有一篇paper讲到< Profiling a warehouse-scale computer >, 文中以Datacenter tax的方式描述了以下几个场景对CPU的损耗:

#### Protobuf Mgmt

Protobuf是Google内部的数据存储和传输的通用编码方式，但是数据的串行化(Serialization，外行金融民工也可以理解为编码)和解码(Deserialization)都需要占用大量的CPU资源,所以Google自己都有了Build SoC来解决这个问题的动机，但是它的位置在内存和CPU末级缓存之间. 这样一来，大量的现有的以PCIe为主的DPU就全部死翘翘了..所以这也是我前段时间一直在研究的DataCentric-ISA处理的地方:
![图片](assets/d0d007f060b7.jpg)
即类似于P4的Parser和DeParser需要跟CPU紧耦合，所以你们也看到IBM在大机上将压缩、加密等引擎都作为协处理器整合到了CPU中.
![图片](assets/2705b50e1026.png)

#### RPC

另一个耗费资源的东西就是RPC了，远程过程调用（Remote Procedure Call）的缩写形式，通常一个RPC Call需要通过TCP协议，绕过一大堆网络组件

![图片](assets/2d1d27bc09bc.png)

核心问题是一个通信问题，因为远端服务器调用本质上的开销是固定的，能缩减的只有在网络上和编解码上.降低延迟和提高吞吐是这类问题的关键.当然这类问题可以将`通信`抽象成一个`共享内存`的过程.

比较有代表性的是基于DPDK或者RDMA实现的eRPC,可惜对多种语言的支持不好导致其生态不好，另一个做法就是开源社区基于VPP实现的Memif和HostSocket.当然还有IBM在大型机上很多年前就实现了的HyperSockets.

#### Data movement

`memcpy()`和`memmove()`函数，对金融民工来解释这个问题就是，内存不是打包直接发专列的，而是需要一个个字节去搬砖的, 如果这个场景中有专门的硬件代劳不是让CPU一块块搬砖不是好很多么？

```
void * __cdecl memcpy (void * dst, const void * src,size_t count){        void * ret = dst;        while (count--)         {                *(char *)dst = *(char *)src;                dst = (char *)dst + 1;                src = (char *)src + 1;        }        return(ret);}
```

#### Compression

存储空间的瓶颈、通信带宽的瓶颈都是压缩出现的地方，IBM zEDC Express开始到后面整合到CPU中，同样Intel qat也在做压缩类的支持

#### Memory Allocation

内存的分配上，现在软件实现通常是一种递归的数据结构，而这些操作是非常容易并行的在专用ASIC**上实现的。但是这样需要在CPU和内存之间做出一些有趣的东西.取舍也很大，甚至是反智的.

#### Hashing

加解密、查找等多种算法都需要Hash,这些东西是专有的硬件拿出来做还是扔给CPU增加新的协处理器做呢？

#### Kernel

原文的观点如下，他们也认为这是无法加速的税负

However, we do not consider it acceleratable tax – it is neither small, nor
self-contained, and certainly not easy to replace with hardware

#### CPU微架构

其实我们自己也有争论，在设计第三代QFP的时候，是多个低频率的核，还是少量高速的核. 一般来说有一个平方公式，即单核频率比为倍，则实际处理能力比为.  另一方面的问题是，现在很多程序越来越大，经常发现数据中心CPU上出现I-Cache不够的情况，当然这些跟CPU架构相关就不多做解释了。

### DPU投资决策

综上几个问题来看，在PCIe边缘上搞事情的厂家基本上都是没搞明白DPU需要做啥的，一上来就是对标`Pensando`什么的，你不知道P这次也没完全的看清楚? 但是`Pensando`的ASIC架构还是很有趣的可以干很多有趣的事情，只是他们还暂时不知道怎么去玩罢了，毕竟一个从网络出来的厂家离计算还是太远了。

另一个场景是HPC的，最近也有好几个大厂在问我要不要去实现SHARP来对标卖螺丝？实际上nvidia做DPU的最根本目的是下面这个图:

#### 分析

从通用性上来看，找一些DPU厂商能够满足二线公有云的果金属需求的，同时又不太容易被Intel、Marvell、Mellanox替代的，或许一年能有个几万片的生意，但是要做到投资方高估值上市是非常困难的，其实最怕的就是这种企业搞定某个二三线云厂商的集采，每年有固定的量，但是又做不大又做不死的僵尸状态...

而一线云厂商基本上都会自己做，而且注定了这样的部门成本摊薄在了整个云上，可以养得活，所以我对要自己去做ASIC流片的创业厂商几乎都不看好。

而做FPGA的厂商则存在需求更新缓慢和成本太高的问题，而且一些复杂逻辑还是需要片上有一个通用处理器核来处理。

![图片](assets/294cbca0e0a1.jpg)

#### 建议

从DPU创业来看，一方面还是需要在协议层面上去多想想，从体系结构上去优化以下。例如我们最近研究的一个项目很快就会发布出来，体系结构的创新决定了战局，而在传统的体系结构上加东西迟早是要把自己玩死了。我倒是建议从软件上起步，尝试一下DOCA，尝试一下DPDK，从这个角度在Intel、Marvell、nVidia比较成熟的DPU芯片上先开发，然后针对一些特殊场景构建一条高端的FPGA产品线，当然Xilinx和Intel也希望多开源一些原厂的参考设计和培训，让更多做软件的人能够了解硬件，同时也能让硬件的团队更加容易了解软件的瓶颈。至于ASIC流片，说句实话DPU真的非常复杂，甚至超过一颗CPU。当然海思出来不少做SoC的兄弟这也算是一条好的赛道，唯一缺少的就是一个软硬件通吃的架构师了:)

#### 边缘云DPU

实际上这是一个完全被忽视的场景，DPU玩家都盯着WSC这样超大规模数据中心，而在边缘云场景中DPU有大量的用途，但前提故事是你不能按照DPU的概念去使用它. 无论是5GC还是工业云，其实都需要大量的边缘计算资源，而如何在端-网-云上通过体系架构的创新来解决这些问题，例如Serverless on DPU这些场景，如何避免SideCar等额外的开销，如何避免冷启动延迟.

最终还是回到开头那句话:

做好数据本身的`加工和物流`才是整个新基建链条中最为重要的一环,不碰内存子系统只在网络和PCIe上瞎搞的公司都是XX！
