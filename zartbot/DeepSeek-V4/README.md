## 第一部分 · 架构篇：一百万 token 的效率革命

DeepSeek 官方给 V4 定的标题是《Towards Highly Efficient Million-Token Context Intelligence》——**迈向高效的百万级 token 上下文智能**。一句话就把野心说清楚了：不是比谁更聪明，而是比谁能在**一百万 token 的上下文**下，还把算力和显存的账算得过来。

这篇笔记覆盖第一篇（算法和模型结构）。你读完能回答四个问题：V4 长什么样、它凭什么把长上下文的成本砍到脚脖子、mHC 和混合注意力这两个新玩意儿到底在干嘛、以及为什么它换了个叫 Muon 的优化器。

---

### 一、先看全貌：两个型号，一张成绩单

V4 一口气发了两个型号，一大一小：

| | V4-Flash | V4-Pro |
|---|---|---|
| 总参数 | 284B | 1.6T |
| 激活参数 | 13B | 49B |
| hidden_dim | 4096 | 7168 |
| 层数 | 43 | 61 |
| 路由专家数 | 256 | 384 |
| 激活专家数 | 6 | 6 |
| 专家中间层 | 2048 | 3072 |

Pro 在 Coding/Agent 任务上基本追平了 Claude Opus 4.6，但 API 价格便宜一大截（输出每百万 token 3.48 美元 vs Opus 的 25）。对咱做 Infra 的人来说，真正的看点不是分数，而是**效率账**：

在 100 万 token 上下文下——
- **V4-Pro**：单 token FLOPs 只有 V3.2 的 **27%**，KV cache 只有 **10%**；
- **V4-Flash**：更极端，FLOPs **10%**，KV cache **7%**。

这不是调参调出来的，是三个架构级改动硬砍出来的：**mHC**（换了残差连接）、**混合注意力 CSA/HCA**（换了注意力）、**Muon**（换了优化器）。剩下的 MoE 主体基本沿用 V3，只做了微调。

> 渣注里有句大实话：「除了对 Infra 团队不友好以外，这是一个非常不错的工作」。V4 把算法复杂度换成了工程复杂度——省下的算力和显存，都变成了推理框架里要啃的硬骨头。这也正是第二篇 MegaMoE 要收拾的烂摊子。

### 二、地基：从 V3 继承了什么

要看懂 V4 改了什么，得先记住 V3 那套地基（详见 [[deepseek-v3]]）：

- **MLA（Multi-head Latent Attention）**：不存完整的多头 K/V，只存一个低秩 latent 向量，推理时再上投影解压。这是 DeepSeek 从 V2 就开始省 KV cache 的看家本领（详见 [[mla]]）。
- **DeepSeekMoE**：细粒度路由专家 + 共享专家。V3 是 256 路由 + 1 共享、top-8。
- **DSA（Dynamic Sparse Attention）**：V3.2 加的。用一个轻量的 **Lightning Indexer** 给每个 query 动态挑 top-2048 个最相关的 KV token，其余跳过——128K 上下文的注意力计算量直接从 128K 压到 2K（详见 [[dsa]]）。
- **MTP（Multi-Token Prediction）**：多 token 预测，做投机采样的底子。

V4 在这套地基上，改了三样，留了一样：

**改：路由打分函数** —— 从 V3 的 `Sigmoid` 换成了 `Sqrt(Softplus)`。为什么？
- V2 用 Softmax，专家之间是零和博弈：一个分高，别的必然被压低，容易「专家坍缩」——少数专家被反复选中，其他饿死。
- V3 换 Sigmoid，每个专家独立打分互不影响，非竞争，训练更稳。但 Sigmoid 有个毛病：当模型对某专家偏好极强时，梯度饱和趋近 0，想再微调都推不动。
- V4 的 `Sqrt(Softplus)`：输出无上界，区分度更好；`sqrt` 再压一压防止权重过度集中；而且**梯度永远不为 0**，路由永远能被微调。工程上还加了个阈值 20 防 `softplus` 溢出。

**改：前几层用 Hash 路由** —— 最初几个 Transformer 块，把 Dense FFN 换成了 MoE，但走的是**哈希路由**：一张固定的 `[词表 × 激活专家数]` 查找表，按 token ID 直接查该激活哪几个专家，不经过 gating 网络。省掉了浅层路由的开销。

**改：去掉了路由节点数上限** —— V3 为了通信友好，限制每个 token 最多路由到几个节点。V4 把这个限制拿掉了，代价是并行策略要重新精心设计（这是第二篇的活）。

**留：MTP 原样保留** —— V3 已验证有效，V4 一个字没改。

### 三、mHC：给残差连接上一道数学保险

**一句话**：mHC 是把 Transformer 的残差连接，从「一根信息管道」扩成「四根并行管道」，并且给管道之间的信息重组矩阵套上一个数学约束，防止它在深层训练时爆炸。

**先说 HC（超连接）的动机**。标准 Transformer 的残差就是 `x + layer(x)`，一根线往下传。HyperConnection 的想法是：在**深度维度**传更多信息，把残差流的宽度扩成 `n×`（V4 用 n=4）。相当于原来一根水管，现在并排铺四根，层与层之间这四根管子还能互相串水、重组。这给了模型一个和 hidden_dim 正交的新扩展轴，而且开销极小。

**但 HC 有个致命伤**：管道之间的重组矩阵是**无约束**的。矩阵一层层连乘，会产生指数级的放大或缩小——信号要么炸掉要么消失。原文那张图很直观：HC 训到约 12k 步时 loss 突然飙升，和梯度范数的不稳定高度相关。

**mHC 的解法**：把这个重组矩阵**约束到「双随机矩阵」上**（行和、列和都为 1 的非负矩阵，数学上叫 Birkhoff 多面体）。双随机矩阵的谱范数有界为 1，所以变换是**非扩张的（non-expansive）**——信号过一层不会被放大，天然稳定。而且双随机矩阵在乘法下封闭，堆多少层都稳。

怎么把一个任意矩阵投影成双随机矩阵？用 **Sinkhorn-Knopp 算法**——名字唬人，其实就是**交替做行归一化和列归一化**，迭代几次就收敛：

```python
for _ in range(sinkhorn_iters):   # V4 用 20 次
    comb = comb / row_sum(comb)   # 行归一化
    comb = comb / col_sum(comb)   # 列归一化
```

> 渣注点破了一个认知陷阱：论文里「流形约束」这个词把概念放大了，容易让工程师高估复杂度。**本质就是个双随机矩阵 + 行列归一化**。作者自己也说「20 次就够了，保证不炸就行，工程上怎么简单怎么来」。
>
> 但有个更深的暗示值得记：Sinkhorn-Knopp 同时也是求解**最优传输（Optimal Transport）**的核心算法。所以 mHC 可能不只是个稳定器——它在学一种「把上一层的 4 路残差流，以最小成本重新分配给下一层 4 路」的最优传输方案。4 个通道像 4 个生产者，要把「质量」最省地送给下一层 4 个消费者。

### 四、混合注意力（核心）：SWA + CSA + HCA，三张不同比例尺的地图

这是 V4 长上下文效率的命根子，也是最值得讲透的一节。

当上下文冲到百万级，**注意力就是那个二次复杂度的计算瓶颈**。V4 的破法不是单一手段，而是**三种注意力交错叠加**，每种负责一个「比例尺」：

#### SWA（滑动窗口）——高清局部地图
每个 token 只关注最近的 128 个 token（未压缩、原始精度）。成本极低，精确捕捉词语搭配、句法这些**短程依赖**。缺点：完全看不见远处。

#### HCA（重度压缩注意力）——低清全局概览图
把连续 **128** 个 token 的 KV **压成 1 条**摘要，然后在这些摘要之间做**全注意力（dense）**。因为对象数从 n 掉到 n/128，即使全连接成本也不高。它给的是「文档大陆板块」级别的宏观结构——哪几段属于同一主题、板块之间什么关系。缺点：细节全被压没了。

#### CSA（压缩稀疏注意力）——中清交通干线图
两步走：先把连续 **4** 个 token 压成 1 条（轻度压缩，带 overlap 让块边界更平滑），再套 V3.2 那套 **DSA**——用 Lightning Indexer 给每个 query 只挑 top-k 个最相关的压缩块算注意力。它给的是「忽略城市内部毛细血管、但清楚标出连接远方城市的高速主干道」这种中等分辨率的稀疏骨架。

**三张地图叠起来看**：模型处理每个 token 时，同时参照高清局部（SWA）、低清全局（HCA）、中清骨架（CSA），既不会「只见树木不见森林」，也不会「太宏观丢细节」，还能靠 CSA 抓住决定性的长距离依赖。V4-Pro 的层配置是 **31 层 HCA + 30 层 CSA + 1 层 MTP**，CSA 和 HCA 逐层交错，一层看宏观、下一层带着宏观认知去中观找连接，层层迭代把这张「世界地图集」画细。

#### 几个务必记住的工程细节
- **共享 KV 的 MQA**：压缩后的每个 KV 条目同时当 key 和 value 用，且所有 query 头共享同一份 KV（MQA 方式）。因为压缩后 KV 的 seq 维和 Q 的 seq 维不一样，MQA 是最省事的处理。
- **分组输出投影**：V4 的注意力头维度很大，直接把输出投回 hidden_dim 计算量爆炸。所以先把输出分成若干组，每组投到一个低维中间输出，再投到最终结果，分两步摊薄成本。
- **部分 RoPE**：RoPE 只作用在每个向量的**最后 64 维**，前面的维度不带位置编码。而且因为 KV 条目既当 key 又当 value，作者对 value 输出也补了一次位置 RoPE，让注意力输出携带相对位置信息。
- **Attention Sink**：借鉴 GPT-oss，给每个头设一个可学习的 sink logit 加到注意力分母上，允许某个 query 头把总注意力调到不等于 1、甚至接近 0（详见原文引用的 Attention Sink 分析）。

> **数学视角（选读但推荐）**：原文用范畴论的 **Nerve 构造**给了这套设计一个优雅的解释。把序列看成一个范畴（token 是对象，注意力关系是态射），完整注意力等于构建一个稠密的高维单纯复形，复杂度随 n 爆炸。而：
> - HCA = **商范畴的稠密 Nerve**（把 128 个 token「捏合」成一个街区，只算街区到街区，抓 0 维同调=全局连通分支）；
> - CSA = **保持同伦等价的稀疏化**（中等分辨率 + 只留骨架边，尽量保住那些代表长程语义的「洞」和「环」）；
> - SWA = **局部 Nerve**。
>
> 三者合起来就是对「理想中那个算不动的完整语言 Nerve 空间」的一个**多尺度单纯逼近**。Top-k 选择在数学上不是最优（理想是持久同调），但它 GPU 友好、易并行——是拓扑保真度和计算可行性之间的务实妥协。

### 五、效率账：省下来的 10x 和 3.6x 是怎么来的

光说省了多少不够，得看账怎么算的。V4 的省，来自**两把刀**：压缩注意力（砍 seq 长度）+ 低精度（砍每条的字节数）。

**KV cache 账**（V4-Pro vs V3.2）：

| 序列长度 | V3.2 | V4-Pro | 节省 |
|---|---|---|---|
| 8K | 367.9 MB | 55.3 MB | 6.6x |
| 128K | 5886 MB | 576 MB | 10.2x |
| 1M | 47092 MB | 4464 MB | **10.5x** |

节省比随长度增大——因为压缩注意力砍的是 seq 那一维，上下文越长省得越狠。V4 的 KV 用**混合精度**存：RoPE 那几维用 BF16 保精度，其余维度用 FP8，光这一招就比纯 BF16 省近一半；indexer 内部甚至用 FP4。

**FLOPs 账**（V4-Pro vs V3.2，单 token）：

| 序列长度 | V3.2 | V4-Pro | 比值 |
|---|---|---|---|
| 8K | 91.7 G | 109.3 G | 0.84x（短序列反而略亏）|
| 128K | 208.8 G | 131.6 G | 1.59x |
| 1M | 1083 G | 298.6 G | **3.63x** |

注意 **8K 时 V4 反而略亏**——因为 mHC、压缩器、indexer 这些新结构有固定开销，短序列摊不薄。但一过 32K，压缩注意力的省就压倒固定开销，越长赢越多。这跟我们做 TPU 推理的直觉一致：**任何稀疏/压缩方案都有固定成本，只有序列足够长才回本**。

> 对照我们 Wiki 里的 [[dsa]] 实测：V3.2 的 DSA 在 128K 把注意力算量压到 2K（64x）。V4 的 CSA 是「先压 4:1 再套 DSA」，等于在 DSA 之上又叠了一层压缩，所以长上下文的账比 V3.2 还漂亮。

### 六、Muon 优化器：从「逐点挪」到「整体转」

V4 大部分模块的优化器从 AdamW 换成了 **Muon**（Embedding、LM Head、RMSNorm、mHC 的静态偏置和门控这些仍留 AdamW）。

**核心区别一句话**：AdamW 把权重矩阵当一堆**互不相干的散点**逐个调（element-wise）；Muon 把整个权重矩阵当一个**几何整体**来更新（matrix-wise）。

**Muon 干的事**：拿到动量矩阵后，先做**正交化**再更新权重。正交化用 **Newton-Schulz 迭代**——本质是把这个「野路子」更新矩阵，投影到离它最近的 **Stiefel 流形**（所有列向量两两正交且模长为 1 的矩阵集合）上，找一个「最规矩的标准姿势」去更新。

用地球打比方：权重矩阵活在一个巨大的欧氏空间里，AdamW 在这个平直空间里做梯度下降，容易走「错觉捷径」。Muon 认为好的更新应该像**刚性旋转**（不随意拉伸扭曲空间），而这种「好姿势」正好住在 Stiefel 流形这个弯曲但结构优美的子空间上。Newton-Schulz 就是把更新「拉」回这个流形。

实现上就是几次 GEMM（`X@X.T`、`A@A`、`B@X`），BF16 精度，能吃满 GPU 的 TensorCore：

```python
def newtonschulz5(G, steps=5):
    a, b, c = (3.4445, -4.7750, 2.0315)
    X = G.bfloat16(); X /= (X.norm() + eps)
    if G.size(0) > G.size(1): X = X.T   # 始终让方阵取小的那维，省算力
    for _ in range(steps):
        A = X @ X.T
        B = b * A + c * A @ A
        X = a * X + B @ X
    ...
```

V4 用 **10 轮迭代：前 8 轮用一组「快收敛」系数把奇异值快速拉近 1，后 2 轮换「稳定」系数把奇异值精确钉在 1**。还借了 Kimi《Muon is Scalable》的两个改进：加权重衰减（防 RMS 无限增长溢出 BF16）、把 Muon 的 update RMS 缩放到和 AdamW 相近的 0.2~0.4。

**为什么 Infra 团队喜欢 Muon**：AdamW 要存两套状态（一阶矩 + 二阶矩），额外内存 = 2× 模型参数；Muon 只要一个 momentum buffer。**HBM 受限时，这意味着同样的卡能塞更大的模型或更大的 batch**。而且 Muon 也能配 ZeRO-1 分布式，只是更新阶段通信量稍大。因为注意力里已经直接对 Q 和 KV 做了 RMSNorm 防 logit 爆炸，Muon 里也就不需要 QK-Clip 了。

### 七、小结 + Infra 篇预告

V4 架构篇的账本合起来看：**结构变复杂了，但每一处复杂都换来了实打实的算力和 KV cache 下降**，尤其长上下文。这对国产卡上做低成本推理是重大利好。

三个改动各司其职：
- **mHC** —— 更稳的残差连接（双随机矩阵约束，深层不炸）；
- **SWA+CSA+HCA** —— 多尺度注意力，长上下文的效率命根子；
- **Muon** —— 更省内存、几何上更对路的优化器。

但账的另一面，是这些省全压到了**推理和训练框架**头上：MegaMoE（计算/通信/内存访问全重叠）、TileLang kernel、训推一致的 batch-invariant 确定性算子、FP4 量化感知训练、张量级 checkpointing、异构 KV cache + 磁盘前缀复用……这些就是第二篇要啃的，正好接上我们 DeepEP 那本的 EP 通信功底。

---

### 八、TPU 视角 & 我们的连接

- **CSA/HCA 对 TPU 推理意味着什么**：我们在 tpu-inference 里已经趟过 V3.2 的 DSA 移植（见 Wiki [[dsa]] 及 DSA TPU v7x 推理设计）。核心教训是「**保持压缩 KV、在 kernel 内部按需解压，别先解压再算**」——MLA kernel 本身就是 absorbed attention，从不解压 KV。V4 的 CSA 在 DSA 之上又加了 4:1 压缩，这条「不解压」的原则只会更重要。
- **混合精度 KV**：V4 的「RoPE 维 BF16 + 其余 FP8 + indexer FP4」和我们在 tpu-inference 里对 KV cache 的精度处理是同一套思路，FP8 indexer cache 比 BF16 省一半（见 Wiki DSA 对比分析）。
- **Muon on TPU**：Newton-Schulz 全是 GEMM + BF16，对 TPU 的 MXU 同样友好——如果哪天在 MaxText 上复现 V4 训练，Muon 的 TensorCore/MXU 亲和性是个加分项。
