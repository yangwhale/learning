## 1. DEC VAXCluster概述

在1980年代，出现了基于网络和消息进行核间通信的松耦合的分布式系统, 它们基于低速网络通信并且有独立的操作系统。另一方面则是紧耦合的基于共享内存进行大带宽通信的大型机。而DEC走了一个近耦合(Closely Coupled)的中间路线，构建了一套集群系统VAXCluster大获成功. **而如今在NVLink  ScaleUP网络构成的大型机系统, 和基于ScaleOut的分布式系统选择上, 这个话题很有借鉴意义.**

下面这一段是Gordon Bell在研发VAX集群的策略：**即构建同构的分布式计算产品，并能适配不同规模的计算场景而不需要修改代码。这与当下的需求非常契合:**

![图片](assets/5f3cff7f9ab4.png)

VAXCluster基于高速网络构建的共享存储并且基于内存块(Block)转移的软硬件融合的平台， 负责通信的CI Port和如今的RNIC有太多相似之处，机头无盘或者支持本地盘，机尾挂载存储。通信协议和RDMA也有大量相似的地方。

40年后重新再来看这个集群架构，特别是在去IOE去小型机十多年后的今天有非常重要的意义。VAXCluster基于内存页为单位配合分布式锁构建的这套集群架构具有极大的借鉴意义。其分布式锁架构被Oracle RAC一直沿用至今。如今在我们面临弹性构建的K8S等分布式推理集群的时候，是否能够借助RDMA和分布式锁及共享存储框架实现一套新的分布式集群的大模型推理操作系统？

## 2. CI Port架构

在VAXCluster硬件架构中最重要的就是一个基于消息(Message-Oriented)的高速计算机互连总线，也被称作CI Bus，而连接在CI Bus上的网络设备被称为CI Port。CI Port负责仲裁/选路和数据传输，并且也可以让VAX主机通过网络引导实现无盘启动并共享后端存储。

CI Port设计的初衷有两个：

从分布式节点中尽可能多的Offload通信带来的开销

提供标准的基于消息的软件接口，用于处理器之间的通信和设备访问

CI Port的上层软件系统被称为 VMS System Communications Architecture(SCA)。SCA提供三种通信服务：Datagram/Message/block data Transfer。针对Datagram和RDMA Unreliable Datagram(UD)相同，而Message和RDMA RC下支持的SEND/RECV语义相同， block data则是类似于RDMA的单边READ/WRITE操作直接将内存页按照一定的BlockSize转移到其它节点，并且也同样保证可靠性传输。

CI-Port队列定义也和RDMA有大量的相似性，或者几位RDMA的发明者在构建U-Net的时候也充分的借鉴了DEC VAXCluster架构，Command Q/Response Q/Message Free Q的定义和RDMA中的SQ/RQ/CQ几乎一致

## 3. 共享存储

VAXCluster把原有基于控制信号和寄存器状态的存储协议改成了一套基于消息的存储协议，并且实现了控制面和数据面分离，控制面通过SCA Message语义传输，数据面通过SCA Datablock传输，而本地盘还使用了(Mass Storage Control Protocol, MSCP)并且实现了共享存储

## 4. 集群管理

这一部分是值得我们借鉴的，集群管理类似于现在的K8S来进行集群内节点的加入和剔除。并且通过共享文件和分布式锁来实现资源的共享。

在VAXCluster中，集群的通信管理都被定义为System Communication Architecture(SCA)，它是一个基于CI总线的架构，包含如下三种通信服务:

Sending Datagram：不保证消息可靠传输，类似于UD

Sending Sequenced Message：类似于RC

Named-Buffer transfer：保证可靠传输，类似于RDMA READ/WRITE

SCA义了6种服务来支撑整个集群：

Cluster Configuration Management

Buffer Management

Connection Management

Directory service

Datagram and Sequenced-message service

Named-Buffer transfer services

### 4.1 集群配置管理

连接在CI总线上的要么是一台VAX计算机(机头)，要么是一台HSC(Hierarchical Storage Controller,机尾),架构最大支持连接224个节点，但是当时实现为16个节点，因此采用polling的方式检测节点是否存活。类似于RDMA RC建立QP的过程，SCA软件通过一系列握手通信的方式在节点间建立端口到端口(Port-to-Port)的虚链路(Virtual Circuit).这些VC被称作Path。

### 4.2 Buffer管理

在SCA中最重的一个组件就是在Node间的Buffer管理，SCA软件控制两种buffer：

SCA Control Message ：用于建立和删除SYSAP-to-SYSAP连接并控制Control buffer的使用量

SYSAP Data Message：SYSAP是VAX的系统应用，用于分配相应的buffer空间和Credit并提供给SCA

### 4.3 Connection管理

用于两个节点的SYSAP建立通信连接

### 4.4  目录服务

DNS服务的前生，用于SYSAP通信时找到对端，在K8S中也有类似的CoreDNS服务进程。

### 4.5 Datagram and Sequenced-Message服务

消息通信服务，类似于UD和RC的SEND RECV

### 4.6 Named-Buffer Transfer Services

类似于RDMA WRITE/READ

## 5. 分布式锁

VMS分布式锁管理(Distributed Lock Manager,DLM)主要是提供一个在VAXCluster中各个节点间的同步机制，并切确保透明和可靠的数据共享。Oracle早期也借助于VAXCluster的DLM实现并行服务(OPS)，直到现在Oracle RAC中的CacheFusion在多个实例操作相关page时，需要对关联的Page上锁，而它的分布式锁机制还和VAXCluster的DLM相同。

在VAXCluster中通过对资源的抽象和分布式锁提供透明的数据共享，任何一个处理进程操作和单机类似，但整个系统获得更好的弹性，可以动态的添加处理器/存储控制器/磁盘/磁带等.

更详细的内容可以参考: https://wiki.vmssoftware.com/Distributed_Lock_Manager

### 5.1 Lock Manager Description

在VAXCluster中， 分布式锁管理器(DLM)通过提供一组服务供协作进程(Cooperating Process)同步访问共享资源，从而实现VMS系统的高度透明的数据共享。

**Unique Resource ID**在VAXCluster中，任何资源都有一个全局唯一的抽象的名字并在所有的协作进程中达成一致。当这个资源变成全局可访问时(资源名进入Global namespace)就会由DLM进行管理锁管理.

**Resource Tree**资源本身会有相应的依赖，因此可以将资源名重新组织并构成一个树状结构(类似于DNS中的域名)，可以在树上不同的节点上锁来优化锁的粒度。当然锁结构不光是为了对某些需要共享的资源进行保护，也可以对一些事件定义锁来保证事件同步和事件的通知。

**AST/BAST**当一个进程需要对资源进行访问时，它会调用ENQ服务(lock)，处理完成后调用DEQ服务(unlock).ENQ服务收到进程的锁请求后，可以通过DLM授予或者拒绝，当然还可以将其放入一个队列等待并异步通知，当上锁成功后，系统会触发一个异步的通知，即AST(Asynchronous System Trap)。

另外还有一个BAST(Blocking Asynchronous System Trap)，针对一些锁竞争较低的资源，它实现了一个doorbell处理极致，可以在获取锁时特殊标记为BAST锁， 然后它一直访问而不需要频繁的上锁/解锁，直到另一个进程需要访问时，系统通过BAST触发通知资源获得者来主动释放资源。

当然AST/BAST适合于低Contention的场景，对于高Contention的场景可切换到Request-Release的方式。

**Value Block**

每个锁关联一个16Bytes的Value Block，并存在全局内存中可以原子更新，可以在获得锁时选择性的带回，或者在EX/PW模式锁被释放时这个字段可以被更新。

ValueBlock可以被用来实现磁盘数据的本地Cache，我们可以在其中定义一个SequenceNumber来表示资源的版本号，当资源被更新后，相应的版本号也需要更新。这样就实现了MVCC的并发控制。

### 5.2 设计约束和目标

DLM设计上面临众多约束，首先是需要保证非常高的可靠性，因为VMS分布式文件系统，数据库等众多关键业务需要用DLM维持其数据的完整性和一致性。另一个问题是DLM必须设计的足够的通用来适配各种应用。同时DLM还需要保证极高的性能，同时能够容忍任意数量的进程或者节点出现故障。

而设计原则是尽量降低节点间的消息数量，特别是伴随着节点数增加时，系统的开销必须要小于节点的线性增长度, 最终这条曲线会伴随着节点增加而趋近于一个成本常数。

### 5.3 DLM和连接管理器的关系

随着DLM的开发，很明显需要将管理处理器的动态配置功能和管理资源namespace的功能隔离开，即资源和底层的处理器和集群管理等功能进行隔离。因此连接管理器(Connection Manager,CM)出现，DLM依赖于CM来提供几个重要的服务。 CM实际上是一个集群管理软件，用于检测集群的成员变更并编排节点加入或删除，同时还需要在配置变更时协助DLM来重建描述锁的namespace和状态的数据库，并且还需要避免分区故障，例如通过投票Voting来找出一个最大连通的节点子集，然后配合DLM完成成员关系变更，从而避免脑裂带来的不一致行为。

CM提供基于VC的通信服务，并且对消息进行可靠保序传递，当VC出现故障时需要尽快的终止连接并且剔除有故障的节点或者端口。

### 5.4 DLM操作示例

**Resource Tree** 资源树，DLM允许资源名以层次后结构构建成一个资源树，例如Root资源可以代表一个设备，子节点可以表示为文件等，然后在下面的孙子节点可以表示为一条记录等。程序可以根据不同的颗粒度来锁定资源

![图片](assets/965c280c0a01.png)

**Lock Request** 一个进程申请对一个资源上锁

**Root-Lock** 对资源树的根资源申请锁定

**SubLock**	对资源树中根以下的节点申请锁定

**Resource Manager**	控制给定资源树上的锁请求授予的节点，它为该资源树维护了关于所有已授予和正在等待的锁请求的信息。所有的节点都是资源管理器的一部分，承担着部分资源树的部分子集的状态管理。

**Directory Service**	目录服务提供了一种定位资源管理器的机制，目录功能分布式存放于VAX机群的多个节点中，每个节点负责资源树的一部分，这种分布式的架构可以提升性能并且避免失效带来的影响。

**Lock Mode**	锁的模式，例如NL，PR，EX等，参考前面章节。

#### 5.4.1 针对一个根资源的初始锁请求

当一个进程需要获取一个Root-Lock时，DLM需要指定当前管理资源树的节点，通过对锁请求中携带的资源名(Resource Name)进行Hash获得相应的目录节点(Directory Node,DN)，Hash后由CM管理成员关系向量，进程向选定的DN发送一个消息请求资源锁，

注：这里可以采用一致性Hash算法来优化

DN收到请求后，可以做出三种响应：

目录服务中没有查到相应的资源树，告知请求节点自己管理当前申请的资源

目录服务中已经有相应的资源了，告诉请求节点向资源的Master节点重新发送请求

DN自己就是资源的拥有者，直接对资源请求进行响应。

当这是资源第一次被申请上锁时，符合DN的第一种情况，DN更新自己的资源树，并响应请求节点A，节点A在自己的资源管理器中添加对该资源的管理

![图片](assets/b46c1466d672.png)

后续的子节点的锁申请类似，如果进程驻留在资源管理器节点，此时由于RootLock已经获得，直接本地操作锁申请即可而不需要发送任何新的消息。

![图片](assets/84b9d7f25286.png)

如果驻留在其它节点需要一个Root-Lock请求，则需要通过Hash并向DN查询资源管理器节点，然后再向RM发送消息申请锁资源：

![图片](assets/7cb9bfc687a5.png)

#### 5.4.2 后续的根锁(RootLock)和子锁(SubLock)请求

当资源的Root-Level锁被建立后， 目录服务器就有了该资源管理器(RM)节点的信息，后续所有的锁请求都会被直接发送到这个RM节点：

![图片](assets/ab3e7a34cde3.png)

#### 5.4.3 释放锁请求

当驻留在资源管理节点RM上的进程需要释放锁时，除非该锁时资源上剩余的最后一个锁，否则不需要发送任何消息。在这种情况下，将向目录节点DN发送一条消息，表明该节点不再管理资源。然后，目录节点删除资源对应的目录条目。删除后，下一个请求该资源的节点可以成为这个资源的RM。

当释放锁的进程不在资源管理器RM节点时，它会主动向资源管理器节点发送一个请求消息。

![图片](assets/31705f959637.png)

#### 5.4.4 锁转换请求

锁管理允许更改已授予的锁的模式(Lock Mode)，这个操作被称为锁转换(Lock Conversion)。锁转换比重新申请一个锁的代价小很多，因为所有的数据结构和依赖关系已经存在，并且也有指定的资源管理器节点，同样如果锁的转换请求的进程驻留在资源管理器节点，则不需要发送任何消息，根据兼容性矩阵进行转换。 当申请锁转换的进程和RM不在同一个节点时，需要1条或者2条消息，当新的转换请求锁转换的模式是兼容的，则采用一条消息单方面的宣告锁转换，否则必须发送一个请求，由RM节点决定是否转换。

![图片](assets/896b42d84273.png)

#### 5.4.5 资源争抢时的操作

当资源出现竞争时操作会复杂一些，资源管理器收到不兼容的锁转换请求时，需要将请求排入队列，并且通过BAST通知其它所有持有不兼容锁的节点，有一个进程正在等待转换。因此每个锁都会有两个队列的数据结构：

Grant Queue：已经授予的锁的节点信息和是否需要BAST通知

Convert Queue：不兼容的锁模式正在等待锁转换的请求 通过这些Queue可以完成异步的锁状态转化

### 5.5 集群配置变更时的DLM

由于节点可能因为失效离开机群或者扩容使得新的节点加入。因此ConnectionManager需要在成员关系出现变更时和DLM配合。

#### 5.5.1 成员关系变更

在成员关系发生变化后， 首先需要使用一个协调节点(Corordinator Node)来领导其它节点完成状态转换。任何节点都可以成为协调器节点，通常是第一个发现成员关系变更的节点，例如当连接因超时而断开活着发现新的节点，可能需要改变成员资格时。

成员关系带来的配置文件变更需要通过一个两阶段提交算法来保证在整个集群内的一致性。协调器节点把添加活着删除节点的配置描述提议给其他节点，其它节点可以选择是否同意配置。如果其他节点能够基于与之同新的节点数量和对这些节点的投票分配来构建一个更优的配置，它会选择拒绝协调者的提议。

最终产生的VAXCluster只能是一组有强连接的节点组成的，其中每个节点都和其它节点有连接。在不不能达成一致时，协调器节点可以放弃提议并推出操作，随机等待一段时间，然后再次启动选举。而在此期间其它节点可以尝试成为协调节点。这样的分歧很快会被解决，并且新的配置也被所有节点所接受。

连接管理器能够为DLM提供VAXCluster系统成员处理器的一致性视图。同时还可以确保目录服务的一致性Hash，此外连接管理器为每个处理器还分配了一个唯一的标识，成为系统集群ID(Cluster System ID, CSID).

#### 5.5.2 重建锁数据库流程

在完成成员关系变更后，连接管理器将引导其它节点重新构建锁数据库，作为协调器的节点此时将负责同步，每个节点开始执行一系列控制如何重建锁数据库的Action Routines(AR), 每个AR描述重建过程中的一个特定步骤，所有节点并行执行AR。 一个或多个AR采用BSP(Barrier Synchronous Parallel)的方式执行，当到达一个同步屏障后，执行节点向同步器节点发送消息，表明已经完成一个步骤，正在通知等待执行下一个步骤。等同步器从VAXCluster所有节点收到此消息后，向每个节点发送消息执行下一步。如此往复，直到所有的AR执行完毕， 并且在所有节点上完成了锁数据库的重新构建。

从DLM角度来看，添加和删除节点操作是相同的，因为最终还是重新分配资源树管理节点的过程，因为这样的迁移可以保证集群工作负担均匀，并且防止出现长时间运行后大量的资源树被迁移到集群中最老节点的状况。

#### 5.5.3 失效检测

在发现远程节点中断后，连接管理器最初会认为这种情况是暂时的，并且尝试在一段时间内等待连接恢复，在这段时间内，正常的操作都可以被处理。对于远端节点，连接管理器也可以暂时将对它请求的消息缓存在发送Buffer中等重连接后重新按顺序发送， 发送方并不感知连接短时中断的问题。

如果在一段时间内无法重连，则向集群声明远端节点失效并触发集群重配置。在重配置期间，所有的节点都禁止锁定，但仍然可以发出锁请求，这些锁请求会被进程暂时阻塞，等待状态转换完成后再处理。

#### 5.5.4 重建锁数据库

每个节点以以下方式重新构建锁数据库。首先禁用新的锁请求。然后扫描锁数据库并删除所有目录信息，因为成员关系的更改将重新分配目录节点。 其他节点上的进程持有或请求的锁信息也会被丢弃。这些操作将导致在一段时间内不存在目录节点和资源管理器。唯一保留的信息是实际驻留在节点上进程发出的锁请求。此时，节点使用与最初获得锁相同的算法重新获得在成员关系更改之前持有的所有锁。

等待被授予的锁将按照排队时分配的序列号重新排序，以便保留它们等待的顺序。通过重新获取锁的过程，将创建新的目录条目并选择新的资源管理器。 由于每个节点重新获得自己的锁，而不再是VAXcluster系统成员的节点所持有的锁将被释放。一旦重新获得了所有锁，就会尝试授予正在等待的锁，因为删除由失败节点贡献的锁请求可能会允许授予等待请求。一旦完成了这些操作，DLM将恢复工作。

### 5.6 死锁检测

由于DLM是许多应用程序使用的通用的同步工具，因此不能基于对其使用方式的假设进行简化。分布式死锁检测算法要求是尽量减少死锁搜索过程中的消息数量，并且找到所有死锁，而且不能报告假的死锁。

从DLM的角度来看，可以分为两类：转换的死锁和多资源的死锁。分类的原因在于基于转换的死锁很容易被资源管理器检查到，而多资源的死锁需要使用更加复杂的分布式死锁检测算法实现。

#### 5.6.1 转换的死锁

转换的死锁设计单个资源上的多个锁定请求，以便资源管理器可以随时使用所有的信息来识别它们。我们来考虑一个请求，将一个锁在一个模式转换到另一个更加严格的模式(例如，从CR模式转换到EX模式)，如果在CR模式下还持有另一个锁，则转换请求必须等待第二个锁被释放或转换为兼容模式。如果尝试将第二个锁从CR转换到EX模式，则会导致死锁。 因此当第二个锁仍然以原有模式持有时，第一个转换请求不能被授予，而第二个转换请求也不能被授予，因为它必须等待第一个锁被授予。

#### 5.6.2 多资源死锁

多资源死锁可以通过在进程的“等待”图("wait-for" graph)中进行周期性的搜索来识别。我们可以用两个资源和两个进程来构造一个例子，假如P1已经持有资源R1上的锁，等待资源R2上的锁请求。如果在R2上持有锁的进程阻塞P1的请求，试图获得R1上与P1所授予的锁不兼容的锁，这样就会导致死锁。其持有和依赖关系就会构成一个环，分布式死锁检测算法则是通过发送消息遍历集群范围内的“等待”图的边。

但是环路检测的成本很高，而且在一个DLM中死锁并不会经常性的发生，因此通常情况下是在一个进程等待时间超过某个配置的阈值后才会启动死锁搜索算法，这样大大的减少了搜索的数量。

例如，如果系统S1上的进程PA的锁请求等待时间超过锁的等待阈值，则会为其发起死锁搜索。超时检测在管理资源的节点上进行，因为能够更加容易的利用相关的特定资源上的所有锁请求的信息辅助死锁搜索。如果转换超时，则搜索转换请求队列，以确定在超时转换请求之后发出的任何转换请求的手语模式是否与超时转换请求的请求模式不兼容。如果找到一个，则便是存在转换死锁，并选择一个受害者(Victim)，然后对受害者响应一个完成请求，并携带错误状态信息表明存在死锁。被授予的锁永远不会收到受害者选择的影响。

如果没有发现转换死锁，则启动更广泛的多资源死锁搜索。遍历进程的“等待”图，首先是拥有超时锁请求的进程，然后搜索返回到同一进程的路径，即环路检测。从锁的请求开始，对每个在资源上持有阻塞锁的进程进行检测，以确定该进程在其它资源上是否有等待锁。对于找到的每个等待锁，采用递归算法，直到找不到等待的锁，或者找到初始的进程。在前一种情况下不存在死锁，因为等待图没有环。后一种情况下死锁存在，因为已经发现了一个循环依赖的环。

需要注意的是，图搜索如果跨越了VAXCluster的系统处理器边界，则需要发送消息，指示目标处理器继续搜索。这些消息表明，搜索应该从某个锁开始，并继续发现拥有超时锁请求的进程为最终目标。

在实现中，存在两种必须要考虑的可能性：

第一种情况，发现一个阻塞的锁属于驻留在远程系统上的进程，这种情况下，必须通过表示进程正在等待的所有锁，在远程系统上继续搜索。

第二种情况，进程正在等待远程系统管理的锁。在这种情况下，必须通过识别正在阻塞等待锁的所有其它锁，继续在远端系统上进行搜索。

下面是一个例子，节点N1上的进程P1拥有的锁L1请求等待超时，然后触发死锁搜索。锁的资源管理节点是N2，因此在N2上开始死锁搜索，在资源上发现了节点N3上进程P2拥有的阻塞锁L2. 然后向N3节点发送消息，指定从锁L2开始搜索，目标是通往进程P1的路径。N3收到后，确定进程P2正在等待节点N4管理的锁L3.因此一条消息被发往N4，继续从L3开始搜索，目标是找到进程P1.然后发现L3被进程P1拥有的锁L4阻塞。

由此发现了一个环，那么选择一个作为受害者，它的等待的锁请求以死锁状态返回完成。死锁消息包含了迄今为止找到的最佳受害者身份，并将消息发送到受害者所在的节点。（在死锁搜索中使用了<R . Obermarck , " G obal Deadlock Detection ." IBM Research Report RJ2845(36131)(June 1980)>描述的类似算法的有趣扩展）。

为了防止在不包括具有超时Lock请求的进程上进入死循环，并且减少最坏情况下的搜索复杂度，可以使用bitmap来指示在搜索中是否已经访问过某个进程。VAXCluster中每个节点都有一个bitmap用于表示节点上的进程。当搜索开始时，所有的bitmap清零。如果某个进程参与了死锁搜索，则将该进程对应的位设置为1，如果随后到达一条消息，表明该流程应该参与搜索，则该消息将被忽略，因为已经搜索了该流程的所有路径。

节点永远不知道死锁搜索何时完成，因为当没有发现死锁时，消息就会消失。因此必须提供某种方法来确定何时可以在新的搜索中重用bitmap。通常的做法是给死锁搜索分配一个“生命周期时间戳”来完成。在此方案中，当集群成员关系发生变化时，连接管理器CM指定一个节点为时间戳服务器。

要启动死锁检测时，节点向时间戳服务器请求时间戳，时间戳表示的是一个稍微在未来一点的时间，一旦该时间戳发出，时间戳服务器将不会发出另一个时间戳，直到该时间戳超时。时间戳的默认值是now+50毫秒,这是基于最差的搜索情况来估计的。在死锁消息中使用时间戳来指定特定的死锁搜索过程。无论何时解收到死锁消息，它的时间戳都会与存在bitmap中的时间戳进行比较决定如何使用bitmap，有三种可能的情况：

消息时间戳超过bitmap时间戳：bitmap正在被先前的死锁搜索使用，并且它的生命周期已经过期，在这种情况下bitmap可以用于新的死锁搜索，将新的时间戳保存并将bitmap清零，然后继续搜索。

消息时间戳和bitmap相同：bitmap和当前的消息同期，可以被当前消息更新bitmap。

消息时间戳小于bitmap时间戳：bitmap已经被后一轮搜索抢占，分配给此消息的bitmap已经过期，现在终止这个死锁搜索，并且在稍后用一个新的时间戳重新启动，通常这个避让时间戳为一个生命周期的2倍。

bitmap不仅提供了如上的搜索性能优化，同时也可以防止算法遇到意外的死锁时发生死循环。例如进程PA等到PB，PB等待PC，PC等待PB，进程PB和PC有一个死锁，以PB或者PC始发的死锁检测都会发现问题，而在以PA为始发的死锁搜索过程中，会发生死循环，而bitmap防止了死循环的发生。

## 6. 分布式文件系统

VAXCluster通常需要并行的响应多个主机的存储请求，因此开发了一个扩展的QIO处理器(Extended QIO Processor)实现同步的文件访问，DLM提供锁服务来供数据块的同步访问和仲裁。按照一致性顺序取出锁可以防止死锁，同时它还提供了Cache管理能力，并由带序列号的锁和一组同步队列来确保一致性。

### 6.1 概述

VMS文件系统为所有的VMS用户和VMS系统本身的许多其他组件提供基本的文件管理功能。从由一系列数据块(block)构成的Raw磁盘开始，该文件系统提供了许多文件管理，目录，安全性控制与管理文件结构的各种功能。 VMS到文件系统的接口是系统服务，QIO提供了基于block的文件读写接口，并提供了一系列特殊的函数用于文件的创建，访问，修改和删除功能了。

### 6.2 ACP(Ancillary Control Process)

VMS Release 1～3中在I/O路径上使用ACP辅助控制进程来提供多用户的并发访问：

![图片](assets/1b81be196894.png)

但是在VMS支持VAXCluster时，需要机群中的每个主机都能访问磁盘，ACP的很多概念在多机系统中都失效了。虽然可以使用“ACP Master”的方式指定集群中的一个节点成为一个卷的管理者，但是会导致单点故障。并且发生故障后的文件转移也非常困难。因此VAXCluster放弃这种单个ACP构成的隐式同步和缓存管理机制。而是采用一种对称的文件管理方法，在分布式系统中显式的去处理这些问题。

### 6.3 XQP(Extended QIO Processor)

由于集群需要显式的去处理同步访问，而同时支持隐式的ACP和显式文件管理这两套系统显得又些冗余。因此在VMS上开发了新的基于扩展QIO处理器(XQP)的文件系统，即便是只在单台机器访问本地硬盘也采用同样的处理方式，如下图所示：

![图片](assets/dd1474ce5065.png)

相对于master ACP的方案，XQP有更多的优点，它采用一个Async System Trap线程的方案，支持：

一致性：所有的文件操作在整个集群内保持相同的同步访问机制

高性能：避免了在ACP中执行产生的Context Switching

高并发：多个文件操作可以显式的并行执行

### 6.4 同步用到的锁

文件系统需要同步主要来自以下两个原因：

文件结构的完整性：多个用户同时修改文件时需要保护

文件系统语义：提供用户级的同步（例如两个用户同时访问相同的文件)

同步首先通过将文件结构组织为可以同步的单元(unit)来实现，然后通过底层设施来控制并发性。VMS的锁管理服务(DLM)为VAXCluster提供了理想的同步工具，事实上文件为基本的可同步单元(无论是文件本身，还是目录结构，还是卷管理都可以抽象为文件操作)。因此利用文件头的48bit文件ID和32位整数构建的版本号作为DLM中描述资源的名称(Resource Name)，然后通过DLM提供分布式锁实现同步访问。当然为了访问效率还可以提供基于设备/卷等多种颗粒度的锁结构。

#### 6.4.1 Device Lock

设备锁(Device Lock)管理被集群访问的设备的状态。锁的资源名为设备名, 并携带SYS$前缀，主要使用如下四种锁模式：

Lock Mode

设备状态

NL/no lock

idle

CR(并发读)

卷可以被共享访问

PW(保护写)

设备正在挂载

EX(排它)

卷被私有的分配或挂载

设备锁的Value block中包含了有关设备状态的附加细节(设备的所有权，保护模式，是否Mount，是被外部系统Mount等）

#### 6.4.2 Mount Lock

VMS系统中的设备仲裁语义决定了设备锁无法等待实现异步操作，因此任何设备的访问冲突都会产生锁错误。因此附加了一个Mount Lock(资源名: MOU$ + 设备名)来实现对同一设备的序列化的并发Mount尝试。Mount Lock模式为EX，从而允许集群中其它节点在当前Mount操作后对Mount请求进行排队。

#### 6.4.3 Volume Sync Lock

挂载一个卷后会以CR模式创建一个卷同步锁, 用于表示已挂载的卷与该卷锁挂载的设备一对一关联。 对于可共享卷，锁的资源名是从卷标(Volume Label)派生出来，以F11B$v为前缀。对于不可共享卷使用UCB(unit control block，即VMS表示设备的一种数据结构)的系统地址作为卷名称。卷锁的Valueblock包含了额外的flag用于描述卷的状态以及分配情况和缓存管理状态。

在挂载卷的整个阶段，集群成员必须持有设备锁(Device Lock)和卷锁(Volume Sync Lock)，这个阶段通常会超过系统中任何进程的生命周期。因此，常规的与所有者的进程关联的锁不能被使用。取而代之需要使用的是系统持有的锁，而不是某个特殊进程的锁，以保证其生命周期长于进程。这些锁只有在系统有明确命令控制或者离开集群时才释放。 卷同步锁还保证卷上所有空间的同步分配和回收。当XQP希望分配空间是，它需要一个单独的PW模式的卷锁的副本(用于保证只有一个进程在同一时间尝试分配或释放空间，PW和卷同步锁的CR模式是兼容的). 这种形式的卷锁可以被进程持有，但仅适用于较短的时间(实际上是一个Latch)。

卷锁的Valueblock中有一部分用于控制空间分配，它包含了当前空闲块的技术以及相应的block分配bitmap的指针帮助XQP读取Valueblock，并在释放时将其写回到DLM。

#### 6.4.4 File Serialization Lock

文件序列化锁(File Serialization Lock)会对影响单个文件状态的所有操作进行同步，它在DLM中的资源名(Resource Name)为F11B$s+文件号, 它是Volume Sync Lock的子锁。 XQP使用此锁在PW模式下保证文件的序列化操作，确保对任何一个文件一次只执行一个操作(打开，关闭，扩展，删除等)。另外文件序列化锁还可以用于对目录的序列化操作。

文件序列化锁是一个进程级的锁，仅在单个文件操作期间持有，可以确保并发操作时的完整性，但还需要另两个锁来支持文件系统提供给用户的同步语义。

#### 6.4.5 Arbitration Lock

多个用户可以同时打开一个文件，因此用户对于文件的读写操作需要由文件系统提供仲裁，仲裁锁用于仲裁跨集群的文件访问。其在DLM中的资源名为F11B$a+文件号，并同时以所在卷的锁的资源名作为前缀，它包括如下几种模式：

Lock Mode

文件访问权限

NL/no lock

没有锁

CR(并发读)

文件打开，可以被读取，并同时允许其它的读写操作

CW(并发写)

文件打开，可以被读写，并同时允许其他的读写操作

PR(保护读)

文件打开，可以被读取，仅允许其它的读操作

PW(保护写)

文件打开，可以被读写，仅允许其它的读操作

EX(排它)

文件以排它的方式打开，其它节点无法操作

仲裁锁需要在文件打开的整个时间内被持有，多个独立的进程级锁对于访问权限控制开销太大，因此用一个系统拥有的锁表示来表示每个集群上节点对文件访问的所有状态，其锁模式表示为该集群成员中的“最高”访问模式。

#### 6.4.6 Blocking Lock

某些对于文件结构维护的操作要求在一段时间内保持稳定。例如在磁盘重建操作期间需要通过使用特权的文件函数来锁定整个卷的所有文件操作。在集群范围内实现这样的功能需要使用一个特殊的锁，即阻塞锁(Blocking Lock)。 阻塞锁的资源名是F11B$b+卷锁名，这是因为如果对每个文件的锁操作性能会更低，因此采用了一个Resource Tree上更高的节点，以卷的范围进行阻塞锁定。正常情况下这是一个由系统拥有的并且每个集群中节点持有的锁，锁模式为CR，此状态会记录在卷控制块(Volume Control Block,VCB)中。而每个文件函数开始时只需要检查本地的卷状态即可。 当卷锁定函数开始执行时，它尝试将锁提升到EX模式，由于EX和CR不兼容，系统通过使用DLM BAST的方式通知集群中其它的节点释放CR模式的锁，然后再将它提升到EX模式。

此后卷将保持锁定状态，因为每个文件函数操作开始时对于阻塞锁的检查都会失败。在这种情况下XQP会尝试着重新获取阻塞锁，等卷锁定函数解锁卷并释放EX模式阻塞锁时，其它所有等待该锁的进程也将重新获得CR模式的阻塞锁，此后文件操作就可以正常进行了。

#### 6.4.7 锁顺序和死锁保护

对于单个文件函数的执行需要同时申请多个锁，因此处理不当容易产生死锁。XQP通过以下固定的顺序处理锁来避免死锁：

Blocking Lock

Directory Serialization Lock

File Serialization Lock

Volume Lock

其它特殊的锁

需要注意的是，目录和文件可以构成层次化结构，但VMS文件系统允许创建任意的连接，因此目录连接可以指向目录层次结构中的其它位置。当两个进程试图对目录结构进行遍历时，可能会导致死锁。这种被判定为异常，由VMS系统将它们返回给调用者。

### 6.5 Caching

VMS中的文件结构很复杂，通常文件操作需要检查或修改文件结构中的多个部分，为了保证高性能，VMS对于文件实现了一个扩展的cache组建，这些Cache可以吸收大概75%的磁盘I/O访问，包含如下功能：

一个通用的基于数据块的Buffer Cache 来存储最近从磁盘读取的数据块

一个文件控制块(FCB,file control block)列表描述所有打开的文件属性和状态及最近引用的目录

一个区段缓存(extent cache)持有磁盘中的空闲空间用于快速的分配和回收。并且这段空间在磁盘的存储bitmap中(控制空间分配的主要结构)被标记为已使用,以确保系统崩溃时的安全性，系统崩溃时，区段缓存将暂时丢失，但是由于在bitmap中已经标记为已使用，因此崩溃前分配给文件的空间不可能在崩溃后再次分配给其它文件，丢失空间通常在挂载卷后通过磁盘重建来恢复。

一个文件ID缓存保持一组未分配的文件号，用于快速的分配和释放。与区段缓存类似，这些缓存中的文件号在磁盘的bitmap中也被标记为已使用。

当配额管理启用后，需要产生一个配额缓存时是的保存当前活动用户的配额记录。

由于XQP的引入，在分布式集群上维持Cache一致性需要进行特殊的处理。而这些问题通过使用VMS锁服务构建的分布式锁可以解决。

在XQP中利用一个System Page pool来管理这些block-buffer cache，然后通过Sequence Counter(实际上是一个MVCC的版本号)和分布式锁的Blocking AST(BAST)来维持一致性。

#### 6.5.1 Block Buffer Cache

Block-Buffer Cache按照512B作为一个block，并且还定义了相应的Descriptor和Hash表。缓存一致性使用传统的锁机制和序列号技术实现。XQP处理每个文件结构块时都会有一些同步锁进行控制。锁的ValueBlock包含一个序列号，表示该锁所控制的块的最后一次更新，当读取一个块时，文件系统将当前序列号与缓存中保存的块副本进行关联。在修改一个块时，文件系统增加序列号，并在文件操作结束时，用更新的序列号释放锁，如果缓存中仍有任何数据块，则相应的锁不会完全释放。相反，锁被降级为NL模式以保持Valueblock的连续性。

如果另一个系统的XQP随后引用了这个文件结构块，并且在它自己的缓冲区找到了旧的副本，那么该系统将发现缓存中的Descriptor的序列号和锁的Valueblock中的序列号不匹配，这种不匹配表明块已经被修改，并且缓存内容已无效，必须从磁盘刷新。

我们前面观察到卷同步锁(Volume Sync Lock)和文件序列化锁(File Serialization Lock)可以确保文件结构完整性。因此所有的文件结构数据在这两类锁下读写，都可以保证缓存一致性。 卷上和空间分配相关的block，例如存储bitmap，file number bitmap等，都由卷锁处理。而其它所有块(例如文件头/目录内容)都由它们所属文件的文件序列化锁处理。文件序列化锁带有两个序列号来区分文件头或者文件数据的更新。 Block-buffer Cache分为如下四个buffer pool：

文件头和文件号bitmap block

Storage bitmap block

目录，Quota文件和其它数据block

目录index block

之所以需要这种分区，是因为每种类型的一个或两个缓冲区可能必须并发可用。例如在创建一个文件时可能需要并发的访问文件头/Storage bitmap(用于空间分配)和目录(用于创建目录)。每个buffer pool都采用LRU来管理，因此，缓冲区管理器可以保证每种类型的一个或者两个缓冲区的并发访问，而不需要任何显式的缓冲区锁定和释放机制。

每个Buffer有一个Buffer描述符(BFRD),包含了标识和管理当前缓冲区的所需信息：

![图片](assets/d63861eaedf8.png)

BFRD结构如上图所示，包含以下字段

Logical  Block Number(LBN)和Unit Control Block(UCB)用于标识磁盘地址和缓冲区所包含的块的卷。

Lock Basis管理缓冲区的锁的资源名(ResourceName)的根，用于DLM中的Lock Request方便找到资源管理器

Sequence Number来自于锁中的ValueBlock，它可以用于判定缓存是否已经失效

Flags，类似于MESI标记valid和modified

ProcessID，标记这个block是由哪个进程控制的

NEXT, Hash Chain的链表指针

用于状态队列的队列指针

Buffer Lock Block(BFRL)和每个Buffer关联用于锁管理，当然为了访问效率也可以多个buffer共享同一个锁，

BFRL包含如下信息：

REFERENCE CNT：一个引用计数器

NEXT：hash-chain的链表指针

LOCKID：锁的ID

LOCK BASIS：锁的资源名的根

PARENT ID：锁在资源树中Parent的ID

Buffer和锁通过两个Hash Table查找，一个用于BFRD，以磁盘上Block的LBN(Logical Block Number)为Key索引，而锁以LOCK BASIS为Key。然后有一个Cache Header把两个hash绑定在一起，Cache header包括如下字段：

两个Hash Table指针

BFRD和BFRL列表

可用计数器和描述符组成的Buffer Cache的四个分区

性能计数器

若干个同步队列

每个同步队列包含如下功能：

Cache同步队列：对于Cache描述符的修改需要串行化

Pool等待队列：如果Buffer Pool中剩余的Buffer空间不足，XQP在处理文件函数时需要等待。

Ambiguity队列：用于同步文件头的锁名称有时会改变，例如一个Multiheader文件包含多个文件头构成的一个Chain，它需要利用主文件头的文件串行化锁来完成同步访问。因此，锁在文件被删除并且扩展头被另一个文件重新使用时，扩展头的名称会发生更改。VMS软件发现一个文件头缓冲区属于另一个进程的锁时，就会使用该队列，允许当前正在执行的XQP等待，直到header buffer稳定下来。

由于每个主机CPU都有自己的Buffer Cache，对于这些Cache的访问并不需要锁管理器来同步，而是采用一种排队机制来节省开销。当XQP必须等待其中一个Buffer header queue时，它只需要将表示当前文件操作的I/O包发送到适当的队列并暂停执行。一段时间后，系统另一个进程将纠正第一个进程的任何等待条件，完成此操作后，其它进程检查相应的队列，进程重启并从同步队列中移除I/O包。

#### 6.5.2 Buffer管理

在Block-Buffer Cache中，每个Buffer处于两种状态之一：

可用(可能包含也可能不包含磁盘数据)

它属于某一个进程(并且只能属于一个进程)

Cache需要仔细的管理以避免资源死锁，并防止个别进程“占用”。资源死锁发生在一个进程部分的执行了文件函数时，然后发现需要额外的I/O Buffer，由于是部分执行，因此该进程还会持有一些锁，如果没有更多的缓冲区可用，进程将不得不等待。于此同时另一些进程持有相应的I/O Buffer而同时又在等待前一个进程持有的其它锁时，将会导致死锁，这样的死锁DLM无法检测到。

通常通过启动文件函数之前预留足够的Buffer，可以来避免资源的死锁。因此，文件系统的所有文件函数都可以使用已知的最小数量的缓冲区来完成。如果这个最小的数目不可用，那么XQP必须在Pool等待队列上等待，而此时XQP并没有持有任何锁，因此死锁不会发生。

Buffer的预留可以通过简单的减少Cache Header中的可用计数器来实现，直到需要时，进程才实际使用Buffer，并在BFRD中的 state queue linkage和owner PID(Process ID)表示Buffer状态。可用的Buffer被链接到一个LRU列表，并且其Process ID字段置零。

当一个进程需要读写磁盘某个块时，首先进程需要获得一个Buffer，进程可以通过磁盘块对应的LBN在hash表中查找，如果没有找到Cache，那么从LRU列表中删除最老的一个Buffer来获得， 从LRU列表中删除并拥有该Buffer后，需要将自己的PID写入BFRD的PROCESS-ID字段。

Buffer的进程间隔离：BFRD中的PID用于区分不同进程，因此进程不能获取和访问BFRD中PID和自己不一致的Buffer 如果Buffer用于文件头，由于Multiheader带来的Lock Basis可能被修改的情况，XQp必须使用Ambiguity队列等待。而其它情况下Buffer一旦被获取，LockBasis不会被修改，如果发现另一个进程拥有了相应的buffer，则代表文件同步机制已经被破坏，这将导致系统Crash。

在许多情况下，需要大量的buffer，例如当一个文件有很多header或者需要扫描一个很大的目录时。如果Cache可用，则XQP将会继续申请使用buffer。但是当Cache header中的Pool Available Counter到了最小阈值时，XQP将停止申请使用Buffer，而是采用进程内Swap的方式来处理。

在文件操作结束时，进程内列表保存的所有Buffer必须要释放返回给Cache，并且修改后的Buffer都要在返回时写回磁盘。XQP确保每个Buffer都与一个BFRL关联，该BFRL是这个Buffer的同步锁，当返还Buffer给Cache时，XQP需要释放这些锁。在Cache中保留的Buffer的锁不会被释放而是降级为NL用于保持Buffer的序列号。 对于修改的buffer回刷硬盘可以通过分布式锁的Blocking AST机制实现。

文件系统允许用户进程修改文件结构(例如磁盘重建程序等..)，当用户对storage bitmap进行修改时，需要考虑Block buffer Cache带来的影响。而在这种情况下，可以将读写直接路由到XQP，bypass原有的cache机制。

#### 6.5.3 File Control Blocks

和Block-buffer Cache一样，打开文件和目录的文件控制块(File Control Block)在Cache中也需要保持一致性。锁管理器DLM提供的Blocking AST(BAST)机制可以解决这个问题。每个集群成员都为集群可访问卷上的每个打开的文件持有仲裁锁。仲裁锁关联了一个系统的BAST例程，但是文件的访问冲突并没有等待机制，当用户修改文件属性时，必须更新整个集群的各种文件控制块, 它通过申请仲裁锁到EX-Lock模式来触发BAST执行。BAST例程只会将本地的文件控制块标记为“Stale”状态。而锁Ex-Lock请求一旦进入队列将被立即取消，因为通常它需要被授予。在其它节点上，对文件的下一次操作时，XQP看到“Stale”状态的文件控制块，直接刷新即可，然后重新建立BAST。

#### 6.5.4 Quota Cache

在配额功能打开后，通常会有一个文件记录各个用户的配额，因此使用Quota Cache机制可以很好的保证配额访问的局部性并降低配额功能带来的管理开销。Quota Cache机制如下图所示：

![图片](assets/73ada9c4dab5.png)

首先每个配额缓存条目必须被单独的处理，对于每个配额缓存都有一个锁来保证配额的全局一致性，可以将相关的配额缓存信息封装到锁的16Byte Value Block中实现。锁的资源名是F11B$q+文件所有者+卷名，并且以PW模式下持有锁。当集群中另一个XQP希望使用相同的配额时，它必须找到另一个可用的Quota Cache条目，然后在PW模式下排队申请锁，这会在当前持有锁的XQP上触发一个BAST。
