## 1. 概述

"Agentic LLM" 是当前大模型领域最热门的发展方向之一, 也是有明确商业化变现能力的场景. 但是它和以往的 Chat 类任务有很大的区别. 以前的 AI 就像个问答机, 你问一句, 它答一句. 现在的Agentic LLM通常是你给它一个复杂任务, 比如"帮我写个爬虫程序抓取天气数据", 它会自己上网查资料、写代码、运行代码、调试错误, 这个过程需要来来回回几十甚至上百步. 尽管每个单独工具调用或者反馈都很短(通常数百个Token), 但是上下文会逐轮累积, 并增加到极长的长度. 这样会导致Agentic LLM的工作负载变成存储I/O密集型: 短追加(short-append)的模式导致了非常高的KVCache命中率(通常 ≥ 95%). 这使得**KVCache的加载效率成为主导性能的因素, 而并非是纯粹的算力主导**.

然后为了优化Agent Workload, 现有的LLM推理系统通常采用 Layer-wise Prefill, PD分离以及外部KVCache存储的架构. 在这些系统中, Prefill节点以以逐层的方式加载 KV-Cache, 以便在单个批次内容纳尽可能多的请求. 当Prefill完成后, Decode节点通常通过高性能RDMA网络从Prefill Node接收KVCache并执行Decode 生成token, 然后将其KVCache存储在分布式存储中, 以便多轮之间进行重用.

然而, 这种架构也引入了一个关键的限制.Prefill节点必须从远程存储中加载大量的 KV-Cache. 结果, Prefill节点的存储网络带宽成为了整个系统的吞吐量瓶颈, 尽管此时Decode节点通常拥有大量未使用的存储网络带宽.

![图片](assets/25a07d3d9d48.png)

正如上图所示, 从存储网络来看, Prefill节点和Decode节点的负载是非常不均衡的. Decode节点的存储带宽并没有被充分利用. 简单的为Prefill提供更多的带宽成本高昂, 并且在通用集群中通常不切实际. 因此, 利用其它可用的带宽资源(Decode节点的存储网络带宽和Prefill-Decode节点之间的ScaleOut RDMA带宽来加速Agentic LLM工作负载下的KVCache加载成为一个方向.

例如 Mooncake 这样的项目已经在尝试缓解 KV-Cache 加载瓶颈, 它将 KV-Cache 缓存在一个分布式 DRAM** 池中, 并采用一种亲和性感知的调度器来最大化 DRAM KV-Cache 的命中率. 然而, 在内存受限的场景中, 例如强化学习 (RL) 中的 rollout 阶段, DRAM 被用于存放从 HBM** 卸载的大量训练状态, 这种方法便无法使用. 考虑到 DRAM 和 SSD 之间的成本比较, 在工作集巨大的场景 (例如在线服务) 中, 它也不具备成本效益.

现有系统总是将 KV-Cache 从存储直接加载到Prefill节点中, 它们无法利用Decode节点的远程存储带宽.  因此作者提出了 DualPath, 实现了双路径 KV-Cache 加载:

![图片](assets/39a1a69e0620.png)

当然这种设计会带来一些潜在问题和挑战:

KVCache在ScaleOut网络传输, 会对模型原有的在ScaleOut上的集合通信带来干扰.

系统必须在线决定动态在哪条路径加载数据,  并确保同时在 GPU 和 NIC 上实现负载均衡.

一个优化的双路径加载数据通路设计, 在常见的 P/D 比例下不会引入内在拥塞

一个动态调度策略, 共同平衡Prefill和Decode引擎的计算和网络利用率

DualPath业务收益:

显著提高了系统吞吐和TTFT, 并保持了 TPOT.

## 2. 背景

首先介绍了现在流行的Decoder-Only的Transformer架构, Attention层使得请求内的 token 能够相互交互, 而 FFNs 则独立地处理每个 token. 模型基于前面的 token 预测后续的 token, 并将Attention中的 Key 和 Value 作为 KV-Cache 存储在 HBM 中, 以避免重复计算.

### 2.1 LLM 推理概述
PD分离
将推理过程中的Prefill和Decode两个阶段分离, 并将它们分配给专用的Prefill引擎(PE)和Decode引擎(DE)处理:

**Prefill:** 计算密集型且可批量处理

**Decode:** 内存带宽密集型且对延迟敏感

通过 PD 分离, PE 加载命中的 KV-Cache 并执行Prefill; 然后, 它们将 KV-Cache 传输到 DE, DE 再执行自回归解码. 这种设计减少了阶段间的干扰, 实现了针对特定阶段的优化, 并提高了可扩展性, 使其成为现代 LLM 服务的既定架构. 为了支持多轮对话, KV-Cache 通常被存储在分布式存储中, 以便在多轮之间重用.
Layerwise Prefill
长上下文的 Prefill 受限于 HBM 容量, 因为整个批次的激活值和 KV-Cache 都必须驻留在其中, 这迫使 BatchSize 大小受限, 导致 GPU 利用率低下. LayerKV 和 PrefillOnly 通过利用 Prefill 计算中的强局部性来解决这个问题: 每一层只需要其自身特定层的 KV-Cache. 因此, KV-Cache 可以按层进行分配和释放, GPU 在处理前向批次时仅需保留一层的 KV-Cache. 这将有效批处理大小 (以 token 计) 大约增加了与层数相等的倍数, 从而提升了 Prefill 的吞吐量.

### 2.2 Agentic LLM的使用方式

LLM 越来越多地为智能体应用提供支持, 这些应用在长会话中执行多轮推理并与环境交互 (例如, 通过终端命令、代码执行或寻求人类反馈). 如图所示, 在一个典型的轮次中, 模型接收由先前上下文加上一些新追加的 token (通常是工具输出或用户输入) 组成的提示, 并生成下一个动作或响应.

![图片](assets/a5dacf593167.png)

单个智能体运行是一个包含数十甚至数百轮的轨迹: 上下文逐轮增长, 最长可达一百万 token. 因为大部分上下文 (在作者的追踪数据中通常 >95% 的 token) 在轮次间被重用, 所以每一轮中的绝大多数 token 都能命中 KV-Cache; 只有新追加的上下文需要进行 Prefill 计算. 由于智能体轨迹的极端长度, 基于 DRAM 和 HBM 的 KV-Cache 存储 (如 Mooncake** ) 只能存储一小部分 KV-Cache, 这使得使用容量更大但更便宜的、基于外部 SSD 的 KV-Cache 存储成为刚需.

智能体 LLM 推理工作负载在智能体 LLM 训练中也很普遍, 训练通常采用强化学习 (RL) 方法. 在一个典型的 RL 训练循环中, 智能体 LLM 首先经历一个 rollout 阶段, 在此阶段它被提示生成大量的多步智能体轨迹. 然后, 这些轨迹由一个单独的奖励模型进行评分. 最后, LLM 的参数被更新, 以增加高分输出的可能性并减少低分输出的可能性. 在 rollout 阶段, 大量数据 (如奖励模型和优化器状态) 被卸载到主机的 DRAM 中, 进一步限制了可用于 KV-Cache 的 DRAM. 这再次强调了需要一种外部的、高容量的 KV-Cache 存储, 能够高效地容纳长的智能体 rollout 上下文.

渣注
简单来说对于单个200K上下文的请求, Prefill需要缓存 50GB~100GB的KV-Cache数据, 为了提高BatchSize, 需要按层在CPU DRAM和HBM之间频繁的替换, 并且CPU的DRAM容量也有限, 因此需要外部的SSD存储. 这些不光是在推理服务阶段, 在模型RL训练阶段由于Reward Model和LLM参数更新以及Optimizer的内存占用等, 对CPU DRAM的占用更高, 给CPU DRAM剩下的缓存KVCache的空间更小, 因此必须要使用外部的基于SSD的高性能高容量的KV-Cache存储.

### 2.3 现代 AI 数据中心架构

现代 AI 数据中心通常采用三个独立的网络, 如下所示:

![图片](assets/e2b603ee471a.png)

**ScaleUP**: 每个节点配备 8 个通过高速 NVLink 互连的 Hopper GPU.

**ScaleOut**: 每个 GPU 都配有一个专用的 400 Gbps 网卡 (作者将其称为计算NIC,CNIC), 以最大化节点间的通信带宽.

**FrontEnd**: 独立于ScaleOut网络结构, 每个节点还具有一个高达 400 Gbps 的存储网卡 (SNIC), 提供对数据集、模型检查点和磁盘上的 KV-Cache 的快速访问, 以及云上多租户场景下的VPC租户隔离.

这种架构的一个基本原则是计算网络和存储网络相互隔离. 这种分离对于最大化存储和应用性能至关重要. 通过将 GPU 之间的高强度东西向计算流量与存储流量隔离开来, 该架构防止了它们之间的干扰, 并显著降低了计算通信的延迟. 这种设计还确保了即使在执行数据密集型任务 (如读取大型数据集或写入数 TB 的模型检查点) 时, GPU 间的通信也能保持高度可靠和可预测.

渣注
但是实际上DualPath已经打破这种计算与存储网络隔离, 如果能够很好的控制干扰, 实质上为什么不直接将FrontEnd和ScaleOut合并在一个网络内呢? 这样所有的ScaleOut网卡都可以直接访问存储, 那不是比DualPath带宽更大, 而且数据路径更加干净么?

## 3. 瓶颈与动机

作者观察到在智能体推理任务期间存在严重的 GPU 未充分利用现象. 调查揭示, 由于每个节点上单个存储网卡的带宽有限, KV-Cache 的加载速度是瓶颈所在. 分析表明, 三个决定性因素共同导致了这一瓶颈, 总结如下图所示:

![图片](assets/506c82206d26.png)
工作负载特性
智能体工作负载表现出高 KV-Cache 命中率, 这需要更多的 I/O 和更少的计算, 从而造成了严重的 I/O 瓶颈. 智能体工作负载天然是长上下文、短追加、多轮次的. 在每一轮中, GPU 需要从持久化存储中读取整个上下文的 KV-Cache, 并为追加的 token 执行预填充计算. 作者从代表性的编码任务中收集的追踪数据显示, 平均轮次为 157 轮, 这表明了 LLM 参与多轮交互的趋势. 平均上下文长度为 32.7k, 而平均追加长度仅为 429, 这意味着 KV-Cache 命中率高达 98.7%. 在这种情况下, "缓存-计算比" (cache-compute ratio), 定义为需要加载的 KV-Cache 与所需计算量之比, 对于 DeepSeek-V3.2 来说大约是 22 GB/PFLOP, 这对存储带宽构成了显著的瓶颈. 需要注意的是, DeepSeek MLA 模型的 KV-Cache 大小已经经过高度优化; 对于 KV-Cache 更大的模型 (见表 1), 情况甚至更糟. DeepSeek-V3.2 的比率高于 DeepSeek-V3, 这得益于其稀疏注意力设计, 降低了计算需求.

硬件发展趋势
硬件发展趋势并不完全适合智能体推理工作负载. 近年来, 网络带宽和 HBM 容量的增长已经落后于 GPU FLOPS (每秒浮点运算次数) 的增长, 这使我们在智能体工作负载下面临内存和通信的壁垒. 如图所示, 从 NVIDIA Ampere 到 Blackwell, I/O-计算比率下降了 14.4 倍.

![图片](assets/1f3e13b7cba3.png)

低网卡带宽限制了 KV-Cache 的加载速度, 使得 GPU 处于空闲状态. 此外, 小的 HBM 容量限制了 GPU 核能够同时计算的 BatchSize 大小, 阻碍了计算单元 (如 Tensor Core) 的充分利用.
系统架构缺陷
现有的 LLM 推理系统在不同引擎类型之间表现出严重的存储网络利用率不平衡. 在流行的 PD 分离系统中, 命中 token 的 KV-Cache 仅由 Prefill 引擎直接从远程存储加载. 这种设计将所有的存储 I/O 压力集中在 Prefill 端的 SNIC (存储网卡) 上, 而 Decode 引擎上的 SNIC 则基本处于空闲状态. 因此, 聚合的存储网络带宽无法得到充分利用.
机会
在 PD 分离架构上进行智能体推理的根本性能问题是, 对 KV-Cache 检索的高 I/O 需求以及推理引擎之间不均衡的存储网络带宽利用. 与此同时, 作者观察到,  ScaleOut 的网络流量呈现出间歇性模式, 其总带宽远大于存储网络: 模型推理中使用的集合通信操作以亚毫秒级的间隔爆发. 因此, 一个机会自然而然地出现了: 可以利用 Decode 节点的 SNIC 带宽从存储中加载 KV-Cache, 然后利用速度更快的 ScaleOut 的空闲带宽, 将其传回 Prefill 节点.

## 4. DualPath 系统概览

为了打破 Prefill 端的存储 I/O 瓶颈, 作者提出了一个双路径加载架构, 它从根本上重新思考了在 PD 分离推理中 KV-Cache 是如何被检索的. 基于此架构设计并实现了 DualPath. DualPath 采用了两种在第二章中展示的、被广泛采纳的技术:

**PD分离**, 它分离了 prompt  和 decode 处理以获得更好效率

**逐层 Prefill** , 它避免了 LayerKV 和 PrefillOnly 所识别的在 Prefill 引擎上的 HBM 瓶颈, 并提高了 GPU 利用率.

DualPath 系统由以下组件构成:

**推理引擎 (Inference Engines)**. 每个引擎管理一个 GPU. 引擎被分为 Prefill 引擎 (PEs) 和用于 Decode 引擎 (DEs).

**流量管理器 (Traffic Manager)**. 在第五章中将详细叙述. 每个引擎包含一个流量管理器, 用于执行 (1) 主机-设备内存拷贝 (H2D & D2H), (2) PE 和 DE 之间的 KV-Cache 传输, 以及 (3) 通过存储网卡从/向存储进行 KV-Cache 读写. 作者采用一种以 CNIC 为中心的流量管理方法, 以防止 KV-Cache 流量影响模型推理中的通信.

**请求调度器 (Request Scheduler)**. 在第五章中将详细叙述. 它是一个中央调度器, 接收客户端请求并将其分配到各个引擎. 它也负责在两条路径间动态分配数据流量.

### 4.1 DualPath Load

除了常规的"存储到 Prefill "路径, DualPath 引入了一条新颖的"存储到Decode"路径, 允许 KV-Cache 先被加载到 Decode 引擎, 然后通过 ScaleOut 上的高带宽 RDMA 传输到 Prefill 引擎. 通过在两条路径间动态分配负载, 系统聚合了所有引擎的存储 NIC 带宽, 包括那些原本空闲的 Decode 端 NIC, 并消除了限制现有系统的不对称带宽饱和问题. 这种方法将存储 I/O 从一个单一瓶颈的资源转变为一个全局可池化和可调度的能力. DualPath 的具体数据流描述如下.

Prefill  PE 读取路径 (Prefill PE read path)

Decode阶段
在 DE 缓冲区接收到完整的prompt KV-Cache (包括通过 PE 读取路径加载的 KV-Cache 和新追加 token 的 KV-Cache) 后, Decode阶段开始. DE 首先分配 HBM 并执行主机到设备 (H2D) 的传输 (上图a 中的标签 8 和 9; 上图b 中的标签 6 和 7), 然后在开始解码前释放 CPU 内存. DE 缓冲区的设计给 DRAM 和 CNIC 带来了带宽压力 (一次额外的 H2D), 这本可以通过 GPU Direct RDMA 直接绕过. 然而, 由于在这种场景下生成长度通常很短, 首个 token 时间 (TTFT) 占总端到端请求时间的比例不可忽略. 引入 DE 缓冲区有助于减少 GPU 内存使用. 在 Decode 期间, 每当一个完整的 token 块 (例如 64 个 token) 被累积, 它就会立即被持久化到磁盘.
不同的BLock Layout
采用了两种不同的块布局: 全量块 (Full Block) 和分层块 (Layer Block), 它们分别包含所有层和单层. 对于所有与存储的交互, 采用全量块. 在 PE 读取的情况下, KV-Cache 加载到 PE HBM 和传输到 DE 缓冲区的过程是以逐层流式方式发生的, 两者都使用分层块. 类似地, 对于 DE 读取路径, 从 DE 缓冲区到 PE HBM 的传输使用分层块.

### 4.3 实践挑战

DualPath 架构从根本上重塑了数据移动: KV-Cache 可以直接从存储加载到 Prefill 引擎, 或者间接地通过 Decode 引擎加载, 从而聚合了所有引擎的存储带宽并打破了 Prefill 端的 I/O 瓶颈. 然而, 在一个实际系统中实现这一高层设计引入了三个相互关联的挑战.
细粒度数据传输
逐层执行范式虽然对于克服 HBM 容量限制至关重要, 但它将 KV-Cache 分割成大量细粒度的块. 在存储、主机 DRAM 和 GPU HBM 之间传输这众多细粒度的数据块必须产生最小的开销, 并与计算无缝重叠, 才能实现吞吐量的提升.
流量隔离
DualPath 中复杂的数据路径在 ScaleOut 和 PCIe 链路上都引入了额外的 KV-Cache 传输流量. 一个主要担忧是, 这种流量可能会干扰对模型执行至关重要的、延迟敏感的集合通信操作: 例如EP中的 AllToAll 和 TP/CP 中的 ReduceScatter/AllGather. 由于这些集合通信对端到端推理延迟至关重要, 一个关键挑战在于如何在不降低模型推理性能的情况下, 利用空闲的 I/O 带宽.
动态负载均衡
由于为 KV-Cache 加载采用了两种不同的路径, 系统必须迅速为每个请求决定使用哪条路径. 一个简单的策略可能会使一条路径过载, 重现原来的瓶颈. 流量调度器必须实时平衡多个因素: 存储 NIC 的队列长度、GPU 上的计算负载以及请求的工作负载特性.

## 5. 以 CNIC 为中心的流量管理器

现代 LLM 推理系统采用一系列先进的数据传输技术——例如片上 CUDA 拷贝引擎和 GPUDirect Storage(GDS)——来在存储、主机内存和 GPU HBM 之间高效地移动数据. 然而, 所有这些机制都可能干扰模型执行期间对延迟敏感的集合通信 (例如, 专家并行中的 AllToAll). 这主要出于两个原因:

这类传输技术通常在独立的路径上运行, 这些路径不与ScaleOut网络共享相同的 QoS 控制;

现有的 GPU 不支持 PCIe QoS, 这使得保护模型推理通信免受其他争用 PCIe 带宽的流量的干扰变得困难.

此外, 由于集合通信以快速的、亚毫秒级的突发形式发生, 依赖一个基于软件的流量整形器在这些高优先级流量窗口之间穿插低优先级 I/O 操作是不切实际的.

为了解决这个问题, 作者提出一种**以 CNIC 为中心的数据传输方法**, 这在他们的生产部署中被广泛采用: 所有进出 GPU 的数据流量, 包括本地的 H2D/D2H 拷贝, 都必须通过 GPU 配对的 CNIC (计算网卡), 走 GPUDirect RDMA 的数据路径. 通过将所有流量整合到计算网络上, 因此可以利用计算网络原生的 QoS 能力来强制执行严格的流量区分.

### 5.1 流量隔离

对于基于 InfiniBand 的网络, 利用虚拟通道 (Virtual Lanes, VLs) 来强制隔离不同类别的流量. 所有的模型推理通信流量被分配到一个专用的高优先级 VL, 而所有其他流量, 包括 KV-Cache 传输, 则被映射到一个独立的低优先级 VL. 作者将所有网络交换机和 NIC 的 VL 仲裁器配置为加权轮询策略, 为高优先级 VL 保留大约 99% 的总带宽. 剩余的带宽分配给低优先级 VL 以防止饥饿. 这种配置确保了模型执行流量几乎不受 KV-Cache 传输的影响, 同时仍然允许 KV-Cache 流量投机地利用计算网络中其他时候空闲的带宽.

尽管作者的实验是在基于 InfiniBand 的网络上进行的, 但同样的设计原则可以自然地扩展到其他互连技术. DualPath 可以部署在 RoCE 上, 通过Traffic Class和 DSCP 与硬件数据包队列相结合来实现. 新兴技术如 UnifiedBus 和 Ultra Ethernet 同样在为异构流量融合 QoS 机制, 这可以直接支持 DualPath 的需求.

### 5.2 CNIC辅助的KV-Cache拷贝

现有的 GPU 数据传输技术包括 GPUDirect Storage (将 KV-Cache 从存储后端加载到 GPU HBM) 和 CUDA 拷贝引擎 (通过 PCIe 直接拷贝主机 DRAM 到 GPU). 然而, 这些方法未能将 KV-Cache 流量与模型执行中高优先级的、对延迟敏感的集合通信隔离开来, 严重降低了推理性能.

为了解决现有方法的局限性, 作者采用了一种 **CNIC 辅助的 H2D/D2H 数据路径**. 对于 KV-Cache 加载, 首先将 KV-Cache 从存储后端读入主机 DRAM. 然后, 向 GPU 配对的 CNIC 提交一个 RDMA Write 请求, 以执行本地的 H2D 拷贝. 存储新生成的 KV-Cache 遵循一个对称的过程: 它首先通过 CNIC 传输到主机 DRAM, 然后通过存储网络持久化到存储后端. 这种设计将 CNIC 确立为所有 GPU PCIe 流量的中央 QoS 调度器, 允许其 VL 仲裁器优先处理推理通信流量, 并使用空闲的 PCIe 带宽执行 KV-Cache 传输.

作者还观察到, 在处理大量小数据块时, CNIC 辅助的 H2D 和 D2H 优于 CUDA 拷贝引擎. 测量显示, 通过 `cudaMemcpyAsync` 提交单个拷贝操作会产生大约 5-7µs 的延迟开销. 由于 CUDA 驱动程序的闭源性质未能进一步分解此开销. 相比之下, 提交一个 RDMA Write 工作请求仅涉及用户空间中对 NIC 寄存器的几次 mmio 写入, 耗时仅约 1µs. 此外, 通过利用 doorbell batching, RDMA 工作提交的开销可以被显著摊销.

## 6. 自适应请求调度器

尽管理论分析显示了有希望的结果, 但不均衡的负载会降低硬件利用率. 在这种场景下, 需要同时考虑两个维度的平衡: (1) NIC 流量, 和 (2) GPU 的利用率均衡. 作者将调度分为两个级别:

**引擎间调度 (inter-engine scheduling)**, 它将请求分配给一个 (PE, DE) 对, 并为每个请求选择读取路径 (PE 或 DE);

**引擎内调度 (intra-engine scheduling)**, 它决定哪些请求被包含在每个前向计算批次中.

数据并行在注意力层中被广泛采用, 特别是对于 MLA 模型. 在这种并行配置下, 每个 GPU 服务于一组不同的请求. 这可能导致必须在注意力阶段后同步并一起进入 FFN 阶段的所有 GPU 之间出现工作负载不平衡, 从而导致 GPU 产生等待其他对等节点的空泡 (bubble). 因此, 需要确保它们有相似的注意力层执行时间, 以最小化等待空泡.

算法
以 FIFO 顺序不断添加请求, 只要预测的注意力层执行时间不超过一个预定义的上限, 称为**计算配额 (compute quota)**. 如果添加一个请求会超过这个上限, 对  进行二分搜索, 找到一个更小的  以适应剩余的计算配额, 并对该请求执行分块预填充.

**选打包工(DE)**: 先把所有打包工分成几个大组, 把包裹扔给目前最闲的那个组. 然后, 组长会在自己的组员里, 找一个桌子(HBM)还有空位、手上活儿又不多的组员来接手.

**选仓库门**: "装卸工"和"打包工"都定好了, 就看谁那边的仓库门人少, 就让货物从那边进来.(DualPath), 当然 未来也可以拆分好一边进一点.

**安排工作节奏 (引擎内调度)**

"装卸工"(PE) 面前可能排了一队包裹. 他不能随便装.

**问题**: 如果几个装卸工需要合作搬一件大东西, 必须等所有人都准备好才能一起抬. 如果有人快有人慢, 快的就得干等着.

**解决方案 (计算配额)**: 队长给每个人发了个秒表, 说: "这一轮, 你们每个人的工作时间不能超过5分钟(compute quota)." 装卸工就从自己的任务列表里拿包裹, 只要估计总时间不超过5分钟, 就都拿过来一起干. 如果下一个包裹太大, 5分钟干不完, 他就先把这个大包裹拆开, 只拿一部分进来, 保证自己能准时完工. 这样, 所有合作的装卸工都能在差不多时间完成, 然后一起去干下一件事, 没人需要等待.

## 7. 评估

### 7.1 实现

DeepSeek在内部的推理框架之上实现了 DualPath. 对于 CUDA 核, 内部框架采用了 FlashMLA, DeepGEMM, 和 DeepEP 的组合, 这与当前主流的开源框架 SGLang, vLLM 保持一致. DualPath 的实现涉及在其之上大约 5000 行的代码修改. 采用 3FS 作为分布式存储, 并使用一个类似 io_uring 的接口进行kernel bypass.

### 7.2 实验设置

**测试平台 (Testbed)**. 实验在一个使用 InfiniBand 互连的 GPU 服务器集群上进行. 每台服务器有 8 个 NVIDIA Hopper GPU 和双处理器. 此外, 每个节点都配置了八个连接到 InfiniBand 网络的 400Gbps RDMA NIC 和一个连接到 3FS 的额外存储 NIC. 计算和存储网络是物理隔离的. 集群范围的 3FS 没有内部 DRAM 缓存, 并且可以饱和存储 NIC 的 400Gbps 带宽.

**模型 (Models)**. 在三个模型上进行评估:

DeepSeek V3.2 660B, 一个带有 DeepSeek 稀疏注意力的 MoE 模型, 记为 **DS 660B**;

DS 660B 的 27B 缩小版, 记为 **DS 27B**;

Qwen2.5-32B , 一个带有 GQA 的Dense模型, 记为 **Qwen 32B**.

DS 660B 和 Qwen 32B 对应于 HuggingFace 上公开发布的检查点. DS 27B 是内部的实验模型, 其架构与 DS 660B 相似.

**数据集 (Datasets)**. 从生产环境智能体 RL 训练工作负载中收集了三个具有不同最大上下文长度 (MaxLen) 的智能体追踪数据集. 每个数据集包含 500 个轨迹. 平均交互轮次 (Turns), 每轮平均追加和生成的 token 数 (Append 和 Gen), 平均总 token 数 (Total), 以及平均上下文 token 数 (Context) 总结在表中.

![图片](assets/2b4c80f3c8f2.png)

**基线 (Baselines)**. 将 DualPath (记为 **Ours**) 与以下基线进行比较:

**SGL(MC)**: SGLang (commit 19089aa) 加上 HiCache , 启用了 Mooncake 存储并将 3FS 作为存储后端, 并使用 Mooncake 传输引擎进行预填充-解码分离. 由于SGLang 缺少对 DS 27B 的支持, 没有运行

**Basic**: 未修改的内部推理框架. 由于实现上的差异, 比较 DualPath 和 SGL(MC) 是不公平的. 因此只报告从 Basic 到 Ours 的性能提升.

**Oracle**: 基于 DualPath, 绕过了所有的磁盘读取, D2H & H2D 传输, 以及 PD 间的 KV-Cache 传输. 这个配置代表了假设 I/O 开销为零的理论性能上限.

**P/D 比例和并行策略 (P/D Ratio and Parallelism)**. DS 660B 默认使用 2P4D, Qwen 32B 使用 1P2D, DS 27B 使用 1P1D. 对于 DS 模型, 使用专家并行(EP)和数据并行(DP). 对于 Qwen 32B, 在 DualPath 中仅使用 DP, 而 SGL(MC) 使用 TP=8, 因为 SGLang 不支持该模型的 DP attention.

**指标 (Metrics)**. 对于离线推理场景, 测量整个任务的**作业完成时间 (JCT)**. 对于在线服务场景, 测量**首个 token 时间 (TTFT)**, **第二个 token 时间 (TTST)**, 以及**每输出 token 时间 (TPOT)**.

### 7.3 离线批量推理

本节评估离线批量推理的吞吐性能, 这是 RL 训练中 rollout 阶段的情况. 在此场景中,  个智能体同时开始 rollout, 我们测量所有请求完成时的 JCT.

**改变智能体批次大小 & 最大智能体长度 (MAL)**. DualPath 从更大的批次大小和更长的 MAL 中获益更多. 下图报告了不同批次大小和 MAL 下的 JCT. SGL(MC) 在我们的设置中遇到错误, 未能完成一些大型配置 (标记为 N/A). 在 DS 660B 上, DualPath 相较 Basic 实现了高达 **1.87倍** 的提升, 并且性能接近 Oracle, 表明 KV-Cache I/O 在很大程度上被消除了.

![图片](assets/63f47fdcc426.png)

在 DS 27B 上, DualPath 相较 Basic 提升高达 1.78倍, 但仍比 Oracle 慢 1.09-1.85倍, 这是因为 1P1D 配置下的存储带宽有限 (图 8). 对于 Qwen 32B, 它显示出与 DS 27B 类似的趋势.

![图片](assets/f5f1cd2d97cc.png)

**改变追加长度 & 生成长度**. 当追加和生成 token 较短时, DualPath 优势更明显. 更长的追加长度意味着更大的 GPU 计算压力, 更长的生成长度由于预填充间隔时间更长而降低了 KV-Cache 加载压力. ... 如图 9 所示, 随着追加长度增加, Basic 的性能逐渐接近 DualPath 和 Oracle, 而 DualPath 和 Oracle 的性能变化很小, 表明瓶颈始终在于 GPU 计算压力. 与 Basic 相比, DualPath 在不同追加规模下实现了 1.82-1.99倍的加速. 生成长度缩放的趋势类似.

![图片](assets/9611e7b8f427.png)

**改变P:D比例**. 在所有比例下, DualPath 都比 Basic 表现出显著的性能增益.如图 8 所示, DualPath 在所有配置中平均实现了 1.64倍的加速 (最高可达 2.46倍). Basic 1P1D 和 Basic 1P2D 的性能相当; DualPath 1P1D 和 Basic 2P1D 的性能也相当, DualPath 2P1D 和 DualPath 1P2D 也是如此. 这是因为每对系统具有等效的可用存储带宽 (Basic 只能使用预填充节点的存储带宽, 而 DualPath 可以利用所有节点), 这证实了存储带宽是智能体场景中的主要瓶颈.

### 7.4 在线服务

**方法论.** 评估了系统在不同智能体到达率 (每秒到达数, APS) 下的延迟特性. 智能体根据泊松过程以指定速率到达, 每个智能体在到达时从第 0 轮开始重放到其最后一轮. 在实验中, 服务等级目标 (SLO) 设置为 TTFT ≤ 4 秒, TPOT ≤ 50ms. 在 TPOT 和 TTST 图中, 超过 SLO 阈值的数据点被省略. 实验终止的触发条件是: (1) TTFT 超过 4 秒, 或 (2) 系统达到稳态, 定义为在一个 150 秒滑动窗口内的 TTFT 变化与 30 分钟前相比低于 5%.

如图 10 所示, DualPath 实现了比 Basic 更高的 APS 容量 (DS 27B 为 1.67 倍, DS 660B 为 2.25 倍). DualPath 的 TTST 与 Basic 相当, 而 TPOT 表明 DualPath 相对于 Basic 没有引入额外的解码开销. SGL(MC) 表现出异常低的 TTST, 可能是由于实现问题, 导致前两个 Token 几乎同时到达客户端. 对于 DS 27B, 所有指标的趋势与 DS 660B 类似. 然而, Basic 和 DualPath 的 TPOT 都显著高于 Oracle, 这表明在小模型情况下, 基础的 P-D 传输开销是相当可观的. 我们将其作为未来工作.

图 11 展示了两种模型的平均 JCT. 关于工作集影响的详细分析在第八章中讨论.

![图片](assets/8d68b61b7980.png)

如图 12 (左) 所示, DualPath 在不同 APS 下保持了稳定的 TTFT 组成部分, 而 Basic 的排队时间由于存储带宽不足而急剧增长.

![图片](assets/40e7a7f201bd.png)

### 7.5 消融实验

作者进行了一项消融研究, 以量化 DualPath 中每个技术组件的贡献. 实验在离线推理设置下进行, MAL 为 64K, 智能体批量大小为 1024 和 2048. Basic 和 Ours 之间的差异被分为三个技术: 逐层 prefill (layerwise prefill), 双路径加载 (dual-path loading), 和调度算法 (scheduling algorithm). 通过逐步添加这些技术来展示各自的贡献. 如图 12 所示, 与 Basic 相比, 添加逐层 prefill 平均减少了 17.21% 的 JCT, 缓解了 PE 的 HBM 瓶颈并隐藏了传输开销. 在逐层 prefill 之上添加DualPath加载带来了主要的性能增益, 平均减少了 38.19% 的 JCT, 因为它使请求可以从 PE 或 DE 读取 KV-Cache, 充分利用了分布式存储带宽. 最后, 在DualPath加载之上采用调度算法来决定 KV-Cache 加载路径实现了最佳性能, 与 Basic 相比减少了 45.62% 的 JCT, 证明了跨存储网卡进行负载均衡调度的有效性.

**负载均衡 (Load Balance).** DualPath 的调度算法改善了存储网卡和 attention 层执行时间的负载均衡. 对于存储网卡, 我们的调度算法将负载均衡度 (最大/平均流量比) 从 1.53 提高到 1.18 (相较于轮询调度), 如图 13 所示.

![图片](assets/aa8641574334.png)

对于 attention 层, DualPath 在任务的前 5% 期间将最大/平均执行时间比维持在低至 1.06, 减少了 GPU 空闲气泡 (图 14).

![图片](assets/3c3809080271.png)

存储网卡指标是在一个小时间窗口内, 三台机器上所有存储网卡的最大流量与平均流量之比, 1.0 代表完美均衡. attention 层指标是在一个专家并行组内的所有 GPU 之间为每个前向传播计算的. 随着任务进展, 由于系统负载不足, 这两个比率变得无意义. 因此, 没有展示工作负载的尾部阶段.

使用 1,152 个 GPU 进行了离线和在线实验, 以展示生产级别的可扩展性 (表 3).

由于缺乏精细调整的并行设置和 P/D 比例 (这需要大量的实验预算), 大规模实验没有展示出比多个具有同等成本的小规模单元额外的 JCT 或服务能力增益. 然而, 大规模部署仍然很重要, 原因如下. 首先, 它减少了碎片化, 并为微调并行策略和 P/D 比例提供了更大的灵活性. 其次, 大规模部署提供了更多的调度机会, 以缓解不可预测的突发在线请求下的排队延迟. 这些观察为未来的工作指明了几个方向.

## 8. 讨论

### 8.1 未来潜在的工作

离线推理的工作负载是高度动态的. 例如, 在智能体强化学习任务中, 工作负载在很大程度上取决于研究人员的算法设计, 并且 prefill 阶段在执行的前半段通常比后半段承受着显著更高的压力. 同时, 对这些探索性实验进行性能剖析 (profiling) 的成本很高, 因为一些实验可能只会运行有限的几次. 因此, 需要更具适应性和灵活性的方法来配置并行策略和 P/D (Prefill/Decode) 比例, 例如使用模拟器或在线调整机制. 其次, 调度算法仍有改进空间, 因为期望在大规模部署下实现更低的 TTFT (首个 Token 时间) 百分位数.

### 8.2 工作集分析

如图 11 所示, 给定一个到达率  (即每秒的新轨迹数) 和平均 JCT (作业完成时间) , KV-Cache 的工作集大小可以近似估算为 . 在对 DS 660B 服务的设定中, DualPath 的这个值范围从 APS 为 0.1 时的 69 GB 到 APS 为 0.45 时的 681 GB.

在生产环境中, 工作集会更大, 因为评估假设了零到达间隔时间和零工具调用延迟. 如果由于这些间隙导致 JCT 增加了  倍, 系统的 APS 容量会增加  倍 (因为这些间隙不会给 LLM 推理带来压力), 导致工作集扩大  倍. 这将超出可用的内存并降低分布式内存池的命中率. 进行这类实验需要  倍的机器小时数和  倍的存储空间 (成本按  比例扩展), 考虑到有限的资源, 这是无法承受的.

## 9. 一些展开的分析

总体来看, 这篇工作是非常出色的, 很好的解决了KVCache传输对ScaleOut网络中集合通信的干扰, 同时配合调度算法实现了接近2的性能提升. 但是从体系结构的视角来看, 为什么不在ScaleOut接入存储呢? 或者更进一步的话题, 在未来ScaleUP中接入存储是否也可以呢?

当然这个工作也受到了现阶段体系机构的影响, 为了避免PCIe上的拥塞以及KVCache和集合通信的干扰, 采用了以ScaleOut CNIC为中心的调度算法, 整个数据路径还是非常长且很复杂的:
![图片](assets/4d04c4331f1d.png)

### 9.1 Nvidia RDMA ScaleOut接入存储的限制

对于DeepSeek采用的是IB组网, 可以很好的支持Adaptive Routing, 但是IB直接接入存储会导致成本非常高. 但是如果换用RoCE使用以太网, 实际上NV宣传的Spectrum Adaptive Routing或者基于BRCM的多路径DLB/GLB都有很多缺陷. 在EP之前很多网络采用多轨道(Multi-Rail)的部署来尽量避免Hash冲突, 但是随着EP并行的出现, 这种多轨道的方式也带来了很多缺陷. **因此, 基于以太网的RDMA还是要做好多路径负载均衡才行**. 另一方面这种多轨道的部署使得存储集群挂载到哪个轨道上, 或者每个轨道挂一个, 总体来看都是成本很高的方案.

另一个问题和拥塞控制相关, DeepSeek为了解决PCIe上的拥塞以及避免对集合通信的影响采用了以CNIC为中心的调度. 即便是机内的H2D和D2H流量也需要通过一个local的回环QP使用RDMA来进行CPU和GPU之间的内存交互. 实质性的问题还是在于Nvidia CX系列网卡的拥塞控制做的很简单, 可靠传输也做的不行导致的.

然后还有一个问题, 对于存储集群接入到计算网络中时, 肯定要设置一定的收敛比, 按照业务收益来看1:3的收敛比即可.

### 9.2 ScaleOut/ScaleUP接入存储的探讨

实际上DeepSeek这篇文章给出的一个结论就是: FrontEnd无法承载Agentic LLM的推理和RL训练的需求, 增加FrontEnd带宽也会带来成本的显著上升. 那么既然ScaleOut都能够通过CNIC传输KVCache并且不干扰集合通信. 那么直接通过ScaleOut接入分布式存储也应该可行, 并且在集合通信间歇期, ScaleOut的网络带宽更大.  对于编程而言, 采用GDS或者GIDS也是非常容易实现存储接入的. 可能我只需要很简单封装一个put/get命令即可. 只是在这个场景下, 需要将集合通信和Storage的QoS做好即可.

对于阿里云CIPU而言, 首先我们采用的Window based拥塞控制, 可以很容易的对带宽进行控制, 并针对集合通信和KVCache的QP动态的分配带宽资源进行流控, 特别是在集合通信高优先级抢占下, 存储流量丢包的情况下, 由于完善的SACK和快重传机制也可以很快的重传数据. 在存储写的方向上一个简单的window抑制即可, 在读方向上对于READ的inflight描述符进行控制也可以很好的以CNIC为中心进行流控. 另外我们完善的多路径负载均衡能力也可以在计算-存储的多跳网络中实现负载均衡最大化Fabric的利用率.

我们注意到Nvidia其实也在考虑ScaleOut接入存储, 如下图所示, 他们也在尝试着构建东西向(EW, 即ScaleOut网络)接入存储节点的能力

![图片](assets/f6285852c15b.png)

结论
ScaleOut可以承载存储流量, 并接入到存储网络中, 实现FrontEnd和ScaleOut并网. 对于存储和集合通信的干扰问题也可以通过QoS很好的解决. 相对于DualPath会更加干净.

进一步来看, 由于PCIe带宽约束和缺乏QoS控制机制, 将能够接入存储的NIC直接接入到ScaleUP Fabric甚至在ScaleUP域内构建一些额外的内存节点也是工业界正在演进的方向. 例如Nvidia未来在CX10中有接入NVLink的打算来避免PCIe的诸多限制. 在前面一篇文章已经有详细的讨论[《谈谈基于ScaleUP的存储扩展和三网融合》](https://mp.weixin.qq.com/s?__biz=MzUxNzQ5MTExNw==&mid=2247497421&idx=1&sn=e921514eb03119288952963e82b47850&scene=21#wechat_redirect)
