网工常有三板斧,`Ping` `Traceroute` `iperf`. 但是这些工具已经用了很多年了不适应现代网络的效率了. 于是最近花了一些时间写了一些让自己顺手的小工具，并开源出来供大家使用.

### HTTP测速

代码及下载地址: https://github.com/zartbot/sdwan-perf

可使用平台: 

`windows` `mac` `linux(arm+x86)` `Cisco IOSXE`

iperf测试通常需要server端和client端同时搭建，在SDWAN等环境下测量网站的速度就不是那么方便了，而且通常我们在构建特定业务QoS时需要直接对公网的HTTP或者HTTPS测试，并且还要检测p95的延迟数据和并行访问的成功率，于是基于go fasthttp库写了一个测速软件

![图片](assets/206bcefdd09a.gif)

您可以使用如下方式测速：

```
./sdwan-perf_linux -role=client -url=https://www.google.com
```

默认会使用10个client测试60s并给出实时统计的延迟和带宽:

```
   SDWAN Performance Test Report+--------------+---------------------------+--------------------------+|    Stats     |        Latency(ms)        |  Bandwidth(Per Session)  |+--------------+---------------------------+--------------------------+| mean         |               166.82ms    |                75.35Mbps || Jitter       |               770.53ms    |                          ||              |                           |                          || Min          |                87.00ms    |                 0.80Mbps || p25          |               102.00ms    |                72.74Mbps || p75          |               110.00ms    |                78.44Mbps || p90          |               113.00ms    |                81.64Mbps || p95          |               116.00ms    |                83.34Mbps || p99          |               136.00ms    |                86.03Mbps || Max          |              9978.00ms    |                91.97Mbps |+--------------+---------------------------+--------------------------+| Count: 16102 | Error: 312 | Timeout: 300 | Total-BW:    7534.52Mbps |+--------------+---------------------------+--------------------------+
```

您可以使用如下方式自己搭建server

```
./sdwan-perf_linux -role=server -port=8000,8001,8002,8003
```

然后对自己的server进行不同大小文件的测速, 多核情况下最大吞吐可以到20Gbps HTTPS流量，基本满足大多数测速场景

```
./sdwan-perf_linux -role=client  -duration=100 -server=127.0.0.1 -port=8001 -size=1000000 -num=100
```

### Traceroute

代码及下载地址: https://github.com/zartbot/ztrace

可使用平台: 

`mac` `linux(arm+x86)` `Cisco IOSXE`

Windows因为RawSocket的问题，同时又不想引入libpcap就暂时没有支持了，MAC TCP Traceroute模式也因为安全限制不支持.

系统自带的traceroute通常做完一次需要耗时1分钟:

![图片](assets/83f00c24ae2c.gif)

通常对于路径上的运营商情况也不了解，丢包情况也不清楚，因此还要whois或者其它手段去查运营商会非常麻烦，于是就自己动手搞了一个一秒出结果的并行traceroute的工具, 可以自动关联地理位置信息和运营商信息，然后在终端对于高延迟和高丢包的结果会自动染色成黄色、红色:

![图片](assets/f3851a5428e0.gif)

使用方式:

```
Usage of ./ztrace_linux_x86:  -dest string     Destination   -path int     Max ECMP Number (default 16)  -proto string     Protocol[icmp|tcp|udp] (default "udp")  -rate float     Packet Rate per second (default 1)  -src string     Source   -ttl int     Max TTL (default 64)  -wide     Widescreen mode
```

### DNS服务器

代码及下载地址: https://github.com/zartbot/zadns

可使用平台: 

`windows` `mac` `linux(arm+x86)` `Cisco IOSXE`

其中仅有Linux X86平台支持带Tensorflow的神经网络DGA域名过滤, ARM准备编译一个libtensorflow后再开放出来.

![图片](assets/ab0d0d8d4326.gif)

由于CDN的广泛部署及其负载均衡的复杂性，通常使用不同的DNS服务器**会有不同的结果，而主机只能设置Primary和Secondary DNS服务器，缺乏基于域名的灵活规则，同时针对木马等高危软件或者高危地区IP地址结果缺乏过滤机制，虽然有OpenDNS**或者Umbrella等业务，但是给到这样的集中式平台反馈的CDN站点还是有些问题.

#### 优化功能

`ZaDNS` 可以在第一接收到DNS请求时，根据域名向特定的服务器组并行发送多个DNS请求，并等待600ms后将所有的结果汇总发送给客户端，此时DNS `TTL=0`，保证客户端至少在600ms内能够连接到服务器.

![图片](assets/f52064735d50.png)

与此同时我们将返回的结果放入服务器本地缓存，并调用一个LFU ProbeCache，对访问频率最高的N个地址进行TCP延迟探测。由于客户端前一次收到DNS TTL=0的记录不会触发本地缓存，因此第二次DNS请求时，我们可以根据TCP探测的结果自动回复最快的三个服务器地址给客户端:

![图片](assets/76cfe980e60f.png)

所以最终就会看到这样的效果:

![图片](assets/66fabebc6a31.png)

#### 安全功能

在Linux X86平台支持带Tensorflow的神经网络DGA**域名过滤功能， 具体原理参考<支持AI的ZaDNS服务器>[1]

通过这样可以很好的过滤大量使用DGA算法产生的和木马控制主机C2的连接. 然后最近28天的所有DNS请求日志也自动存放在log目录下，可以供您自己分析和回看.例如监控家里的IOT设备是否被劫持了，小孩是否访问了不健康网站...

同时我们还针对很多Netflow**服务器需要反查五元组信息的需求，我们会自动将访问到的A、AAAA记录构建到PTR缓存中，Netflow服务器或者安全日志分析服务器可以直接向ZaDNS发送PTR查询IP地址对应的Domain进行轻量级的应用识别.

#### ZaDNS零信任和网优

这部分您可以根据自己的业务需求和路由器控制面整合

很多云原生业务都有DNS实现业务发现和注册，这些在K8s里面已经很普遍了，而针对广域网侧或者用户端侧，例如SDWAN、SDA，我们的实现是LISP**动态加载Overlay地址进转发表实现的，或者在SDWAN路由器上构建防火墙实现广域网的安全隔离。为了将整个链路拉通对齐，传统的零信任网络架构需要一个控制器，那么不干脆把这些东西全部整合进DNS里面。但是DNS无法对终端进行鉴权，还好DNS over TLS和DNS over HTTPS的出现可以解决这些问题了、这样我们就可以通过DNS知道客户端需要访问某个服务，然后自动的通过DNS鉴权客户后修改网关的ACL和路由表实现动态的策略通行。
