## 📘 1. Overview

今年年初其实就在关注这一部分的内容, 并写了一篇文章[《谈谈大模型架构的演进之路, The Art of memory.》](https://mp.weixin.qq.com/s?__biz=MzUxNzQ5MTExNw==&mid=2247493032&idx=1&sn=206eed2e4127b9971a1e0c380f70b082&scene=21#wechat_redirect), 试图通过大模型架构构造一个自己能够产生代码运行的通用计算机架构, token as instruction. 当脑子里补出这图的时候, 就豁然开朗了. 大模型从自回归可能真的要走向自生成Instruction的路了.... 那么构造一个大模型的冯诺伊曼架构大概就如下了.

![图片](assets/5f95f7bf6148.png)

从体系结构的视角上,Attention如果要作为一个图灵完备的控制计算单元, 那么必须要具备条件分支和循环/递归的能力. 当然另外还要考虑一些不可计算问题, 例如停机问题, 进行一些有效的控制.

相对于RNNs的模型按顺序处理序列, 但是其串行计算的方式使其难以高效的训练. Transformer的巨大优势在于并行处理整个序列, 训练速度快, 并且通过自注意力机制拥有全局感受野. 但是Transformer是一个固定深度的前馈网络, 它缺乏RNN的循环归纳偏置(recurrent inductive bias). 这导致它在很多需要迭代或递归推理的任务上泛化能力较差.

那么如何既要实现Transformer的并行性高效计算的能力, 同时又要高效的引入条件分支和和循环/递归的能力呢? 这就是Universal Transformer(UT)以及Mixture-of-Recursions需要解决的一些工作.

当然这些循环不要, 把模型做深一些的流水线是否可行呢? 很显然模型越深, 推理的延迟越大, 并且在FFN层转换到大量跨机的访问.

UT的做法是在Attention中引入更多的计算实质性的问题, 还是需要更好的Data Locality来提升整个系统的性能.

MoR的做法是希望尽早的退出循环, 降低平均的深度

## 📘 2. Universal Transformer

这篇于 ICLR 2019 发表的论文介绍了一种名为 Universal Transformer (UT)[1] 的新型序列模型. 它旨在结合两种主流序列模型架构的优点: Transformer 的并行计算能力和全局感受野, 以及 RNN (Recurrent Neural Network) 的递归归纳偏置 (recurrent inductive bias). 论文的核心论点是, 标准 Transformer 由于其固定的、有限的层级深度, 在处理需要迭代或递归推理的任务时泛化能力不足, 而 UT 通过引入深度方向的递归和动态计算机制, 成功地解决了这一问题, 并在理论上达到了图灵完备性.

### 2.1 核心思想和动机

**动机**: 标准的 Transformer 模型在机器翻译等任务上取得了巨大成功, 其优势在于完全并行的自注意力机制, 训练速度远超序列化的 RNN. 然而, 它的一个关键弱点是其结构是固定的 "层堆叠" 模式. 一个 N 层的 Transformer 只能执行 N 步固定的、顺序的计算. 这使得它在学习需要根据输入长度进行不同步数迭代计算的算法任务 (如字符串拷贝、加法) 时, 泛化到未见过的长度时表现很差. 它缺少 RNN 那种对任意长度序列进行迭代处理的 "归纳偏置".

**核心思想**: Universal Transformer (UT) 将 Transformer 从一个 "深度的前馈网络" (deep feed-forward network) 改造为一个 "深度的循环网络" (deep recurrent network). 但这个 "循环" 并非像 RNN 那样在序列的时间/位置维度上进行, 而是在模型的深度 (depth) 维度上进行.

具体来说, UT 对序列中的每一个位置 (position) 都维护一个表征向量. 在每个计算步骤 (time-step) 中, 模型会并行地更新所有位置的表征. 这个更新过程包含两个阶段:

`序列内通信`: 使用自注意力机制, 在所有位置之间交换信息, 类似于标准 Transformer 的一层.

`局部转换`: 对每个位置的表征独立地应用一个相同的转换函数 (Transition Function), 这类似于 RNN 的状态更新门.

由于转换函数和自注意力模块的参数在所有深度步骤中是共享的, 模型可以理论上迭代任意次, 从而模拟出任意深度的网络. 这引入了 RNN 的递归特性, 同时保留了 Transformer 的并行计算优势.

### 2.3 动态停机机制 (Dynamic Halting)

为了让模型更加灵活和高效, 论文引入了`自适应计算时间 (Adaptive Computation Time, ACT)`机制, 即动态停机.

主要的想法是: 序列中的不同Token可能需要不同程度的计算. 简单的词可能一步就处理好了, 而复杂的、有歧义的词可能需要更多轮的迭代. 具体实现: 在每个递归步骤和每个位置, 模型会额外输出一个`停机标量(halting scalar)`. 模型会累加这些标量, 当某个位置的累加概率超过一个阈值 (如 1.0) 时, 该位置的计算就 "停止". 在后续的步骤中, 该位置的表征向量会直接被复制, 不再更新.

![图片](assets/7c704a325fd2.png)

这种做法有如下一些优势:

`效率`: 为简单任务分配更少的计算资源.

`性能`: 允许模型为困难的输入分配更多的计算资源, 从而可能获得更好的性能.

`正则化`: 论文发现, ACT 机制也像一种正则化手段, 鼓励模型用更少的步骤解决问题, 反而能在某些任务上获得比固定步数更优的结果.

`lstm, gru`: 在每次循环后,使用一个类似 LSTM 或 GRU 的门控单元来更新状态,将 RNN 的思想融入 UT.

### 2.4 UT小结

标准 Transformer 的局限: 其计算的 "顺序深度" (sequential depth) 是由网络层数固定的, 与输入长度无关. 这意味着它无法模拟那些所需计算步骤随输入长度增长的算法. 从理论上讲, 拥有有限状态和固定计算深度的模型无法做到图灵完备.

UT 的优势: UT 的计算深度 T不是一个固定的超参数, 而是可以根据输入长度 N 动态变化的 (例如, 我们可以设置 T=N. 由于其参数共享的递归结构, 它可以执行任意多次计算. 论文指出, 通过这种方式, UT 可以模拟 Neural GPU, 而 Neural GPU 已被证明是图灵完备的. 因此, 在某些假设下 (如提供足够的内存和输入相关的计算步数), Universal Transformer 是图灵完备的.

Universal Transformer 的主要贡献可以归纳为两点:

`引入递归归纳偏置`: 通过在深度上共享自注意力和转换函数的权重, UT 将 Transformer 的并行性和 RNN 的递归性完美结合. 这使得模型能够学习迭代式算法, 并在多种任务上展现出更强的泛化能力.

`实现条件计算`: 通过集成动态停机机制 (ACT), UT 能够根据输入的复杂度动态调整每个位置的计算量. 这不仅提升了模型的效率和性能, 还起到了一种有效的正则化作用.

## 📘 3. Mixture-of-Depths

### 3.1 Overview

在谈Mixture-of-Recursions之前, 我们来谈谈Mixture-of-Depths[3], 这也是Google DeepMind在2024年的一个工作. 主要目的是解决标准 Transformer 模型在计算资源分配上的低效问题. 传统 Transformer 对序列中的每个 token 都施加相同的计算量, 而 MoD 则让模型学会动态地、有选择地将计算资源分配给序列中最重要的 token.

`实现机制`:

`静态计算预算`: 通过预先设定一个上限 k, 限制在任何给定层中能参与自注意力 (self-attention) 和 MLP 计算的 token 数量.

`动态路由`: 网络使用一个 top-k 路由机制来动态决定哪些 token 应该被处理.

`硬件友好`: 由于 k 是预先定义的, 这个过程使用一个静态的计算图 (static computation graph) 和已知的张量大小, 这与其他可能引入动态计算图的条件计算技术不同, 对现有硬件非常友好.

`结果`:

模型不仅学会了动态分配计算, 而且效率很高.

在相同的训练 FLOPs 和训练时长下, MoD 模型可以达到与基线模型相当的性能.

更重要的是, MoD 模型每次前向传播所需的 FLOPs 更少, 在推理采样阶段的速度可以提升高达 50%.

与相关技术的关键区别:

`与MoE的区别`: MoE 的路由是在多个 "专家" (通常是不同的 MLP) 之间选择. MoD 的选择更简单: 要么执行一个标准的 Transformer 块 (自注意力 + MLP), 要么通过一个残差连接直接跳过, 节省计算. 于此同时MoD还可以和MoE结合起来构成MoDE.

与Early-Exit机制的区别: 早退机制一旦决定一个 token "退出", 它就会跳过所有后续层. 而在 MoD 中, 一个 token 可以在中间层被跳过, 但在更深的层中重新参与计算. 作者推测这是一个非常有用的特性.

### 3.2 MoD详细实现

#### 3.2.1 定义计算预算

MoD 通过降低计算容量 (capacity) 来实现比标准 Transformer 更小的计算预算. 容量定义了参与某项计算 (如自注意力) 的 token 总数. 标准 Transformer 的容量是 T (序列中的总 token 数). MoD 将自注意力或 MLP 的容量设置为 k (其中 k < T). 这直接带来了计算量的节省. 例如, 如果容量减少一半 (k = T/2), 自注意力中查询与键的点积计算量将变为原来的 25% (因为  vs ).

#### 3.2.2 路由机制

论文考虑了两种学习型路由方案 (如图所示):

![图片](assets/14fc99525d47.png)

`Token-choice`: 每个 token 根据自己的偏好选择计算路径. 缺点是可能导致负载不均衡, 需要额外的平衡损失函数.

`Expert-choice`: 每条计算路径 (即 "专家") 根据所有 token 的偏好, 选择 top-k 个 token 进行处理. 这样能保证完美的负载均衡.

图中显示MoD 采用了 Expert-choice 路由. 实际上我个人认为描述成Transformer-Block-Choice可能更好. 即一个Transformer block选择top-k 个最需要计算的 token, 其余的 T-k 个 token 自动走bypass路径. 而这种机制也就构成一种并行的分支跳转的执行方式.  类似的和下图SIMT的分支执行对比, 把PC类比于模型自身的深度

### 3.3 采样问题与解决方案 (Sampling)

top-k 操作是非因果的 (non-causal), 因为确定一个 token 是否属于 top-k 需要知道它后面 token 的路由权重, 这在自回归采样中是不可能的.论文提出了两种解决方案:

`辅助损失 (Auxiliary Loss)`: 在训练时, 对路由器的输出增加一个二元交叉熵损失. 目标是让被选入 top-k 的 token 的路由输出趋近于 1, 未被选入的趋近于 0. 这样在采样时, 只需看单个 token 的路由输出值 (例如, 是否大于 0.5) 就可以做出决定.

`辅助预测器 (Auxiliary Predictor)`: 训练一个独立的小型 MLP, 它的任务就是预测一个 token 是否会被选入 top-k. 这个预测器在采样时使用, 不影响主模型.

实验证明, 这是一个很容易学习的辅助任务, 准确率很快能达到 99%.

颜色: 单元格的颜色深浅表示路由决策. 深蓝色: 表示该 token 在该层被选中, 参与了计算 (即通过了自注意力和 MLP 模块).浅色: 表示该 token 在该层被跳过, 走了残差连接 (节省了计算).

论文中提到, 这个模型采用了每隔一个块进行一次路由的策略, 并且容量 k 设定为序列长度的 12.5%.整张图非常稀疏, 大部分区域都是浅色的. 这直观地证明了 MoD 确实在大量地跳过计算. 只有大约 12.5% 的 token 在每个路由层被处理, 这正是 MoD 模型计算效率提升的直接来源.

最引人注目的模式是图中靠右侧出现的一条垂直的深蓝色条纹. 这意味着某些特定的 token (或在序列中处于相似位置的 token) 在模型的多个不同深度被持续地、反复地选中进行计算. 这强烈暗示了模型已经学会了识别序列中的"重要"或"困难"部分. 对于那些简单的、信息量少的 token, 模型选择跳过它们以节省算力. 而对于那些对理解上下文或做出下一个预测至关重要的 token, 模型则会持续地将它有限的计算预算 (12.5% 的容量) 投入到这些 token 上. 论文作者也推测, 这些被频繁选中的 token 可能与那些具有更高熵 (即更难预测) 的输出相关. 这正是 MoD "动态分配计算" 核心思想的完美体现.

右图: 路由器权重分布直方图 (Distribution of router weights)

这个直方图展示了模型在做出路由决策前, 路由器为序列中所有 token 生成的原始路由权重 (router weights) 的分布情况.

X轴: 路由器的输出值 (经过 sigmoid 激活后, 范围在 0 到 1 之间).

Y轴: 对应权重值的 token 数量.

这个分布非常清晰地分成了两部分: 一大部分 token 的权重聚集在 0.0 附近, 一小部分 token 的权重聚集在 1.0 附近. 在中间区域 (如 0.5 附近) 几乎没有 token. 这种近乎完美的二元分离不是自然形成的, 而是通过第 3.5 节中提到的辅助损失 (auxiliary loss) "训练" 出来的. 这个辅助损失的目标就是迫使路由器做出明确的决策: 对于最终被选入 top-k 的 token, 它们的权重被推向 1. 对于未被选中的 token, 它们的权重被推向 0. 这个干净的分离是解决 top-k 非因果性问题的关键. 在自回归采样时, 我们无法看到未来的 token, 也就无法执行真正的 top-k 操作. 但是, 有了这样一个分布, 模型可以采用一个非常简单的因果策略: 对于任何一个新生成的 token, 只需检查其路由器的输出值. 如果大于一个固定的阈值 (例如 0.5), 就处理它; 如果小于, 就跳过它. 由于训练后的权重分布如此极端, 这种简单的阈值判断几乎可以完美地复现训练时的 top-k 决策, 从而在性能损失极小的情况下实现了高效的因果采样.

### 3.5 混合深度与混合专家 (Mixture-of-Depths-and-Experts, MoDE)

既然 MoD 可以让 token 跳过整个计算块 (自注意力 + MLP), 而 MoE 可以让 token 在 MLP 部分选择不同的专家, 那么将两者结合, 就可以构建一个更加灵活、高效的模型. 论文提出, MoD 带来的性能提升可以与 MoE 的优势叠加, 如下左图:

![图片](assets/6080670cd80b.png)

作者提出了两种MoD和MoE融合的策略:

分阶段 MoDE (Staged MoDE): 这是一个两步走的路由过程.

第一步 (MoD 决策): 在自注意力计算之前, 一个 MoD 路由器首先决定一个 token 是否要参与本层的计算. 如果决定 "跳过", 这个 token 就直接走残差连接.

第二步 (MoE 决策): 对于那些被 MoD 路由器选中 "计算" 的 token, 它们再进入 MoE 的路由器, 决定去哪个 MLP 专家那里进行计算.

这种方法的最大好处是它允许 token 跳过自注意力 (self-attention) 步骤. 自注意力是 Transformer 中计算成本很高的部分, 因此能够跳过它可以带来显著的计算节省. 但是结构相对复杂, 需要两个独立的路由机制串联工作.

集成式 MoDE (Integrated MoDE):这是一个一步到位的路由过程.

它巧妙地将 MoD 的 "跳过" 选项整合进了 MoE 的专家选择中.具体来说, MoE 的专家池里不仅有常规的 MLP 专家, 还加入了一个或多个特殊的 "空操作" 专家 ("no-op" experts). 这个 "空操作" 专家不做任何实际的 MLP 计算, 只是执行一个恒等变换 (即残差连接). 这样, MoE 的路由器在为 token 选择专家时, 它的选项就变成了: {MLP 专家1, MLP 专家2, ..., 空操作专家}. 如果一个 token 被路由到了 "空操作" 专家, 效果就等同于在 MoD 中被 "跳过" 了. 这种方法简化了路由机制, 只需要一个路由器就能同时实现 MoD 和 MoE 的功能, 设计上更加优雅.但是这种实现方式主要是在 MLP 阶段进行路由, token 无法跳过自注意力计算.

## 📘 4. Mixture-of-Recursions(MoR)

今年7月, Google发了一篇论文《Mixture-of-Recursions: Learning Dynamic Recursive Depths for Adaptive Token-Level Computation》[4]也是尝试用递归的方法来解决一些计算效率的问题.

### 4.1 Overview

这篇文章指出了当前效率优化的两个主要方向:

`参数效率 (Parameter Efficiency)`: 通过共享或减少模型权重来降低内存占用, 典型方法如层绑定(layer tying).

`自适应计算 (Adaptive Computation)`: 只在需要时投入更多的计算资源, 比如对简单的 token 进行浅层处理, 对复杂的 token 进行深层处理. 典型方法如早退机制(early-exiting)和专家混合模型(MoE).当然还有前面一篇介绍的MoD.

而MoR就是将这两个方向结合起来: 通过递归 (Recursion), 即反复使用同一组网络层, 实现参数共享.  通过路由器 (Router), 动态决定每个 token 需要经历多少次递归实现自适应计算.

这样的做法有几个优势:

`计算效率`: 注意力机制的计算复杂度是序列长度的二次方. MoR 只对在特定递归深度下仍然"活跃"的 token 进行计算, 大大减少了总计算量 (FLOPs).

`内存效率`: 传统Transformer需要对所有的token缓存KV, 非常消耗内存.  MoR 只缓存活跃 token 的 KV 对 (称为递归级 KV 缓存), 显著降低了内存占用和 I/O 开销

`性能`: 实验证明, MoR 在同等计算预算下, 性能超越了传统的 Transformer 和普通的递归 Transformer

作者将 MoR 的概念与"潜空间推理 (latent space reasoning)"联系起来. 传统的"思维链 (Chain-of-Thought)"是在文本序列上生成显式的推理步骤, 而 MoR 则是通过多次应用同一个参数块, 在模型的隐藏表示(潜空间)中进行"非语言的思考". 与其他在生成前进行固定深度思考的方法不同, MoR 在解码每个 token时都能进行自适应深度的思考.

一些核心的贡献:

统一的效率框架: 首次在单一架构中统一了参数共享, token 级自适应思考深度, 和内存高效的 KV 缓存.

动态递归路由: 引入了一个从头开始训练的路由器, 用于分配每个 token 的递归深度, 避免了传统早退方法中耗时且可能降低性能的后处理步骤.

广泛的实验验证: 在多种模型规模和计算预算下, MoR 证明了其优越性, 建立了新的性能-成本帕累托前沿.

高效的架构: MoR 通过选择性注意力计算减少了训练 FLOPs, 通过减少 KV 缓存大小提升了推理吞吐量.

![图片](assets/5d316782f4b0.png)

这张图由三个部分组成, 从左到右, 它们分别从`微观组件`、`宏观架构` 和 `具体实例` 三个层面解释了 MoR 模型.

**第一部分 (左)**:Recursion Block, 这部分展示了 MoR 模型最基本的计算单元. 这个"递归块"本身就是一个小型的 Transformer 模型, 由一个固定的层堆栈 (Layer 0 到 Layer L-1) 组成. 这些层就是标准的 Transformer 层, 在这个层堆栈的末端(或内部), 有一个关键组件——路由器. 它的作用是为每个通过这个块的 token 计算一个"继续"或"退出"的决策.

具体工作流程如下:

一个 token 的隐藏状态表示  进入这个递归块.

它依次通过 Layer 0 到 Layer L-1 进行计算和特征提取.

路由器接收处理后的 token 表示, 并计算出一个路由分数, 图中举例为 g = 0.7.

这个分数将决定该 token 的命运: 它是应该退出计算循环, 还是应该继续进行下一次递归.

这个递归块的所有参数都是共享的. 无论模型进行多少次递归, 使用的都是同一套 Layer 0 ... Layer L-1 和路由器的权重. 这就是参数效率的来源. 循环箭头清晰地表明, 这个块的输出可以再次作为它自身的输入.

**第二部分 (中)**: 整体架构 (Full Model Structure), 这部分展示了左边的"递归块"如何构成整个 MoR 模型.从宏观视角来看: 整个模型的核心就是一个被反复调用的共享递归块 (图中的灰色方框). 一个 token 进入模型后, 会首先经过这个递归块一次. 然后, 根据路由器的决策, 它可能会再次进入同一个块进行第二次计算, 甚至是第三次、第四次... 最多可以达到  次. 即构建了一个动态递归深度.

"Mixture"的含义: 对于一个序列中的不同 token, 它们经历的递归次数(深度)是不同的, 是一个混合体. 有些 token 可能只递归 1 次就退出了, 而另一些复杂的 token 可能会递归完整的 次.这个视图清晰地展示了 MoR 如何统一两大效率范式:

`参数共享`: 整个模型只有一个核心计算块, 参数量很小.

`自适应计算`: 并非所有 token 都执行相同次数的计算; 计算量是根据 token 的需要动态分配的.

**第三部分 (右)**: 实例与可视化 (Example Routing Pattern), 它用一个具体的例子生动地展示了 MoR 的自适应计算能力.

顶部网格图:

`X轴 (Sequence Position)`: 代表输入文本序列中的每一个 token 位置.

`Y轴 (Layer/Recursion Depth)`: 代表递归的深度或步骤.

单元格颜色:`深色 (Computed)`: 表示在该递归深度, 对应位置的 token 是活跃的, 参与了计算.`浅色 (Skipped)`: 表示在该递归深度, 对应位置的 token 已经提前退出 (exited), 不再参与计算.

观察到的模式: 最关键的发现是, 随着递归深度的增加 (从下往上看), 深色单元格的数量在逐渐减少. 这意味着模型在每一轮递归中都只关注越来越少的"重要"或"困难"的 token, 从而节省了大量的计算资源.

底部示例文本的颜色编码: 文本中的每个词根据其经历的递归次数被赋予了不同的颜色.

`红色 (3次递归)`: "People", "defensively confident", "Drugs". 这些是句子中的核心语义词汇, 具有很高的信息含量. 模型认为它们最"难"理解, 因此对它们进行了最深层次的"思考".

`蓝色 (2次递归)`: "who", "their", "may" 等功能词或次要内容. 它们的重要性介于中间.

`紫色 (1次递归)`: "and", "---", 换行符 \n. 这些是常见的连接词、标点符号或格式控制符. 模型认为它们最"简单", 只需要一次浅层处理即可.

### 4.2 Method

这一章详细阐述了 Mixture-of-Recursions (MoR) 的技术实现. 它首先介绍了作为基础的递归 Transformer, 然后深入探讨了 MoR 的两大核心组件: 路由策略和KV缓存策略.

#### 4.2.1 预备知识 (Preliminary)

首先作者介绍了递归Transformer, 心思想是参数共享. 它不再为每一层都使用独立的参数, 而是将模型划分为  个递归块, 每个块内部重用同一组共享参数 . 这样做可以在不增加模型参数量的情况下, 显著增加模型的有效计算深度.

**参数共享策略 (Parameter-sharing strategies)**: 论文探讨了多种共享方式, 表1 中重点介绍了两种:

![图片](assets/ecd4ed6931e8.png)

`Cycle策略`: 将一小组层 (例如, 第0, 1, 2层) 作为一个单元, 循环地重复使用. 比如, 共享单元有3层, 那么展开后的层序列就是 [0,1,2, 0,1,2, 0,1,2].

`Middle-Cycle策略`: 这是论文最终采用的策略. 它认为模型的第一层 (处理初始输入) 和最后一层 (生成最终输出) 的功能比较特殊, 应该保留其独立的、全容量的参数 ( 和 ). 而中间的层则采用 Cycle 策略进行参数共享. 这种设计在参数效率和模型表达能力之间取得了更好的平衡.

**递归模型的效率优势 (Enhanced training and inference efficiency)**

`训练效率`: 在使用FSDP(Fully Sharded Data Parallel) 等分布式训练技术时, 普通模型每进行一次前向/后向传播, 都需要一次 all-gather 操作来收集完整的层参数. 而递归模型可以在一次 all-gather 后, 重复使用这组参数进行  次计算, 大大提高了通信效率.

`推理效率`: 递归架构催生了连续深度批处理 (continuous depth-wise batching). 因为所有递归步骤都使用同一套参数, 所以可以将处于不同递归深度的 token 打包到同一个 batch 中进行计算, 从而消除 GPU 的空闲等待时间 ("bubbles"), 极大地提升了吞吐量.

**现有方法的局限性 (Limitations in prior works)**

作者指出了之前递归模型的两大痛点:

`KV 缓存瓶颈`: 尽管模型参数共享了, 但每一层 (展开后) 的 KV 缓存通常是独立的. 这意味着 KV 缓存的大小并没有减少, 在推理时依然是巨大的内存和 I/O 瓶颈.

`计算冗余`: 大多数递归模型对所有 token 应用固定的递归深度, 忽略了不同 token 的复杂性差异, 导致对简单 token 的计算是冗余和浪费的.

这两个痛点正是 MoR 要解决的核心问题.

#### 4.2.2 MoR

首先是路由策略: Expert-choice vs. Token-choice.路 由器的核心任务是决定每个 token 应该经历多少次递归, 即"思考"多深.

![图片](assets/f4732683c996.png)

**Expert-choice Routing (图a)** :一轮递归都可以看作一个"专家". 在进入第 r 轮递归前, 路由器会评估所有当前活跃的 token, 并从中选择分数最高的 top-k 个 token 进入本轮计算. 其余的 token 则被"提前退出".

: 路由器在第轮为 token  计算出的分数.

: 所有 token 分数的  百分位阈值. 只有分数超过这个阈值的 token 才会被选中.

: token  通过共享递归块进行的实际计算.

这种方法的优点和缺点:

`优点`: 负载均衡. 由于每轮递归处理的 token 数量 k 是固定的, 计算负载非常稳定, 便于系统优化.

`缺点`: 因果性破坏. 在自回归生成任务中, top-k 的选择需要看到整个序列的信息, 这会"泄露未来". 论文中提到, 这需要通过**辅助路由器(Auxiliary Router)或辅助损失(Auxiliary Loss)** 等技术来解决, 训练模型在只看到历史信息时也能做出正确的选择.

**Token-choice Routing (图b)** 在计算开始时(通常在第一次递归后), 路由器就为每个 token 一次性地决定它总共需要经历多少次递归(即分配给哪个"深度专家").路由器为每个 token t 计算一组分数  (对应不同的递归深度), 然后选择分数最高的那个深度  作为其最终递归次数.

`优点`: 无信息泄露. 决策是基于当前 token 自身状态做出的, 天然符合自回归的因果性.

`缺点`: 负载不均衡. 可能会有大量 token 被分配到同一个深度, 导致某些递归步骤计算量巨大, 而另一些则很空闲. 这需要通过平衡损失 (Balancing Loss) 等技术来缓解.

![图片](assets/487210ad9ee6.png)

**KV 缓存策略: Recursion-wise Caching vs. Recursive Sharing**

**Recursion-wise KV Caching**(图c-上): 与路由策略紧密耦合. 在第 r 轮递归中, 只有被路由器选中的 k 个活跃 token 才会缓存它们的 KV 对. 并且, 在这一轮的自注意力计算中, query 也只关注这 k 个 token 的 KV 对. 它具有如下优点:

`显著减少 KV 缓存`: 每一层的缓存大小从  (序列长度) 降为 k (活跃 token 数).

`显著降低计算量`: 注意力计算的 FLOPs 从  降低到 .

`提升内存访问效率`: 实现了"块局部计算 (block-local computation)", 数据访问更集中.

**Recursive KV Sharing** (图c-下) : 一种更激进的内存节省策略. 所有 token 在第一次递归时都计算并缓存它们的 KV 对. 在所有后续的递归中, 活跃的 token (Queries) 会重用这第一次缓存的、完整的 KV 对. 优点:

`极致的内存节省`: 整个递归部分只需要一层 KV 缓存, 内存占用降至最低 .

`降低Prefill延迟`: 在处理长文本时, 预填充阶段是主要耗时部分. 使用此策略, 预填充只需要计算一次递归块, 速度极快.

缺点:

`性能风险`: 让深层递归的 token 去关注浅层计算出的 KV 对, 存在"表示不匹配"的风险, 可能导致性能下降.

`计算量下降有限`: 注意力 FLOPs 仅下降到 , 并且 I/O 依然很高, 因为需要读取完整的 KV 缓存.

### 4.3 一些实验结果

这一章的核心目标是回答以下几个关键问题:

在相同的计算/数据预算下, MoR 是否比标准 Transformer 和普通递归 Transformer 更好? (主要结果)

MoR 的优势是否能随着模型规模的增长而保持? (IsoFLOP 分析)

MoR 在实际部署中的推理速度有多快? (推理吞吐量评估)

`基础架构`: 基于 Llama 的 Transformer 架构 (SmolLM).

`数据集`: FineWeb-Edu, 一个高质量的教育内容数据集.

`评估`: 主要两方面, 1. `困惑度(Perplexity)`: 在 FineWeb-Edu 验证集上评估负对数似然 (NLL), NLL 越低, 模型性能越好.2. `Few-shot准确率`: 在六个下游任务 (如 LAMBADA, MMLU) 上进行评估, 衡量模型的通用能力.

#### 4.3.1 主要结果

这一节旨在通过直接对比, 确立 MoR 的基本优势. 实验设计包含了两种对比方式: `同等计算预算 (isoFLOPs)` 和 `同等数据量 (fixed-token)`.

![图片](assets/6bf7037ddd6b.png)

核心结论:

MoR 在同等训练计算量下, 以更少参数获得更优性能.

MoR 在同等训练数据量下, 以更少计算量获得更优性能.

MoR 的不同设计变体性能对比 :

`路由策略`: Expert-choice (42.6%) 优于 Token-choice (40.0%), 表明路由的粒度和稳定性对性能至关重要.

`缓存策略`: 独立缓存 (Recursion-wise KV Cache) 优于共享缓存 (Recursive KV Share), 但差距不大. 这暗示了在内存非常受限的场景下, 使用 KV Share 是一个可接受的trade-off.

#### 4.3.2 IsoFLOP分析

这一节旨在验证 MoR 是否遵循缩放定律 (Scaling Laws), 即随着模型和计算规模的增长, 其性能是否能持续提升. 这是一个架构能否成为主流的关键标准. 作者的结论是MoR 的性能 (以验证集损失 Loss 为衡量标准) 随着计算预算的增加而稳定下降, 遵循了缩放定律. MoR 在所有规模和预算下都始终优于普通的 Recursive 模型.

然后是模型性能的对比, MoR 略微逊于 Vanilla. 这可能是因为在小模型上, 参数共享带来的"容量瓶颈" (representational bottleneck) 影响较大, 限制了模型的表达能力.MoR 的性能追上甚至超越了 Vanilla. 特别是在中低计算预算下, MoR 的优势更加明显.

但是这个工作的Scale都很小, 模型规模: 135M, 360M, 730M, 1.7B 四个级别. 计算预算: 2e18, 5e18, 16.5e18 三个级别. 没有更大规模的实验感觉有点缺陷.

#### 4.3.3 推理性能评估

这一节从实际部署的角度评估 MoR 的价值. 性能再好, 如果推理太慢, 应用价值也会大打折扣. MoR支持`连续深度批处理 (continuous depth-wise batching)`. 这是递归/MoR 模型独有的优化技术, 可以在一个 batch 中混合处理处于不同递归深度的 token, 从而消除 GPU 空闲, 保持高利用率.结论是MoR 显著提升推理吞吐量.

另一方面考虑性能与吞吐量的权衡.

![图片](assets/c39365003f0b.png)

**子图 (a): 推理吞吐量的帕累托前沿 (Pareto frontier of throughput)**: 这个图是衡量 MoR 实际部署价值的关键. 它展示了模型性能 (Log-Likelihood) 与推理速度 (Throughput) 之间的权衡关系. Vanilla: 标准的 Transformer 模型.MoR-2, MoR-3, MoR-4: 分别代表最大递归深度为 2, 3, 4 的 MoR 模型.

`X 轴 (Throughput)`: 推理吞吐量, 以 Vanilla Transformer 的速度为基准进行归一化. `越靠右越好`.

`Y 轴 (Log Likelihood)`: 模型在验证集上的对数似然, 代表模型性能. `越靠上越好`.

对于BatchSize采用两种场景:

`B = 32 (圆形点)`: 使用固定的 batch size = 32 进行测试.

`B = Max (方形点)`: 使用各自模型所能支持的最大 batch size 进行测试. 这个场景更能体现 MoR 因内存占用小而带来的优势.

结论:

`MoR 全面超越 Vanilla`: 所有的 MoR 数据点都位于 Vanilla 数据点的右上方. 这意味着 MoR 形成了一个新的、更优的帕累托前沿. 无论你需要什么样的性能, MoR 都能提供比 Vanilla 更快的速度; 无论你需要什么样的速度, MoR 都能提供比 Vanilla 更好的性能.

`递归深度与吞吐量的关系`: 随着递归深度  的增加 (从 MoR-2 到 MoR-4), 吞吐量显著提升. 例如, 在 B=Max 场景下, MoR-4 实现了 2.06 倍的加速. 这是因为更高的  意味着参数共享得更多 (模型更小), 并且有更多的 token 可以在浅层提前退出, 从而减少了总计算量和 KV 缓存.

`性能与吞吐量的权衡`: 增加  会带来轻微的性能下降 (Y 轴略微降低). 这是参数共享带来的"容量瓶颈"的体现. 但这种性能损失非常小, 换来的吞吐量提升却非常大, 因此这是一个非常划算的交易.

`最大 Batch Size 的优势`: B = Max 场景下的加速比远高于 B = 32 场景 (例如 MoR-4 从 1.42x 提升到 2.06x). 这证明了 MoR 减少 KV 缓存的策略能有效转化为更大的批处理能力, 进一步压榨 GPU 性能.

**子图 (b): 参数共享策略对比 (Sharing strategy)** :这个图旨在回答一个问题: 在递归 Transformer 中, 哪种参数共享方式最好?

测试采用了135M和360M两个小模型, 对比策略:

Seq (Sequence): 连续共享.

M-Seq (Middle-Sequence): 中间连续共享, 头尾独立.

Cyc (Cycle): 循环共享.

M-Cyc (Middle-Cycle): 中间循环共享, 头尾独立.

Baseline:

`红色虚线`: 全尺寸 Vanilla Transformer 的性能.

`黑色虚线`: 参数量与递归模型匹配的、小尺寸 Vanilla Transformer 的性能.

实验证明 Middle-Cycle 是最有效的参数共享策略

**子图 (c): KV 缓存共享策略对比 (KV cache sharing)** 探索在不同架构下, 共享 KV 缓存对性能的影响.

`绿色 (KV Sharing ✗)`: 不使用 KV 共享. 对于 MoR, 这代表使用 Recursion-wise Caching (递归级缓存).

`蓝色 (KV Sharing ✓)`: 使用 KV 共享. 对于 MoR, 这代表使用 Recursive Sharing (递归式共享, 即所有后续递归都重用第一次的 KV).

在Vanilla 和普通 Recursive 模型中, 启用 KV 共享 (蓝色条) 对性能的影响很小, 甚至在 Vanilla 上略有提升(可能起到了正则化作用). 这证明了共享 KV 缓存的可行性.而 MoR 中出现有趣的分化:

`MoR-E (Expert-choice)`: 启用 KV 共享后, 性能明显下降 (蓝色条远高于绿色条).

`MoR-T (Token-choice)`: 启用 KV 共享后, 性能反而略有提升 (蓝色条低于绿色条).

一些解释:

**为什么 MoR-E 性能下降?** Expert-choice 的路由非常精确, 它在每一层都选择了最需要计算的 token. 这种精确性依赖于当前深度的局部上下文. 如果强行让它去关注第一次递归时产生的全局 KV 缓存, 就会引入"陈旧"和"不相关"的信息, 从而干扰其精确的计算路径, 导致性能下降.

**为什么 MoR-T 性能提升?** Token-choice 的路由决策是一次性的, 可能不够精确, 容易出现负载不均衡. 在这种情况下, 允许它访问完整的、第一次递归时产生的全局 KV 缓存, 反而提供了额外的上下文信息, 弥补了其路由决策可能存在的不足, 从而提升了性能.

### 4.4 消融实验

主要围绕 MoR 框架的三个核心设计维度展开: 参数共享策略、路由策略和 KV 缓存策略.

#### 4.4.1 参数共享策略

对比了四种策略: Cycle, Sequence, Middle-Cycle, 和 Middle-Sequence. 在两种模型规模 (135M, 360M) 上进行了测试.**Middle-Cycle 是最优选择**.

#### 4.4.2 路由策略

![图片](assets/39c7d0c67231.png)

Expert-choice 路由的消融研究的设计变量:

解决方案: Auxiliary Router (训练一个独立的辅助路由器用于推理) vs. Auxiliary Loss (在主路由器上加一个辅助损失, 迫使其输出接近 0 或 1).

激活函数: sigmoid vs. tanh.

路由器架构: MLP, Linear (单层线性变换), Wide-MLP.

z-loss: 是否使用 z-loss 来惩罚过大的 logits, 提高稳定性.

结论:

Auxiliary Loss 优于 Auxiliary Router: Auxiliary Loss 方法在性能上 (Few-shot Acc 40.0) 优于 Auxiliary Router (39.4), 且实现更简单. 它能让主路由器自己学会如何在推理时做出正确的 top-k 预测.

简单架构效果好: 简单的线性 (Linear) 路由器 (Acc 40.1) 表现最佳, 甚至优于更复杂的 MLP (40.0) 和 Wide-MLP (39.4). 这说明对于路由决策这个任务, 过于复杂的非线性变换可能没有必要, 甚至会带来不稳定性.

Sigmoid 优于 Tanh: sigmoid 函数将输出限制在 (0, 1) 区间, 更适合作为概率或门控分数, 表现优于 tanh.

z-loss 影响甚微: 在最优配置下, z-loss 对性能影响不大, 但可以略微减少 "死令牌" (dead tokens) 的比例, 提高路由的覆盖率.

Token-choice 路由的消融研究设计变量:

均衡策略: Balancing Loss (用一个显式的损失函数来惩罚不均衡) vs. Loss-free (通过调整路由器偏置项来动态平衡).

激活函数: softmax vs. sigmoid.

路由器架构: MLP, Linear, Wide-MLP.

结论:

Balancing Loss 更稳定: 尽管两种策略的最终性能相似 (Acc 均为 39.4), 但 Balancing Loss 策略的负载不均衡度 (MaxVio) 显著更低. 这意味着它的路由行为更稳定、可预测, 是更优的选择.

Softmax + MLP 效果好: 对于选择哪个"深度专家"这个多分类问题, softmax 激活函数和 MLP 架构的组合表现最佳.

负载均衡的挑战: 作者提到, 即使使用了平衡损失, 模型在训练的早期阶段仍然难以实现负载均衡. 这是因为不同的"深度专家" (即不同递归次数) 本质上是异构的, 平衡它们的负载比平衡 MoE 中同构的专家网络要更困难.

#### 4.4.3 KVCache策略

对比了使用和不使用 KV 共享 (即 Recursive KV Sharing vs. Recursion-wise KV Caching) 在 Vanilla, Recursive, MoR-E, MoR-T 四种模型上的性能, 结论如下:

递归模型对 KV 共享具有鲁棒性: 在 Recursive 模型中启用 KV 共享, 性能下降很小. 作者在附录中分析, 这是因为共享的参数块倾向于在不同递归深度上生成方向(cosine similarity)和大小(magnitude)都非常相似的 KV 对, 为共享提供了天然的基础.

MoR-E 和 MoR-T 的分化:

KV 共享损害了 MoR-E: 对于路由精确的 MoR-E 模型, 强制共享全局 KV 缓存会引入噪声, 破坏其依赖局部信息的计算路径, 导致性能下降.

KV 共享帮助了 MoR-T: 对于路由较弱的 MoR-T 模型, 共享全局 KV 缓存提供了额外的上下文信息, 反而弥补了其路由决策的不足, 带来了性能提升.

### 4.5 分析

本章包含三个独立的分析点, 分别是计算最优缩放分析、路由分析和测试时缩放分析.

![图片](assets/5815798555d2.png)

#### 4.5.1 Compute-optimal Scaling Analysis

在固定的计算预算 (isoFLOPs) 下, 选择增加模型大小还是增加训练数据量对 MoR 更有利? 这与传统模型有何不同?

作者为每种架构 (Vanilla, Recursive, MoR) 拟合了性能曲线, 并从这些曲线中推导出在给定计算预算下的"最优模型尺寸".MoR 的斜率更平缓: 图中 MoR (绿色线) 的最优路径斜率比 Vanilla 和 Recursive 更接近水平. 斜率更平缓意味着, 为了达到下一个性能里程碑, MoR 模型更倾向于增加模型参数量 (横向移动), 而不是像传统模型那样需要同等比例地增加训练数据量. 换句话说, **MoR 模型是"数据高效"**.

这很可能是因为 MoR 的核心计算单元是一个共享的参数块. 提升这个共享块本身的容量和表达能力 (即增加模型大小) 变得至关重要, 其重要性甚至超过了简单地用更多数据来训练一个容量有限的共享块.

#### 4.5.2 路由分析

路由器到底学到了什么? 它的决策是随机的, 还是有意义的?

**定性发现 (语义重要性)**: 路由器学会了将更多的计算资源 (更深的递归) 分配给语义信息丰富的 token. 例如, "People", "-ensively confident", "Drugs" 这些核心词汇被分配了3次递归.而功能词 (如 "and")、标点符号和格式符 (如 "---") 等信息量较低的 token 则被分配了较少的递归次数 (1或2次). 这证明了 MoR 的自适应计算是有意义的.

**定量发现 (路由器分数分布)**: 图(b)展示了使用 Expert-choice + Auxiliary Loss 的路由器在每个递归步骤的输出分数. 蓝色 (被选中) 和橙色 (未被选中) 的分布被完美地分开了. 被选中的 token 的路由分数几乎全部集中在 1.0 附近, 而未被选中的 token 则全部集中在 0.0 附近. 这说明 Auxiliary Loss 这种训练方法非常有效, 它成功地训练出了一个高度确定性的路由器. 在推理时, 路由器可以非常容易地判断哪些 token 应该继续, 哪些应该退出.

#### 4.5.3 Test-time Scaling Analysis

MoR 模型训练好之后, 我们还能在推理(测试)时进一步提升它的性能吗?

作者训练了最大递归深度分别为  的 MoR 模型. 在测试时, 他们逐步增加允许的最大"思考"深度, 并观察模型性能 (Log-Likelihood) 的变化.在每一个 MoR 模型内部, 随着允许的递归深度增加, 模型的对数似然都在稳步提升. 这意味着即使模型是在最大深度为  的条件下训练的, 在推理时给予它更多的"思考"步骤, 它的表现也能变得更好.这个现象可以被看作是一种潜空间内的思维链 (Latent Chain-of-Thought). 每一次额外的递归, 都是对 token 表示的一次再提炼和再加工, 从而得到更精确的最终输出.

### 4.6 相关的工作

首先分析了递归Transformer模型, 核心的内容是阐述参数共享和提升模型效率是正交的. 相关的工作如下:

Universal Transformer (Dehghani et al., 2018): 最早提出通过重复应用一个 Transformer 块来模拟深度网络, 证明了递归的可行性.

Looped Transformers (Giannou et al., 2023): 将递归 Transformer 视为可编程计算机, 证明其理论上的强大能力.

Relaxed Recursive Transformers (Bae et al., 2024): 这是本文作者之一的前作. 通过在每个循环中加入 LoRA 适配器来"放松"参数共享的约束, 缓解了严格共享带来的性能损失.

Latent Reasoning (Geiping et al., 2025): 最近的工作表明, 递归深度与模型的潜空间推理能力密切相关.

MoR 继承了所有这些工作的基础, 即使用递归架构来实现参数效率. 之前的大多数工作都集中在`如何更好地进行参数共享` (如用 LoRA 放松约束) 和证明递归的潜力上. 它们大多采用固定的递归深度. MoR 则开辟了一个新的维度: 在递归架构的基础上, 引入了`动态的、token 级的自适应计算`, 解决了以往递归模型计算冗余的问题.

其次是分析了自适应计算 (Adaptive Computation), 相关工作如下:

Early Exiting: 为"简单"样本 (如 token 或序列) 在网络的浅层设置出口, 让它们提前结束计算. 这是 MoR 自适应深度的主要灵感来源之一.

Mixture-of-Depths (MoD) (Raposo et al., 2024): 这是一篇非常重要的相关工作. MoD 在标准 (非递归) Transformer 的每一层都设置一个路由器, 选择一部分 token 进行计算, 其余 token 则直接通过残差连接跳过该层.

MoR 和 MoD 都采用了 token 级的路由思想来实现自适应深度. MoR 将 MoD 的路由思想应用到了递归 Transformer 上. 但是架构不同, 带来新优势:

参数量恒定: MoD 的参数量随物理深度增加而增加. MoR 的参数量是固定的, 不随有效计算深度的增加而改变.

无限计算深度: MoR 理论上可以进行任意多次递归, 其计算深度可以超越模型的物理深度, 从而实现更深层次的"思考". MoD 的最大深度受限于其物理层数.

可以说, MoR = 递归架构 + MoD 的自适应路由思想, 这种结合创造了新的可能性.

然后是路由机制的分析, MoR 直接借鉴了 MoE 领域的这两种成熟的路由策略 (Token-choice 和 Expert-choice).MoR 并没有将 token 路由到空间上不同的专家网络 (horizontal scaling), 而是将它们路由到时间上(或深度上)不同次数的同一个计算块 (vertical scaling). 这是对路由机制应用场景的一次重要创新.

然后是KV Cache的优化, 传统的做法主要是`量化`,`丢弃`和`共享(例如MQA)` MoR 的 Recursion-wise Caching (递归级缓存) 和 Recursive Sharing (递归式共享) 提供了与上述方法正交的、全新的 KV 缓存优化思路:

Recursion-wise Caching: 这是一个与自适应计算紧密耦合的策略, 它根据路由结果动态地决定缓存哪些 token, 实现了更精细的内存管理.

Recursive Sharing: 这种策略在递归架构中尤其强大, 因为它能将prefill阶段的计算量减少  倍, 这对于处理超长上下文的场景具有巨大的加速潜力.

最后是潜空间推理 (Latent Reasoning), 让模型在隐藏状态中进行"思考", 而不是生成显式的文本推理步骤 (如 Chain-of-Thought). 传统的工作是通过插入特殊 token 或多次循环模型的最后状态, 让模型执行固定次数的内部推理.  MoR 的动态递归深度机制天然地支持了自适应的潜空间推理. 模型可以根据输入的复杂性, 动态地决定需要进行多少步"潜思考".

### 4.7 一些结论

总结了 MoR 的三大支柱 (参数共享, 自适应计算, 高效缓存) 和其最终带来的价值 (性能与效率的双赢). 但是也指出了这个工作的局限性:

MoR 的自适应递归深度天然地支持潜空间推理. 未来的一个关键工作是, 在真实的推理数据集上对 MoR 进行后训练 (post-training), 探索路由器如何动态地学习适应 CoT 链的复杂性.

受计算资源限制, 本文的实验只做到了 1.7B 参数规模.

本文的 Expert-choice 路由虽然能完美地实现负载均衡, 但其容量因子 (capacity factors, 即 top-k 的 k) 是在训练时预先设定的. 此外, 使用辅助损失训练的路由器输出过于"黑白分明" (接近0或1), 使得在推理时很难灵活地调整 k 值. 未来方向: 设计一种更具适应性的模型, 能够在训练和推理阶段都利用不同的容量因子. 这将赋予 MoR 更大的灵活性. 例如, 在推理时, 可以根据硬件负载或任务需求, 动态地调整 k 值, 从而在性能和成本之间进行实时权衡.

与稀疏算法的兼容性 (Compatibility with sparse algorithms), 将 MoR 与其他结构化稀疏 (structured sparsity) 技术结合, 进一步优化计算. 例如:在 MoR 的递归块内部, 动态地激活或修剪某些子网络或参数.将剪枝 (pruning) 或量化 (quantization) 等技术与 MoR 结合. 潜在价值: 实现多层次的稀疏性——在 token 级别 (MoR) 和参数级别 (剪枝/量化) 同时进行优化, 有望将模型效率推向新的极限. MoR 与这些技术是高度互补的.

扩展到多模态:  虽然MoR是模态无关的, 但是需要做更多的工作进行扩展验证. 特别对于像视频或音频这样的长序列数据, 自适应计算带来的内存和吞吐量增益可能会更加巨大. 通过为视频的每一帧或音频的每一段动态调整处理深度, MoR 可以在这些计算密集型领域解锁显著的效率提升.

## 📘 5. 一些总结和分析

### 5.1 Weight Sharing

实质上这些模型构造的递归都有一个潜在的归纳偏置的优势, 但是也带来了在循环递归过程中的参数共享对模型性能潜在的影响, 这是值得质疑和深度分析的一块内容. 去年翻译整理了论文《Position: Categorical Deep Learning is an Algebraic Theory of All Architectures》[5], 内容见[《大模型时代的数学基础(8)-CDL范畴深度学习》](https://mp.weixin.qq.com/s?__biz=MzUxNzQ5MTExNw==&mid=2247488968&idx=1&sn=8bfd5194e645f9c37469ca2793bbcee7&scene=21#wechat_redirect).

作者指出了当前模型设计的两种主流方法:

![图片](assets/efee858cf76b.png)

作者的立场是, 一个真正的通用深度学习指导框架, 必须用一个统一的数学理论来桥接上述两种方法. 他们认为这个理论应该源于计算机科学, 并且必须能够同时推广群论(GDL的基础)和函数式编程(JAX等框架的基础). 范畴论 (Category Theory) 正是满足这一要求的自然候选者.在范畴论的基础上, 这篇论文提出了一个观点: **权重共享不应被视为一种工程技巧或外部强加的约束, 而应被理解为计算过程本身的一种内在的、代数性的结构.**

论文通过其范畴论框架, 尤其是2-范畴 Para 和 lax algebra 的概念, 为这一观点提供了坚实的数学基础.

在传统的神经网络实践中, 权重共享通常是这样实现的:

定义一个大的参数张量.

在不同的计算位置, 重复使用这个张量的相同部分.

在反向传播时, 将来自不同位置的梯度累加到这个共享的参数上.

论文认为这种方法是"自下而上"的, 存在缺陷:

`非结构化`: 它没有在理论层面解释为什么可以或应该进行权重共享. 它是一种手工操作, 缺乏形式化的保证.

`描述性而非规定性`: 现有框架可以实现权重共享, 但难以从高级别规定一种计算必须具有权重共享的结构.

例如, 在设计一个RNN时, 我们凭直觉知道所有时间步的循环单元应该共享权重, 但这个"应该"并没有一个形式化的数学根基.

为了解决这个问题, 论文引入了 Para 2-范畴, 它的设计初衷就是为了形式化地处理参数和参数操作.

1-态射 (1-morphism): 一个参数化的函数 , 其中  是参数空间 (权重),  是使用这些参数的计算.

2-态射 (2-morphism): 一个重参数化 (reparameterisation) 映射. 这是一个从一个参数化函数  到另一个  的映射.

权重共享的核心在于复制 (copying) 参数. 比如, 我们有一个参数集, 但某个计算需要两份独立的参数 . 权重共享就是用同一个  来提供这两份参数. 这个"复制"操作  正好可以被建模为 Para 范畴中的一个 2-态射.

![图片](assets/09d110dc387f.png)

另一方面是Lax algebra的视角: 权重共享的结构是自然涌现 (emergent) 的. 这来自于将代数结构提升到2-范畴中得到的Lax Algebra. 当我们在 Para 2-范畴中定义一个Monad的Lax Algebra时, 代数公理不再是严格的等号, 而是允许存在2-态射. 论文中的 Theorem G.10 给出了一个结论:

在 Para 范畴中, 一个参数化Monad Para(T) 的任何Lax Algebra , 其参数空间 P 必须 构成一个`余幺半群 (comonoid)`.

一个余幺半群  恰好拥有两个关键操作:

余单位counit: , 对应"丢弃"参数的能力.

余乘法comulitplication: ,  对应`"复制"`参数的能力.

这个定理的深刻之处在于:

Lax algebra中的2-态射  正是余幺半群的余乘法 . 这意味着, 只要我们将一个递归/循环结构 (如RNN的展开) 定义为一个Lax Algebra同态, 那么它的参数空间就自动地、必然地被赋予了"可复制"的代数结构.

![图片](assets/2b06fd77f4cd.png)

核心的观点是: **你不是为权重共享添加约束, 而是通过对计算过程施加一个更高级的代数约束, 使得权重共享成为一个必然的、自动满足的结果.** 简单来说, 基于Lax代数的框架下, 你需要做的约束不是针对“权重共享”本身, 而是针对整个递归/循环网络的全局结构. 一旦你正确地定义了这个全局结构, 权重共享的机制就会作为该结构的代数推论而“免费”获得.

首先, 你需要定义一个单元(Cell),例如Transformer block或者RNN block. 在范畴论的语言里, 这被建模为一个**参数化函子的代数**.

`函子`: 对于序列处理, 基础函子是 , 其中  是输入类型,  是隐藏状态类型. 这个函子描述了下一步计算的“形状”: 要么是初始状态 (来自 1), 要么是接收一个输入 A 和前一个状态 S.

`参数化`: 我们让这个函子工作在 Para 2-范畴上, 得到 .

`代数`: 你需要定义一个  的代数, 它是一个参数化的态射: .  是你的参数空间 (权重矩阵和偏置).  是一个函数 , 这就是你的单元的具体计算逻辑.

到这里, 你施加的约束是:

选择正确的函子来描述你的基本计算步骤 (例如, 列表递归、树递归等).

定义一个参数化的计算单元 (P, cell) 作为该函子的代数.

第二步, 定义全局展开结构 (The "Unrolling"):你需要描述这个单元如何被"展开"来处理一个完整的序列 (一个列表). 在范畴论中, 这个过程被建模为一个从初始代数到你的单元代数的同态.

`初始代数`: 对于函子 , 它的初始代数是列表 . 这是所有有限序列的“原型”.

`同态`: 你需要构建一个从这个列表原型到你的Cell状态的映射, 即一个态射 .

最后施加核心约束 (The Main Constraint),这是最关键的一步. 论文的核心观点是, 上述的同态  必须是一个 Lax 代数同态 (Lax Algebra Homomorphism).这意味着  必须满足论文附录 J.1 中公式 (17) 所示的可交换图

### 5.2 基础设施影响

当递归加入到整个Attn或者Transformer block计算时, 参数共享情况下的Data Locality是一个很重要的研究方向, 对于算子和芯片架构都要带来一些深远的影响. 新的并行策略也会诞生出来. 当然在设计算法的时候, 也要进一步考虑GPU或者一些NPU的架构, Attn block的算法和架构远没到可以定制成ASIC的地步, 有一个暴论:未来两三年Attn的架构将发生极大的改变. 特别的来说就是满足前一章的代数结构下的Weight Sharing算子需要构造出来, 并且能够利用GPU高速的并行. 当然你会反驳MoR其实并没有改进Attn的架构, 但是相对于UT它对于循环是更早的退出了,而不是skip中间若干层的计算. 从UT的角度来看, 可能一些原有的CUDA Core的计算比重还会加重. 例如UT中引入的一些LSTM/GRU递归类型, 以及从前面一篇文章最优传输/信息几何视角来看, Softmax函数生成的注意力权重的过程, 完全等价于一个单边的熵最优输运(One-Sided Entropic Optimal Transport, EOT)问题的精确解

那么Nvidia过度的增加Tensor Core的计算密度是否能够满足这个趋势? 某些情况下来看, H200比B200更加平衡一些(B200 DieSize翻倍但是SM数量只有160增加了20%).这是一个更加开放性的问题...

但是, 从归纳偏置的优势, 从计算机体系结构上将Token as Instruction的视角Attention作为一个处理器的Control Unit来看, 这些递归/循环/分支都是特别重要的, 并且大概率它们将会成为整个大模型架构产生重大变革替代Transformer的关键技术.这些也是我一直在写[《大模型的数学基础》](https://mp.weixin.qq.com/mp/appmsgalbum?__biz=MzUxNzQ5MTExNw==&action=getalbum&album_id=3210156532718403586#wechat_redirect)这个专题的原因.
